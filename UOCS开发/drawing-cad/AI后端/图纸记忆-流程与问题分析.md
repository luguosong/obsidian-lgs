# 图纸记忆 — 流程与问题分析

> 跨会话、跨标签页的持久化图纸画像（类比 [[CLAUDE.md]]），目标是让"再次打开同一张图"时 LLM 直接引用已沉淀的图层语义和关键事实，跳过 ReAct 探索。
>
> 本文聚焦数据是**怎么产生的**、**LLM 在哪些环节发挥作用**，以及**目前实现的局限**。

---

## 1. 数据模型

后端 `DrawingMemory.java`，按 `(clientId, fileId)` 唯一，存 Redis Hash（TTL 180 天）。

| Section | 内容 | 上限 | 写入策略 |
|---|---|---|---|
| `summary` | 实体/图层/块总数、单位、extents、drawingType | 单例覆盖 | overwrite |
| `layer-semantics` | 道路/绿地/红线 等 → 图层名列表 + confidence + evidence | 20 | 同 category 完全覆盖 |
| `key-facts` | 客观事实（"绿地面积 18.97 ㎡"等） | 50 | FIFO 追加 |
| `qa-cache` | 已回答过的查询类问题 | 150 | 同 question 去重，超额异步归档到 `cad_drawing_memory_archive` |
| `metadata` | createdAt / updatedAt / **updateCount**（前端「沉淀 N 次」） | — | 任何写操作 +1 |
| `staleness` | staleSince + reason（写操作后置位） | — | 写图纸时打标 |
| `application-form` | 规划许可申请表 12 字段 | — | overwrite |
| `spatial-constraints` | 用地红线 handles/area | — | overwrite |

## 2. 写入路径

### 路径 A — 首次打开图纸自动归纳（一次性）

```
前端 AiAssistant.jsx:553  initDrawingMemory(clientId, fileId, summary, screenshot)
  └─ POST /api/ai/drawing-memory/{client}/{file}/init
      ├─ Stage 1（同步）— DrawingTypeSkillProvider#classify
      │   输入: comprehensive 摘要 + 全图截屏
      │   ⚠ 唯一启用视觉模型的环节
      │   输出: drawingType / subType / confidence
      └─ Stage 2（@Async）— DrawingMemoryInitService#doStage2
          输入: comprehensive 摘要（纯文本，无截图）
          LLM 输出: { summary, layerSemantics[], keyFacts[] }
          → applyResult() 逐条 store.addXxx() 落库
```

| 关键点 | 位置 |
|---|---|
| 前端摘要生成 | `DwgResultSummarizer.summarizeForMemoryInit` |
| Stage 1 视觉 prompt | `DrawingTypeSkillProvider.CLASSIFICATION_SYSTEM_PROMPT_WITH_VISION` |
| Stage 2 归纳 prompt | `DrawingMemoryInitService.SYSTEM_PROMPT` |
| LLM 输出解析 | [[Spring AI]] `BeanOutputConverter<InitResult>` |

### 路径 B — 对话中 LLM 主动调工具（增量）

```
ChatService 工具循环
  └─ DrawingMemoryTools.updateDrawingMemory(section, contentJson, confidence)
      ├─ 校验 section 白名单 / contentJson ≤ 4000 字符 / confidence ∈ {high,mid,low}
      ├─ clientId/fileId 由 RequestContextHolder 注入（LLM 不能传）
      └─ ObjectMapper.readTree → buildXxx → store.addXxx
```

工具描述（@Tool description）显式告诉 LLM 何时调用、并强制要求"调用后必须继续向用户输出实际答案"。

## 3. 视觉模型使用情况

| 阶段 | 视觉？ | 备注 |
|---|---|---|
| Stage 1 图纸类型分类 | ✅ 有截图时启用 | 多模态 prompt，内容仍以"文本通道为主，视觉为辅" |
| Stage 2 layerSemantics + keyFacts 归纳 | ❌ | 仅纯文本摘要 |
| LLM 工具 update_drawing_memory | ❌ | LLM 主对话上下文 |

**结论：图层语义和关键事实完全是 LLM 阅读"前端 WASM 预处理出来的纯文本摘要"产生的，与视觉模型无关。**

## 4. 两个真实例子的产生过程

### 例 1：「红线 · high / 图层：建筑红线, LIMIT_LINE / 依据：图层名含'红线'及'LIMIT_LINE'关键词」

| 步骤 | 谁负责 | 备注 |
|---|---|---|
| 1. 拿到图层名+实体数 | 前端 `extractEntities` 扫 ModelSpace 顶层 | 不递归块定义 |
| 2. 拼成 `## 图层(30): 建筑红线:7, LIMIT_LINE:3, ...` | `summarizeForMemoryInit` | 纯字符串 |
| 3. LLM **自由归类** | Stage 2 SYSTEM_PROMPT 指令 | category / evidence 全是 LLM 写的 |
| 4. 原样落库 | `applyResult` | 无二次校验 |

**前端 `_aggregateSiteAreas.inferCat()` 也有一套"红线/绿地"正则识别，但只用于算面积汇总，跟图层语义路径完全不通气** — 系统里有两套独立的图层分类规则，互不共享。

### 例 2：「绿地面积合计约 18.97 ㎡ · 来源：analysis.areaByCategory」

| 步骤 | 谁负责 | 备注 |
|---|---|---|
| 1. 每个实体的世界面积 | WASM `OdDbPolyline.area()` + `worldArea = area × \|sx·sy\|` | 块内才乘 areaScale |
| 2. 按 category 求和 | 前端 `_aggregateSiteAreas` JS 累加 | 18.97 这个数在这一步定死 |
| 3. 输出文本"绿地: 合计 **18.97 ㎡**" | `summarizeForMemoryInit` | |
| 4. LLM **复读**成 keyFact | Stage 2 SYSTEM_PROMPT | 几乎只是改写句式 |

⚠️ `source: "analysis.areaByCategory"` 是**幻觉指针** — `r.analysis.areaByCategory` 字段在前端代码里根本不存在（真实字段是 `r.analysis.entities[*].category`），LLM 是从 prompt 例子里抄来的。

## 5. 已知问题（按严重度）

### 🔴 严重 — 直接影响数值正确性

1. **块参照只穿透第 0 个 reference**
   `extractComprehensive:1426` `block.references[0].handle`。
   后果：同一块插入 N 次（如标准住宅楼单元、绿化模块），面积只算 1/N。

2. **嵌套块完全不递归**
   `extractBlockEntities` 不调用自身。后果：嵌套深度 ≥ 2 的图纸（楼层块→户型块），里层场地实体面积全部漏算。

3. **`buildKeyFact` confidence 兜底 bug**
   `DrawingMemoryTools:188-192` 不接收工具参数 `confidence`，与 `buildLayerSemantic:175-186` 行为不一致。LLM 在工具参数里传 high，但 contentJson 里没填，最终落库变成默认 mid。

4. **SYSTEM_PROMPT 要求"layers 必须是真实图层名"，代码不校验**
   `applyResult` 直接落库。LLM 幻觉一个图层名（如把 `DL_PAVE` 写成 `DL-PAVE`），会以 high confidence 永久污染记忆。

### 🟡 中等 — 影响数据完整性 / 设计一致性

5. **`addLayerSemantic` 是覆盖而非合并**
   同一 category 多次写入会丢失之前的 layers 列表，与"按 category 去重追加"的语义不符。

6. **顶层 layerStats 不含块内实体**
   `result.modelSpace.layerStats` 只统计 ModelSpace 顶层，块内统计另存在 `blocks[i].layerStats` 不合并。`totalEntities` 同样只是顶层数 — 在大量嵌套块的图纸中严重低估。

7. **`MAX_ENTITIES` 截断时 layerStats 与 totalEntities 数字对不上**
   layerStats 只对返回的 entities 累加，截断后的实体只增 totalScanned 不增 layerStats。

8. **section 白名单两层不一致**
   `DrawingMemory.VALID_SECTIONS` 含 8 个，工具 switch 只接受 5 个。新增 section 容易漏改。

9. **`confidence` 缺省兜底为 mid 与 prompt 意图相悖**
   SYSTEM_PROMPT 第 419 行说"判不准的忽略"，但 record canonical constructor 把 null 当 mid 收下。

10. **镜像（负缩放）信息丢失**
    `Math.abs(sx·sy)` — 面积值正确但镜像翻转的语义信息丢了。

### 🟢 轻微 — 健壮性 / 可观测性

11. **LLM 输出解析无 markdown fence 兜底**
    某些模型会输出 ```json ... ```，`BeanOutputConverter.convert()` 不一定能识别，外层只 log warn 不重试，前端会停在"正在建立图纸记忆"。

12. **comprehensive 摘要截断时不告诉 LLM**
    `MAX_COMPREHENSIVE_CHARS=5000` 简单截断，LLM 可能读到半句就当完整数据归纳。

13. **LLM raw output 没有持久化日志**
    只有一行 `log.info` 计数，事后审查"为什么 DL_路面 被归绿地"无现场可查。

14. **几何变换逻辑分散在两处**
    `_toWorldVerts`（完整仿射）只用于距离分析；面积只乘 `|sx·sy|`。将来支持非均匀坐标系（OCS 倾斜）时两处都要改。
