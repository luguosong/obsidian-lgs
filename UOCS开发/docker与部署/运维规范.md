## 配置管理

- **Nacos 3.1** 是唯一业务配置中心，本地 `application.yml` 仅保留连接与导入配置
- 业务配置必须通过 **Nacos API** 修改，禁止把业务配置写回本地 `application.yml`
- 每个服务使用 `{service}.yaml + {service}-{profile}.yaml` 的两层结构
- 各服务 `resources/nacos-config-reference.yml` 仅作参考，程序不会加载

> Nacos API 命令、配置清单与发布流程见 `upda-cloud/upda-cloud-server/CLAUDE.md`。

## Docker 端口分配约定

Docker Compose 暴露的宿主机端口遵循以下分类规则：

| 范围 | 类别 |
|------|------|
| `170xx` (17000-17099) | 前端服务 |
| `180xx` (18000-18099) | 后端服务 |
| `181xx` (18100-18199) | 中间件服务 |
| `190xx` (19000-19099) | 数据库服务 |

- 不直接暴露原始容器端口（如 MySQL 3306 → 19002）
- 容器内部端口不变，仅改宿主机映射端口
- 容器间通信使用容器名:内部端口（如 `mysql:3306`），与宿主机端口无关
- 新增服务时按类别分配下一个可用端口

> 端口详细清单与容器配置见 `docker-compose/CLAUDE.md`。

### AI 协作保护规则（强制）

为保证测试服务器稳定性，AI（Claude Code / Copilot CLI 等）必须遵守：

- **禁止主动更新测试服务器**：在未收到用户明确指令前，禁止执行任何影响测试服务器的操作，包括但不限于：
  - `svn commit` 触发服务器端同步
  - SSH 登录测试服务器执行 `svn update`、`docker compose` 相关命令
  - 直接修改测试服务器上的文件、配置、容器、数据卷
  - 触发 CI/CD 流水线部署到测试服务器
- **端到端测试一律在开发环境进行**：E2E / 集成测试直接运行本地开发环境（本机 Docker、`localhost`、开发分支代码）进行验证，不得借助测试服务器
- **测试通过后等待用户指令**：本地验证通过后，只描述"已验证通过，是否需要更新测试服务器？"并等待用户的明确指令（如"部署到测试服务器"、"svn commit 并更新 228"）才可执行同步动作
- **明确指令的识别**：用户必须使用包含"测试服务器"、"228"、"svn commit"、"部署"、"更新服务器"等明确词汇；模糊表述（"搞定它"、"全部完成"）不视为同步指令，应用 `ask_user` 澄清
