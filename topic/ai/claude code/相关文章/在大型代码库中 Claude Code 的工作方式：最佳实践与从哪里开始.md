Claude Code 正在数百万行代码的 monorepo、运行数十年的遗留系统、跨数十个仓库的分布式架构以及拥有数千名开发者的组织中投入生产使用。这些环境带来了小型简单代码库所没有的挑战——无论是每个子目录构建命令各不相同，还是遗留代码散落在没有共同根目录的各个文件夹中。

本文总结了我们观察到的、在规模化场景下成功推广 Claude Code 的通用模式。我们用"大型代码库"泛指多种部署场景：数百万行代码的 monorepo、历经数十年积累的遗留系统、跨独立仓库的数十个微服务，或以上任意组合。这也包括那些团队通常不会与 AI 编程工具联系在一起的语言，例如 C、C++、C#、Java、PHP。（Claude Code 在这些场景的表现通常优于大多数团队的预期，尤其是在近期模型版本发布之后。）尽管每个大型代码库的部署都受到其特定版本控制系统、团队结构和积累惯例的影响，本文的模式在这些场景中均可通用，是考虑采用 Claude Code 的团队的良好起点。

## Claude Code 如何在大型代码库中导航

Claude Code 的代码库导航方式与软件工程师相同：遍历文件系统、读取文件、用 grep 精确定位所需内容，并跨代码库追踪引用。它在开发者本机运行，无需构建、维护或上传代码库索引到服务器。

基于 RAG（检索增强生成）的 AI 编程工具通过对整个代码库进行向量嵌入，并在查询时检索相关片段。在大规模场景下，这类系统可能失效——嵌入管道的速度跟不上活跃工程团队的提交节奏。当开发者查询索引时，它反映的往往是几周、几天甚至几小时前的代码库状态。检索结果可能返回两周前被重命名的函数，或引用上一个 Sprint 中已删除的模块，而对这些过时信息毫无提示。

Agent 式搜索避免了这些问题。无需为数千工程师提交的新代码维护嵌入管道或集中索引，每个开发者的实例直接基于实时代码库工作。

但这种方式也有代价：当 Claude 有足够的初始上下文、知道从哪里开始查找时，效果最好。这意味着 Claude 的导航质量取决于代码库的配置程度——通过 CLAUDE.md 文件和 skills 分层注入上下文。如果让它在十亿行代码库中查找某个模糊模式，在工作开始之前就会触及 context window 上限。在代码库配置上投入的团队会获得更好的效果。

## Harness 与模型同等重要

关于 Claude Code 最常见的误解之一，是认为其能力完全由所用模型决定。团队往往关注模型的基准分数和在测试任务上的表现。但实际上，围绕模型构建的生态系统——即 harness（执行框架）——对 Claude Code 表现的影响不亚于模型本身。

Harness 由五个扩展点构成：CLAUDE.md 文件、hooks、skills、plugins 和 MCP 服务器，各自承担不同的功能。团队构建它们的顺序很重要，每一层都建立在前一层的基础上。另外两项能力——LSP 集成和 subagents——共同补全整个配置体系。下面逐一介绍这些组件和能力：

[**CLAUDE.md**](https://code.claude.com/docs/en/memory) **文件优先配置**。这些是 Claude 在每次会话开始时自动读取的上下文文件：根目录文件提供全局视图，子目录文件覆盖局部惯例。它们为 Claude 提供了做好任何任务所需的代码库知识。由于无论什么任务都会在每次会话中加载，保持内容聚焦于通用知识，可以避免其成为性能负担。

[**Hooks**](https://code.claude.com/docs/en/hooks-guide) **让配置持续自我优化**。大多数团队把 hooks 当作防止 Claude 出错的脚本，但它更有价值的用途是持续改进。停止 hook 可以在会话结束时趁上下文还新鲜，反思发生了什么并提出 CLAUDE.md 更新建议；启动 hook 可以动态加载团队专属上下文，让每位开发者在不手动配置的情况下，获得适合其模块的正确配置。对于 lint 和格式化等自动检查，hooks 以确定性的方式执行规则，比让 Claude 记住一条指令产生更一致的结果。

[**Skills**](https://code.claude.com/docs/en/skills) **在不臃肿每次会话的前提下，按需提供所需专业知识**。大型代码库中任务类型众多，并非所有专业知识都需要存在于每次会话。Skills 通过[渐进式披露](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)解决这一问题——将专项工作流和领域知识从常驻上下文中卸载，仅在任务需要时加载。例如，评估代码漏洞时加载安全审查 skill，代码变更后更新文档时加载文档处理 skill。

Skills 还可以按路径作用域限定，仅在代码库的相关部分激活。负责支付服务的团队可以将其部署 skill 绑定到该目录，这样在 monorepo 其他地方工作时就不会自动加载。

[**Plugins**](https://code.claude.com/docs/en/plugins) **将有效实践广泛传播**。大型代码库面临的一个挑战是，*好的*配置可能停留在部落知识层面。Plugin 将 skills、hooks 和 MCP 配置打包成一个可安装的单元，让新工程师在入职第一天安装该 plugin 后，立即拥有与资深同事相同的上下文和能力。Plugin 更新可以通过[托管市场](https://support.claude.com/en/articles/13837433-manage-claude-cowork-plugins-for-your-organization)在组织内统一分发。

例如，我们合作的一家大型零售商构建了一个连接其内部数据分析平台的 skill，让业务分析师无需离开工作流即可拉取性能数据。他们在大规模推广之前将其打包为 plugin 进行分发。

**语言服务器协议（LSP）集成让 Claude 拥有与 IDE 中开发者相同的导航能力**。大多数大型代码库 IDE 已经运行着 LSP，为"跳转到定义"和"查找所有引用"提供支持。将其暴露给 Claude，可以赋予它符号级别的精准导航能力：追踪函数调用到其定义、跨文件追踪引用，并区分不同语言中同名函数。没有 LSP 时，Claude 依靠文本模式匹配，可能定位到错误的符号。我们合作的一家企业软件公司在推广 Claude Code 之前，专门在全组织部署了 LSP 集成，以确保 C 和 C++ 在大规模场景下的导航可靠性。对于多语言代码库，这是投资回报率最高的举措之一。

**MCP 服务器扩展一切能力**。MCP 服务器是 Claude 连接内部工具、数据源和 API 的方式，否则这些资源无法直接访问。最精密的团队构建了 MCP 服务器，将结构化搜索作为 Claude 可直接调用的工具暴露出来。其他团队则将 Claude 连接到内部文档、工单系统或数据分析平台。

[**Subagents**](https://code.claude.com/docs/en/sub-agents) **将探索与编辑解耦**。Subagent 是一个独立的 Claude 实例，拥有自己的 context window，接收任务、完成工作，只将最终结果返回给父 agent。一旦 harness 就位，部分团队会启动一个只读 subagent 来映射子系统并将发现写入文件，然后让主 agent 在掌握全貌的情况下进行编辑。

## 来自成功部署的三种配置模式

如何为大型代码库配置 Claude Code，在很大程度上取决于代码库的结构。但在我们观察的部署中，有三种模式反复出现。

### 让代码库在规模下可导航

Claude 在大型代码库中的帮助能力，受限于它找到正确上下文的能力。每次会话加载的上下文过多会降低性能，上下文过少则让 Claude 盲目导航。最有效的部署都会提前投入，让代码库对 Claude 清晰可读。以下模式反复出现：

- **保持 CLAUDE.md 文件精简且分层。** Claude 在遍历代码库时会累积加载这些文件：根文件提供全局视图，子目录文件覆盖局部惯例。根文件应只包含指针和关键注意事项，其余内容都会成为噪音。
- **在子目录而非仓库根目录初始化。** 当 Claude 作用范围限定在与任务真正相关的代码库部分时，效果最佳。在 monorepo 中，这可能违反直觉，因为工具通常假设具有根目录访问权限。但 Claude 会自动向上遍历目录树并加载沿途所有 CLAUDE.md 文件，因此根级别的上下文永远不会丢失。
- **按子目录设定测试和 lint 命令的作用范围。** 当 Claude 只改动了一个服务时，运行完整测试套件会导致超时，并将上下文浪费在无关输出上。子目录级别的 CLAUDE.md 文件应指定适用于该代码库部分的命令。这对于每个目录有独立测试和构建命令的面向服务的代码库效果很好。在存在深层跨目录依赖的编译型语言 monorepo 中，按子目录划分作用范围更难实现，可能需要针对项目的特定构建配置。
- **使用 `.ignore` 文件排除生成文件、构建产物和第三方代码。** 在 `.claude/settings.json` 中提交 `permissions.deny` 规则，意味着这些排除规则受版本控制管理，团队中每位开发者无需自行配置即可获得相同的噪音过滤效果。在某些代码库中，生成文件本身是开发工作的对象。开发代码生成器的工程师可以在本地设置中覆盖项目级排除规则，而不影响其他团队成员。
- **当目录结构无法自我说明时，构建代码库地图。** 对于代码未以常规目录结构组织的团队，在仓库根目录放置一个轻量级 Markdown 文件，列出每个顶层文件夹及其内容的一句话描述，为 Claude 提供一个在打开文件前即可扫描的目录。对于拥有数百个顶层文件夹的代码库，最好采用分层方式：根文件只描述最高层结构，子目录 CLAUDE.md 文件提供下一层细节，按需随 Claude 遍历目录树时加载。对于较简单的情况，使用 @ 提及 Claude 应参考的具体文件或目录，也能达到相同效果。
- **运行 LSP 服务器，让 Claude 按符号而非字符串搜索。** 在大型代码库中 grep 一个常见函数名会返回数千条匹配，Claude 需要消耗大量上下文来打开文件判断哪条相关。LSP 只返回指向同一符号的引用，因此过滤在 Claude 读取任何内容之前就完成了。配置时需要为你的语言安装[代码智能 plugin](https://code.claude.com/docs/en/discover-plugins#code-intelligence) 和对应的语言服务器二进制文件；Claude Code 文档涵盖了可用 plugin 和故障排查说明。

> [!note] 说明
> 即使是分层 CLAUDE.md 方式也存在会失效的边界情况，例如拥有数十万个文件夹和数百万个文件的代码库，或使用非 git 版本控制的遗留系统。我们将在本系列后续文章中探讨这些挑战。

### 随模型智能演进主动维护 CLAUDE.md 文件

随着模型演进，为当前模型编写的指令可能会对未来的模型产生反效果。当新模型发布时，引导 Claude 处理它曾经有困难的模式的 CLAUDE.md 规则，可能会变得不必要，甚至成为主动约束。例如，要求 Claude 将每次重构拆分为单文件变更的规则，可能曾帮助旧模型保持专注，但会阻止新模型进行它能够很好处理的跨文件协调编辑。

为补偿特定模型局限性而构建的 skills 和 hooks——无论是模型推理能力的局限，还是 Claude Code 自身工具链的局限——一旦那些局限不再存在，就会成为额外开销。例如，一个为在 Perforce 代码库中执行 `p4 edit` 而拦截文件写入的 hook，在 Claude Code 添加原生 Perforce 模式后就变得多余了。

团队应预期每三到六个月进行一次有意义的配置审查，同时在重大模型发布后感到性能停滞时，也值得及时进行一次审查。

### 为 Claude Code 管理和推广分配负责人

单靠技术配置无法推动采用。那些做对的组织，同样在组织层面投入了资源。

传播最快的推广项目，在大规模开放访问前都进行了专项基础设施投入。一个小团队——有时甚至只有一个人——提前完成工具接入，让 Claude 在开发者首次接触时就已融入他们的工作流。在一家公司，几位工程师构建了一套 plugins 和 MCP，在第一天就可以使用。在另一家公司，整个专注于管理 AI 编程工具的团队，在推广开始之前就已将基础设施准备就绪。在两种情况下，开发者的第一次体验都是富有成效而非令人沮丧的，采用率从此蔓延开来。

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a04e25f1984beb50dc5525b_fig2-phases-of-claude-code-rollout-v1%402x.png)

目前从事这项工作的团队，往往隶属于开发者体验或开发者生产力部门——这通常是负责新工程师入职和构建开发者工具的职能部门。一个在多个组织中正在涌现的新角色是 agent manager：一种混合 PM/工程师的职能，专门负责管理 Claude Code 生态系统。对于没有专职团队的组织，最低可行版本是设立一名 DRI（直接负责人）：一个对 Claude Code 配置拥有所有权的人，具有决定设置、权限策略、plugin 市场和 CLAUDE.md 规范的权力，并负责保持其与时俱进。

自下而上的采用会产生热情，但在没有人来集中整合最佳实践的情况下，很容易走向碎片化。你需要有个人或团队来汇总并推广正确的 Claude Code 惯例（例如标准化的 CLAUDE.md 层级结构，或精心筛选的 skills 和 plugins 集合）。没有这项工作，知识将停留在部落层面，采用率将触及瓶颈。

在大型组织中，尤其是受监管行业中，治理问题会早早浮现，例如：谁控制哪些 skills 和 plugins 可用，如何防止数千名工程师各自重复构建相同的东西，如何确保 AI 生成的代码经历与人工生成代码相同的审查流程？为了及早应对这些问题，我们建议从一套明确的已批准 skills、必要的代码审查流程和有限的初始访问权限开始，随着信心建立逐步扩展。

我们观察到，那些在早期就建立跨职能工作组——将工程、信息安全和治理代表聚集在一起，共同定义需求并制定推广路线图——的组织，推广过程最为顺畅。

## 将这些模式应用于你的组织

Claude Code 的设计基于传统软件工程环境：工程师是代码库的主要贡献者，仓库使用 Git，代码遵循标准目录结构。大多数大型代码库符合这一模式，但非传统配置——例如包含大量二进制资产的游戏引擎、使用非常规版本控制的环境，或非工程师向代码库贡献内容的场景——需要额外的配置工作。我们的指南假设的是传统配置，我们描述的模式已在许多客户中得到验证。剩余的复杂性需要根据你的代码库、工具和组织进行具体判断。这正是 Anthropic Applied AI 团队与工程团队直接合作，将这些模式转化为符合你的组织特定需求的地方。
