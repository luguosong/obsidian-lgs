# gstack 深度解析：设计哲学与技术架构

> 本文是对 [[gstack]] 和 [[gstack 使用手册与最佳实践]] 的补充，专注于**底层设计哲学**与**技术架构**——"为什么这样设计"而非"怎么操作"。

---

## 三大设计哲学（ETHOS.md）

[[gstack]] 的所有技能设计都源于三个核心哲学，理解这三个原则能帮你判断边界情况、避免误用。

### 1. Boil the Lake（煮沸湖泊）

> **核心思想**：AI 辅助编程让"做完整"的边际成本接近零。既然如此，就该做完整——但要认清什么是"湖泊"，什么是"海洋"。

**湖泊 vs 海洋**的区别：

| 湖泊（可以煮沸）| 海洋（不要碰）|
|---|---|
| 100% 测试覆盖率 | 从零重写整个系统 |
| 所有边缘情况处理 | 多季度平台迁移 |
| 完整功能实现 | 跨团队架构重构 |

**实践意义**：当 AI 帮你写代码时，不要让"差不多就行"成为默认选项。测试可以完整覆盖，文档可以完全同步，但不要因此陷入试图完整重写一个遗留系统的幻觉。

### 2. Search Before Building（先搜索后构建）

> **核心思想**：写代码之前，先检查是否已有解决方案。三层知识体系决定搜索顺序。

```
Layer 1 — 经验验证的：标准模式、久经考验的方法
    ↓ 没找到
Layer 2 — 新兴流行的：社区常见实践、博客、生态趋势
    ↓ 没找到
Layer 3 — 第一性原理：从具体问题推导原创解法
```

**在 [[gstack]] 中的体现**：`/office-hours` 强制你回答六个追问式问题，目的之一就是在动手前确认"是否已经有更简单的方案"，防止重复发明轮子。

### 3. User Sovereignty（用户主权）

> **核心思想**：AI 模型推荐，用户决策。两个 AI 模型同意某个变更是强信号，但不是命令。

**关键含义**：[[gstack]] 的技能不是自动执行的流水线，每个关键节点都需要人来拍板。这解释了为什么：
- `/careful` 在修改前要求你确认
- `/autoplan` 只把"品味决策"呈交给你，技术细节由 AI 自动处理
- `/codex` 提供第二意见而非直接执行

> [!tip] 三个哲学的优先级
> 实际使用中，**User Sovereignty** 始终最高优先——无论 AI 多有把握，涉及破坏性操作时都要确认。

## 技术架构解析

### 整体调用链

```
Claude Code → 编译二进制（CLI）→ HTTP POST → Bun Server → CDP → Chromium（无头）
```

### 为什么选 Bun 而非 Node.js

官方 ARCHITECTURE.md 给出了四个具体理由：

| 原因 | 说明 |
|------|------|
| **编译单体二进制** | `bun build --compile` 生成约 58MB 可执行文件，用户无需安装运行时 |
| **原生 SQLite** | 直接读取 Chromium 的 SQLite Cookie 数据库，解密不需要额外依赖 |
| **原生 TypeScript** | 开发时无需编译步骤，改完直接运行 |
| **内置 HTTP 服务器** | `Bun.serve()` 即可，无需引入 Express 或 Fastify |

### 浏览器守护进程模型

`/browse` 背后是一个**持久化的 Chromium 守护进程**，而非每次调用都冷启动浏览器：

```
首次调用  ──→  启动 Chromium（约 3 秒）
后续调用  ──→  复用已有进程（约 100-200ms）
空闲 30 分钟 → 自动关闭
```

> [!info] 为什么要用守护进程
> 每次冷启动浏览器需要 3-5 秒，在一次 `/qa` 测试中可能需要 18 次工具调用，总等待时间差异是 "54-90 秒" vs "1.8-3.6 秒"。守护进程让 AI 的 QA 测试体验与调用内部 API 无异。

**会话状态跨命令保留**：Cookie、标签页、localStorage 在命令之间保持不变，这使得"登录 → 操作 → 验证"这类多步骤测试流程成为可能。

**随机端口设计**：守护进程使用 10000-60000 之间的随机端口，支持多个工作区并行运行，互不冲突——这是 Conductor 并行 Sprint 模式的底层基础。

### Ref 系统（元素引用机制）

[[gstack]] 浏览器操作使用 `@e1`、`@e2`、`@c1` 等标识来引用页面元素，而非直接操作 DOM：

- **基于无障碍树定位**：通过 Playwright 的 Accessibility Tree 定位元素，不修改 DOM
- **导航时自动清除**：页面跳转后，旧的 Refs 自动失效，防止悬空引用
- **稳健性更高**：即使页面 CSS 类名或 HTML 结构变化，只要无障碍语义不变，引用仍然有效

### 安全模型四层防护

```
┌─────────────────────────────────────┐
│  Layer 4：内容过滤器（过滤危险指令）│
├─────────────────────────────────────┤
│  Layer 3：数据标记（标记来源）      │
├─────────────────────────────────────┤
│  Layer 2：隐藏元素剥离              │
├─────────────────────────────────────┤
│  Layer 1：内容信封（隔离外部内容）  │
└─────────────────────────────────────┘
```

**额外安全保障**：
- 只绑定 localhost，不暴露到网络
- [[Bearer Token]] 认证（UUID v4）
- macOS Keychain 访问需用户明确批准，Cookie 在内存中解密，不写入明文磁盘

## 效率压缩比数据

> 数据来源：[[gstack]] 官方 ETHOS.md，反映的是 Garry Tan 个人实测数据，实际效果取决于项目复杂度。

| 任务类型 | 传统耗时 | 使用 [[gstack]] | 压缩比 |
|----------|---------|------------|--------|
| 样板代码 / 脚手架 | 2 天 | 15 分钟 | ~100x |
| 测试编写 | 1 天 | 15 分钟 | ~50x |
| 功能实现 | 1 周 | 30 分钟 | ~30x |
| Bug 修复 + 回归测试 | 4 小时 | 15 分钟 | ~20x |

> [!warning] 关于"行数膨胀"争议
> 批评者指出 AI 会膨胀代码行数。Garry Tan 对此的回应是：他使用**逻辑代码变更**而非原始行数来衡量生产力，2026 年的节奏约为 2013 年的 810 倍（日均逻辑代码变更：11,417 vs 14 行）。详见 [ON_THE_LOC_CONTROVERSY.md](https://github.com/garrytan/gstack/blob/main/docs/ON_THE_LOC_CONTROVERSY.md)。

## 并行 Sprint 的实现原理

单 Sprint 已经强大，10-15 个并行 Sprint 才是 [[gstack]] 真正的差距所在。

### 为什么可以并行

1. **守护进程随机端口**：每个工作区的浏览器实例互不冲突
2. **Sprint 结构化**：每个阶段有明确的入口/出口条件，AI 知道何时停止等待人的决策
3. **[Conductor](https://conductor.build/) 工具**：并行调度多个 [[Claude Code]] 会话，每个在独立工作区

### 管理多个并行 Sprint

```
终端 1: 功能 A → /plan-eng-review（等待架构确认）
终端 2: 功能 B → /qa（自动测试中）
终端 3: 功能 C → /ship（等待 PR 合并）
终端 4: 功能 D → Build 阶段
...
```

**你的角色变成 CEO**：只需关注需要人类拍板的决策节点，其余自动运行。这就是"一人抵团队"的实质。

## 与 Karpathy AI 编码四大失败模式的对应

Andrej Karpathy 在其 [AI 编码规则](https://github.com/forrestchang/andrej-karpathy-skills)（17K Star）中总结了四种失败模式，[[gstack]] 有针对性地解决了每一个：

| 失败模式 | [[gstack]] 的解决方案 |
|---------|-----------------|
| **错误假设**（代码编写前未澄清需求）| `/office-hours` 在写代码前把假设暴露出来 |
| **过度复杂**（AI 引入不必要的复杂性）| `/review` 捕获过度工程化问题 |
| **无关编辑**（AI"顺手"改了不相关代码）| `/freeze` + `/careful` 限制修改范围 |
| **命令式而非声明式**（任务描述不清晰）| `/ship` 将任务转化为可验证目标 |
