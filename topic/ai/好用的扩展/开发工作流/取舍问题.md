## 先回答你的担忧：会过重吗？

**结论：是的，但"过重"分三种，对症才能下药。**

让我们拆开看。你能不能先告诉我，下面三种"重"里，你最担心哪一种？

|类型|表现|实际成本|
|---|---|---|
|**A. Context 冗余**|主对话每次都被塞进一堆 system prompt / CLAUDE.md 内容|烧 token、降低 Claude 注意力|
|**B. Skill / 命令过载**|几十个 `/xxx` 命令，记不住、互相重叠|认知负担，决策瘫痪|
|**C. 流程过重**|一个小 feature 要走 8 阶段，杀鸡用牛刀|拖慢节奏，团队抵触|

我猜你最担心的是 **A + B**，对吗？因为你的文档里已经识别出"重叠"（GSD `/gsd-ship` vs GStack `/ship`、OpenSpec tasks vs Superpowers writing-plans），但还没动刀。

## 关键认知：这 4 个工具的"安装成本"其实差异很大

在你做取舍之前，先理解它们**怎么占用上下文**——这是判断是否冗余的基础：

|工具|安装后默认占用 main context 吗？|触发方式|
|---|---|---|
|**GSD**|部分（状态文件被读入）|显式命令 `/gsd-*`|
|**OpenSpec**|几乎不占（制品按需 read）|显式命令 `/opsx:*`|
|**Superpowers**|**会**（skill 描述常驻 CLAUDE.md）|自动触发 + 显式|
|**GStack**|**会**（虚拟角色 prompt）|显式命令 `/xxx`|

看到模式了吗？**显式命令型工具的"安装成本"低**——不调用就不占 context；**自动触发型工具的成本高**——它得让 Claude 知道"什么时候该用"。

这给了你第一个判断原则：

> **重叠的功能里，留显式触发的、砍自动触发的——除非自动触发的那个真的值得常驻。**

你能用这个原则，自己先尝试判断一下：GSD `/gsd-verify-work` 和 Superpowers `requesting-code-review` 留哪个？

## 三种取舍方案（从轻到重）

我给你三个梯度的方案，你根据**项目规模 + 团队人数 + 你个人偏好**来选。不要直接抄，思考一下哪个匹配你。

### 方案 A：**核心二人组**（最轻，适合个人 / 小项目 / 原型期）

```
保留：OpenSpec + Superpowers
砍掉：GSD + GStack
```

**理由**：

- OpenSpec 解决"做什么"——spec 制品是任何规模都需要的
- Superpowers 解决"怎么做"——TDD、review、debugging 三件套
- 上下文管理用 OpenSpec 的 `proposal.md` 兜底（每个 feature 自带上下文）
- 浏览器测试用 **Playwright MCP**（不绑定 GStack）

**什么时候不够用**：项目跨越多个 milestone、需要明确的安全审计、有产品/设计 stakeholder 要 review。

### 方案 B：**三件套**（中等，适合中型项目 / 2-5 人团队）

```
保留：OpenSpec + Superpowers + GStack（只保留 4 个命令）
砍掉：GSD（用 OpenSpec + git worktree 替代）
GStack 只留：/qa、/cso、/ship、/review
```

**理由**：

- GSD 的核心价值是"跨会话状态" + "phase 并行"。**跨会话状态可以用 OpenSpec 的 archive 机制替代**，并行可以用原生 subagent + worktree。
- GStack 的虚拟角色你可能只真正需要 2-3 个：QA（不可替代）、CSO（不可替代）、Ship（流程价值高）。其他角色（CEO/Designer）一次性对话即可，不值得常驻。

**关键洞察**：GStack 有 20+ 命令，但**80% 的价值集中在 20% 的命令**。你能猜到为什么 `/qa` 和 `/cso` 最不可替代吗？

> 因为它们**有外部世界副作用**——`/qa` 真的开浏览器、`/cso` 跑 OWASP 检查。其他命令本质是"换个 prompt 跟你聊"，价值密度低。

### 方案 C：**全员保留 + 严格分工**（最重，适合大团队 / 长周期项目）

如果你真的全留，必须做这 3 件事来对冲冗余：

1. **CLAUDE.md 只列触发条件**：不要把所有 skill 描述全塞进去，列"什么场景该想到什么工具"
2. **每个工具有"禁用清单"**：明确写出 `不要用 /gsd-ship，统一用 /ship`
3. **Slash 命令分命名空间**：所有 GSD 命令前缀 `/gsd:`，OpenSpec 前缀 `/ops:`，避免误触发

## 给你的判断练习

在我替你决定之前，先回答几个问题（这比我直接给答案有用）：

1. **你的项目周期多长？** 几周 vs 几个月 vs 几年——决定你需不需要 GSD 的 milestone 机制
2. **你是单人还是有团队？** 单人的话 GStack 的"虚拟角色"价值打折（你脑子里就能扮演）
