## LLM Wiki

一种使用 LLM 构建个人知识库的模式。

这是一份创意文档，设计为可直接复制粘贴到你的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）中使用。它的目标是传达高层设计思想，而具体的实现细节则由你的 agent 在与你的协作中逐步构建。

## 核心思想

大多数人与 LLM 和文档的交互方式类似 RAG：你上传一组文件，LLM 在查询时检索相关片段并生成回答。这能工作，但 LLM 每次都在从零重新发现知识，没有积累。问一个需要综合五份文档才能回答的细微问题，LLM 每次都得重新找到并拼接相关片段。没有任何东西被沉淀下来。NotebookLM、ChatGPT 文件上传以及大多数 RAG 系统都是这样工作的。

这里的思想不同。LLM 不是在查询时才从原始文档中检索，而是**增量地构建和维护一个持久化的 wiki** —— 一组结构化的、互相链接的 markdown 文件，位于你和原始资料之间。当你添加新资料时，LLM 不仅为后续检索建立索引，还会阅读它、提取关键信息、并将其整合到现有 wiki 中 —— 更新实体页面、修订主题摘要、标注新数据与旧有论断的矛盾之处、强化或挑战正在演进的综合性论述。知识被编译一次，然后*持续保持最新*，而不是每次查询都重新推导。

这是关键区别：**wiki 是一个持久化的、可复利的产物。** 交叉引用已经建好了。矛盾已被标记。综合论述已经反映了你读过的所有内容。每添加一个新资料、每问一个问题，wiki 都变得更加丰富。

你从不需要（或很少需要）自己编写 wiki —— LLM 负责编写和维护所有内容。你负责获取资料、探索方向、提出好问题。LLM 承担所有琐碎工作 —— 摘要、交叉引用、归档、以及让知识库随时间真正有用的那些维护工作。实际操作中，我一边开着 LLM agent，另一边开着 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果 —— 跟踪链接、检查图谱视图、阅读更新后的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这可以应用于很多不同场景。几个例子：

- **个人**：追踪你自己的目标、健康、心理、自我提升 —— 归档日记条目、文章、播客笔记，逐步建立起关于自己的结构化画像。
- **研究**：在几周或几个月内深入一个主题 —— 阅读论文、文章、报告，增量构建一个带有演进论点的综合 wiki。
- **读书**：逐章归档，为角色、主题、情节线索及其关联建立页面。读完时你就拥有了一个丰富的伴读 wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki —— 成千上万的互链页面，涵盖角色、地点、事件、语言，由志愿者社区历时多年构建。你可以在阅读过程中亲自构建类似的东西，由 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，输入来自 Slack 线程、会议记录、项目文档、客户通话。可以让人类在流程中审查更新。wiki 能保持最新，因为 LLM 承担了团队中没人愿意做的维护工作。
- **竞争分析、尽职调查、旅行规划、课程笔记、爱好深度研究** —— 任何你需要随时间积累知识并希望有组织而非散乱管理的场景。

## 架构

共有三层：

**原始资料（Raw sources）** —— 你精心整理的源文档集合。文章、论文、图片、数据文件。这些是不可变的 —— LLM 只从中读取，永远不修改它们。这是你的真实来源。

**Wiki** —— 一个由 LLM 生成的 markdown 文件目录。摘要、实体页面、概念页面、对比、概览、综合分析。LLM 完全拥有这一层。它创建页面、在新资料到来时更新它们、维护交叉引用、并保持一切的一致性。你阅读它；LLM 编写它。

**Schema** —— 一份文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM wiki 的结构、约定是什么、以及在摄入资料、回答问题或维护 wiki 时应遵循的工作流。这是关键配置文件 —— 正是它让 LLM 成为一个有纪律的 wiki 维护者，而不是一个通用聊天机器人。你和 LLM 随着时间共同演进这份文件，不断摸索出适合你领域的最佳实践。

## 操作

**Ingest（摄入）。** 你将新资料放入原始资料集合并让 LLM 处理它。一个示例流程：LLM 阅读资料、与你讨论关键要点、在 wiki 中写一个摘要页、更新索引、更新 wiki 中相关的实体和概念页面、并在日志中追加一条记录。一份资料可能涉及 10-15 个 wiki 页面。个人而言，我更倾向于逐份摄入并保持参与 —— 我阅读摘要、检查更新、引导 LLM 重点关注什么。但你也可以在较少监督下批量摄入大量资料。由你来开发适合自己风格的工作流，并将其记录在 schema 中以备后续会话使用。

**Query（查询）。** 你针对 wiki 提问。LLM 搜索相关页面、阅读它们、并综合生成带引用的回答。回答可以采取不同形式，取决于问题 —— markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布。重要的洞察：**好的回答可以作为新页面回填到 wiki 中。** 你请求的对比、一项分析、你发现的关联 —— 这些都是有价值的，不应该消失在聊天历史中。这样你的探索就能像摄入的资料一样在知识库中复利增长。

**Lint（检查）。** 定期让 LLM 对 wiki 做健康检查。检查：页面间的矛盾、被新资料取代的过时论断、没有入链的孤立页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的知识空缺。LLM 擅长建议新的研究问题和需要寻找的新资料。这能保持 wiki 在增长过程中的健康。

## 索引与日志

两个特殊文件帮助 LLM（和你）在 wiki 增长过程中进行导航。它们各司其职：

**index.md** 是面向内容的。它是 wiki 中所有内容的目录 —— 每个页面列出链接、一句话摘要、以及可选的元数据如日期或来源数量。按类别组织（实体、概念、来源等）。LLM 在每次摄入时更新它。回答查询时，LLM 先读取索引来找到相关页面，然后深入阅读。这在中等规模下（约 100 份资料、数百个页面）出奇地好用，完全不需要基于 embedding 的 RAG 基础设施。

**log.md** 是按时间顺序的。它是一个 append-only 的记录，记录了什么时间发生了什么 —— 摄入、查询、检查。一个实用技巧：如果每条记录以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title`），日志就可以用简单的 Unix 工具解析 —— `grep "^## \[" log.md | tail -5` 给你最近 5 条记录。日志为你提供了 wiki 演化的时间线，并帮助 LLM 了解最近做过什么。

## 可选：CLI 工具

在某个阶段你可能想构建一些小工具来帮助 LLM 更高效地操作 wiki。最明显的一个是 wiki 页面的搜索引擎 —— 小规模时索引文件就够了，但随着 wiki 增长你需要正式的搜索功能。[qmd](https://github.com/tobi/qmd) 是一个好选择：它是一个本地 markdown 文件搜索引擎，支持混合 BM25/向量搜索和 LLM 重排序，全部在本地运行。它既有 CLI（让 LLM 可以通过 shell 调用），也有 MCP server（让 LLM 可以作为原生工具使用）。你也可以自己构建更简单的方案 —— LLM 可以帮你按需 vibe-code 一个简单的搜索脚本。

## 技巧与窍门

- **Obsidian Web Clipper** 是一个浏览器扩展，能将网页文章转换为 markdown。对于快速将资料导入原始资料集合非常有用。
- **下载图片到本地。** 在 Obsidian 设置 → 文件与链接中，将"附件文件夹路径"设为固定目录（如 `raw/assets/`）。然后在设置 → 快捷键中搜索"Download"找到"Download attachments for current file"并绑定快捷键（如 Ctrl+Shift+D）。剪藏文章后按下快捷键，所有图片就会下载到本地磁盘。这是可选的但很有用 —— 它让 LLM 能直接查看和引用图片，而不依赖可能失效的 URL。注意 LLM 不能在一次处理中同时读取带内联图片的 markdown —— 变通方法是让 LLM 先读文本，再单独查看部分或全部引用的图片以获取额外上下文。有点笨拙但足够好用。
- **Obsidian 的图谱视图** 是查看 wiki 形态的最佳方式 —— 什么和什么连接、哪些页面是枢纽、哪些是孤立的。
- **Marp** 是基于 markdown 的幻灯片格式。Obsidian 有对应插件。适合直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以对页面 frontmatter 执行查询。如果你的 LLM 在 wiki 页面中添加了 YAML frontmatter（标签、日期、来源数量），Dataview 就能生成动态表格和列表。
- Wiki 本质上就是一个 markdown 文件的 git 仓库。你可以免费获得版本历史、分支和协作能力。

## 为什么这能工作

维护知识库的繁琐之处不在于阅读或思考 —— 而在于归档管理。更新交叉引用、保持摘要最新、标注新数据与旧论断的矛盾、在数十个页面间保持一致性。人们放弃 wiki 的原因是维护负担增长得比价值快。LLM 不会厌倦，不会忘记更新一个交叉引用，而且一次就能处理 15 个文件。Wiki 能保持维护是因为维护成本接近于零。

人类的工作是管理资料来源、引导分析方向、提出好问题、思考这一切的意义。LLM 的工作是其他所有事情。

这个思想在精神上与 Vannevar Bush 的 Memex（1945）相关 —— 一个个人的、精心管理的知识存储，文档之间通过关联路径相连。Bush 的愿景更接近这个模式而非万维网的现状：私有的、主动管理的，文档之间的连接与文档本身同等重要。他没能解决的是谁来承担维护工作。LLM 解决了这个问题。

## 注

这份文档有意保持抽象。它描述的是思想，而非具体实现。确切的目录结构、schema 约定、页面格式、工具选择 —— 所有这些都取决于你的领域、你的偏好、以及你选择的 LLM。上面提到的每样东西都是可选的、模块化的 —— 选取有用的，忽略不需要的。例如：你的资料可能只有文本，完全不需要图片处理。你的 wiki 可能足够小，一个索引文件就够了，不需要搜索引擎。你可能不关心幻灯片而只需要 markdown 页面。你可能想要完全不同的一组输出格式。正确的用法是将这份文档分享给你的 LLM agent，然后一起协作实例化一个适合你需求的版本。这份文档的唯一职责是传达这个模式。你的 LLM 能搞定剩下的。
