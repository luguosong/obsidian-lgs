OAuth 2.0 授权框架使第三方应用能够获得对 HTTP 服务的有限访问权限，既可以通过协调资源所有者与 HTTP 服务之间的审批交互来代表资源所有者获取访问，也可以允许第三方应用以自身名义获取访问。本规范取代并废弃了 RFC 5849 中描述的 OAuth 1.0 协议。

## 1. 引言

在传统的客户端-服务器认证模型中，客户端使用资源所有者 (Resource Owner) 的凭证向服务器进行认证，从而请求受保护的资源。为了让第三方应用也能访问受限资源，资源所有者需要将其凭证共享给第三方。这带来了以下几个问题和限制：

- 第三方应用需要存储资源所有者的凭证以供后续使用，通常是明文密码。
- 服务器不得不支持密码认证，尽管密码本身存在固有的安全弱点。
- 第三方应用获得了对资源所有者受保护资源的过度访问权限，资源所有者无法限制访问的持续时间或资源范围。
- 资源所有者无法单独撤销某个第三方的访问权限，只能通过修改密码来同时撤销所有第三方的访问。
- 任何一个第三方应用被攻破，都会导致终端用户的密码以及该密码保护的所有数据泄露。


OAuth 通过引入授权层来解决这些问题，将客户端的角色与资源所有者的角色分离开来。在 OAuth 中，客户端请求访问由资源所有者控制、由资源服务器托管的资源，并获得一组与资源所有者不同的凭证。

客户端不再使用资源所有者的凭证来访问受保护资源，而是获取一个 access token——一个表示特定 scope、生命周期和其他访问属性的字符串。access token 由授权服务器在资源所有者的批准下签发给第三方客户端。客户端使用 access token 来访问资源服务器托管的受保护资源。

例如，终端用户（资源所有者）可以授权一个打印服务（客户端）访问存储在照片分享服务（资源服务器）上的受保护照片，而无需将用户名和密码共享给打印服务。相反，她直接向照片分享服务所信任的服务器（授权服务器）进行认证，该服务器会签发给打印服务具有特定委托范围的凭证（access token）。

> [!note] 协议范围
> 本规范专为 HTTP（[RFC2616]）设计。OAuth 在 HTTP 之外的任何协议上的使用不在本规范范围内。

OAuth 1.0 协议（[RFC5849]）最初作为一个信息性文档发布，是一个小型临时社区努力的成果。本标准跟踪规范建立在 OAuth 1.0 的部署经验之上，同时结合了更广泛的 IETF 社区收集的额外用例和可扩展性需求。OAuth 2.0 协议与 OAuth 1.0 不向后兼容。两个版本可以在网络中共存，实现可以选择同时支持两者。然而，本规范的意图是新实现应支持本文档中定义的 OAuth 2.0，而 OAuth 1.0 仅用于支持现有部署。OAuth 2.0 协议与 OAuth 1.0 协议共享极少的实现细节。熟悉 OAuth 1.0 的实现者在阅读本文档时不应对其结构和细节做任何假设。


### 1.1. 角色

OAuth 定义了四种角色：

**resource owner（资源所有者）**
   能够授予对受保护资源访问权限的实体。当资源所有者是个人时，称为终端用户 (end-user)。

**resource server（资源服务器）**
   托管受保护资源的服务器，能够接受并响应使用 access token 的受保护资源请求。

**client（客户端）**
   代表资源所有者并经其授权发起受保护资源请求的应用。术语"客户端"不暗示任何特定的实现特征（例如，应用是在服务器、桌面还是其他设备上运行）。

**authorization server（授权服务器）**
   在成功认证资源所有者并获得授权后，向客户端签发 access token 的服务器。

> [!info] 角色关系
> 授权服务器与资源服务器之间的交互不在本规范范围内。授权服务器可以与资源服务器是同一台服务器，也可以是独立的实体。一个授权服务器可以签发被多个资源服务器接受的 access token。


### 1.2. 协议流程

```
    +--------+                               +---------------+
    |        |---(A)---- 授权请求 ---------->|  资源所有者   |
    |        |                               |               |
    |        |<--(B)---- 授权许可 -----------|               |
    |        |                               +---------------+
    |        |
    |        |                               +---------------+
    |        |---(C)---- 授权许可 ---------->|  授权服务器   |
    | 客户端 |                               |               |
    |        |<--(D)------- Access Token ----|               |
    |        |                               +---------------+
    |        |
    |        |                               +---------------+
    |        |---(E)------- Access Token --->|  资源服务器   |
    |        |                               |               |
    |        |<--(F)---- 受保护资源 ---------|               |
    +--------+                               +---------------+

                     图 1：抽象协议流程
```

图 1 所示的抽象 OAuth 2.0 流程描述了四种角色之间的交互，包括以下步骤：

**(A)** 客户端向资源所有者请求授权。授权请求可以直接发送给资源所有者（如图所示），或者更推荐通过授权服务器作为中介间接进行。

**(B)** 客户端收到一个 authorization grant（授权许可），这是表示资源所有者授权的凭证，使用本规范定义的四种授权类型之一或使用扩展授权类型来表示。授权许可类型取决于客户端请求授权的方法以及授权服务器支持的类型。

**(C)** 客户端通过向授权服务器认证并出示授权许可来请求 access token。

**(D)** 授权服务器认证客户端并验证授权许可，如果有效，则签发 access token。


**(E)** 客户端向资源服务器请求受保护资源，并通过出示 access token 进行认证。

**(F)** 资源服务器验证 access token，如果有效，则处理请求。

客户端从资源所有者获取授权许可的首选方法（如步骤 (A) 和 (B) 所示）是使用授权服务器作为中介，如第 4.1 节的图 3 所示。

### 1.3. 授权许可 (Authorization Grant)

授权许可是表示资源所有者授权（访问其受保护资源）的凭证，客户端用它来获取 access token。本规范定义了四种授权类型——authorization code（授权码）、implicit（隐式）、resource owner password credentials（资源所有者密码凭证）和 client credentials（客户端凭证）——以及用于定义额外类型的[[扩展机制]]。

#### 1.3.1. 授权码 (Authorization Code)

授权码通过使用授权服务器作为客户端与资源所有者之间的中介来获取。客户端不是直接向资源所有者请求授权，而是将资源所有者引导至授权服务器（通过其 user-agent，如 [RFC2616] 所定义），授权服务器随后将资源所有者重定向回客户端并携带授权码。

在将资源所有者重定向回客户端并携带授权码之前，授权服务器会认证资源所有者并获得授权。由于资源所有者仅与授权服务器进行认证，因此资源所有者的凭证永远不会共享给客户端。

> [!tip] 安全优势
> 授权码提供了几个重要的安全优势，包括能够认证客户端，以及将 access token 直接传递给客户端而无需通过资源所有者的 user-agent，从而避免将其暴露给其他人（包括资源所有者本人）。

#### 1.3.2. 隐式授权 (Implicit)

隐式授权是简化的授权码流程，针对使用 JavaScript 等脚本语言在浏览器中实现的客户端进行了优化。在隐式流程中，不是向客户端签发授权码，而是直接向客户端签发 access token（作为资源所有者授权的结果）。授权类型是隐式的，因为不会签发中间凭证（如授权码）（然后用于获取 access token）。

在隐式[[授权流程]]中签发 access token 时，授权服务器不会认证客户端。在某些情况下，可以通过用于将 access token 传递给客户端的重定向 URI 来验证客户端身份。access token 可能会暴露给资源所有者或其他可以访问资源所有者 user-agent 的应用。

隐式授权提高了某些客户端（如浏览器内实现的应用）的响应速度和效率，因为它减少了获取 access token 所需的往返次数。然而，这种便利性应与使用隐式授权的安全影响进行权衡，特别是第 10.3 和 10.16 节中描述的安全问题，尤其是当授权码授权类型可用时。

#### 1.3.3. 资源所有者密码凭证 (Resource Owner Password Credentials)

资源所有者密码凭证（即用户名和密码）可以直接用作授权许可来获取 access token。此凭证仅应在资源所有者与客户端之间存在高度信任时使用（例如，客户端是设备操作系统的一部分或高权限应用），且当其他授权类型不可用时才使用。

尽管此授权类型要求客户端直接访问资源所有者凭证，但资源所有者凭证仅用于单次请求并交换为 access token。此授权类型可以通过将凭证交换为长期有效的 access token 或 refresh token，消除客户端存储资源所有者凭证以供后续使用的需要。

#### 1.3.4. 客户端凭证 (Client Credentials)

客户端凭证（或其他形式的客户端认证）可以在授权范围限于客户端控制的受保护资源，或与授权服务器预先约定的受保护资源时，用作授权许可。客户端凭证通常在客户端代表自身行事（客户端同时也是资源所有者）或基于与授权服务器预先约定的授权请求访问受保护资源时，作为授权许可使用。


### 1.4. Access Token

access token 是用于访问受保护资源的凭证。access token 是一个表示签发给客户端的授权的字符串。该字符串对客户端来说通常是不透明的。token 表示由资源所有者授予、由资源服务器和授权服务器执行的特定 scope 和访问持续时间。

token 可以表示用于检索授权信息的标识符，也可以以可验证的方式自包含授权信息（即由某些数据和签名组成的 token 字符串）。客户端使用 token 可能需要额外的认证凭证，这超出了本规范的范围。

access token 提供了一个抽象层，用资源服务器能理解的单个 token 替代了不同的授权构造（如用户名和密码）。这种抽象使得签发的 access token 可以比用于获取它们的授权许可更具限制性，同时消除了资源服务器理解多种认证方法的需要。

access token 可以根据资源服务器的安全要求具有不同的格式、结构和使用方法（如加密属性）。access token 的属性以及用于访问受保护资源的方法不在本规范范围内，由配套规范如 [RFC[[6750]]] 定义。

### 1.5. Refresh Token

refresh token 是用于获取 access token 的凭证。refresh token 由授权服务器签发给客户端，用于在当前 access token 失效或过期时获取新的 access token，或用于获取具有相同或更窄 scope 的额外 access token（access token 的生命周期可能比资源所有者授权的更短，权限也更少）。签发 refresh token 是可选的，由授权服务器决定。如果授权服务器签发了 refresh token，它会在签发 access token 时一并包含（即图 1 中的步骤 (D)）。

refresh token 是一个表示资源所有者授予客户端的授权的字符串。该字符串对客户端来说通常是不透明的。token 表示用于检索授权信息的标识符。与 access token 不同，refresh token 仅用于与授权服务器交互，永远不会发送到资源服务器。

```
+--------+                                           +---------------+
|        |---(A)-------- 授权许可 ------------------>|               |
|        |                                           |               |
|        |<--(B)----------- Access Token ------------|               |
|        |               & Refresh Token             |               |
|        |                                           |               |
|        |                            +----------+   |               |
|        |---(C)----- Access Token -->|          |   |               |
|        |                            |          |   |               |
|        |<--(D)---- 受保护资源 ------|          |   |               |
| 客户端 |                            |资源服务器|   |  授权服务器   |
|        |---(E)----- Access Token -->|          |   |               |
|        |                            |          |   |               |
|        |<--(F)--- 无效 Token 错误 --|          |   |               |
|        |                            +----------+   |               |
|        |                                           |               |
|        |---(G)----------- Refresh Token ---------->|               |
|        |                                           |               |
|        |<--(H)----------- Access Token ------------|               |
+--------+            & 可选 Refresh Token           +---------------+

             图 2：刷新过期的 Access Token
```

图 2 所示的流程包括以下步骤：

**(A)** 客户端通过向授权服务器认证并出示授权许可来请求 access token。

**(B)** 授权服务器认证客户端并验证授权许可，如果有效，签发 access token 和 refresh token。

**(C)** 客户端通过出示 access token 向资源服务器发起受保护资源请求。

**(D)** 资源服务器验证 access token，如果有效，处理请求。

**(E)** 步骤 (C) 和 (D) 重复执行直到 access token 过期。如果客户端知道 access token 已过期，直接跳到步骤 (G)；否则，发起另一次受保护资源请求。

**(F)** 由于 access token 无效，资源服务器返回无效 token 错误。


**(G)** 客户端通过向授权服务器认证并出示 refresh token 来请求新的 access token。客户端认证要求取决于客户端类型和授权服务器策略。

**(H)** 授权服务器认证客户端并验证 refresh token，如果有效，签发新的 access token（以及可选的新的 refresh token）。

步骤 (C)、(D)、(E) 和 (F) 不在本规范范围内，如第 7 节所述。

### 1.6. TLS 版本

当本规范使用传输层安全 (TLS) 时，合适的 TLS 版本会随时间变化，取决于广泛部署情况和已知安全漏洞。在撰写本文时，TLS 版本 1.2 [RFC[[5246]]] 是最新版本，但部署基础非常有限，可能无法随时用于实现。TLS 版本 1.0 [RFC2246] 是最广泛部署的版本，将提供最广泛的互操作性。

实现也可以支持满足其安全要求的其他传输层安全机制。

### 1.7. HTTP 重定向

本规范广泛使用 HTTP 重定向，其中客户端或授权服务器将资源所有者的 user-agent 引导到另一个目的地。虽然本规范中的示例使用 HTTP 302 状态码，但允许通过 user-agent 可用的任何其他方法完成此重定向，这被视为实现细节。

### 1.8. 互操作性


OAuth 2.0 提供了具有明确定义安全属性的丰富授权框架。然而，作为一个包含许多可选组件的丰富且高度可扩展的框架，本规范本身可能会产生大量不互操作的实现。

此外，本规范将一些必需的组件部分或完全未定义（例如，客户端注册、授权服务器功能、endpoint 发现）。没有这些组件，客户端必须针对特定的授权服务器和资源服务器进行手动和特定配置才能实现互操作。

本框架的设计明确期望未来将定义实现完整 Web 规模互操作性所需的规范性配置文件和扩展。

### 1.9. 符号约定

本规范中的关键词"MUST"、"MUST NOT"、"REQUIRED"、"SHALL"、"SHALL NOT"、"SHOULD"、"SHOULD NOT"、"RECOMMENDED"、"MAY"和"OPTIONAL"应按照 [RFC2119] 中的描述进行解释。

本规范使用 [RFC5234] 的增强巴科斯-诺尔范式 (ABNF) 表示法。此外，规则 URI-reference 引用自"统一资源标识符 (URI): 通用语法"[RFC[[3986]]]。

某些安全相关术语应按照 [RFC4949] 中定义的含义理解。这些术语包括但不限于"attack"、"authentication"、"authorization"、"certificate"、"confidentiality"、"credential"、"encryption"、"identity"、"sign"、"signature"、"trust"、"validate"和"verify"。

除非另有说明，所有协议参数名称和值都区分大小写。

## 2. 客户端注册

在启动协议之前，客户端需要向授权服务器注册。客户端向授权服务器注册的方式不在本规范范围内，通常涉及终端用户与 HTML 注册表单的交互。

客户端注册不要求客户端与授权服务器之间的直接交互。当授权服务器支持时，注册可以依赖其他方式来建立信任并获取所需的客户端属性（例如，重定向 URI、客户端类型）。例如，注册可以通过自签发或第三方签发的断言来完成，或由授权服务器通过可信通道执行客户端发现来完成。


注册客户端时，客户端开发者应当：

- 指定客户端类型，如第 2.1 节所述，
- 提供其客户端重定向 URI，如第 3.1.2 节所述，
- 包含授权服务器要求的任何其他信息（例如，应用名称、网站、描述、徽标图像、法律条款的接受）。

### 2.1. 客户端类型

OAuth 根据客户端与授权服务器安全认证的能力（即维护其客户端凭证机密性的能力）定义了两种客户端类型：

**confidential（机密客户端）**
   能够维护其凭证机密性的客户端（例如，在安全服务器上实现的客户端，对客户端凭证的访问受限），或能够使用其他方式进行安全客户端认证的客户端。

**public（公共客户端）**
   无法维护其凭证机密性的客户端（例如，在资源所有者使用的设备上执行的客户端，如安装的原生应用或基于 Web 浏览器的应用），且无法通过任何其他方式进行安全客户端认证。

客户端类型的认定基于授权服务器对安全认证的定义及其可接受的客户端凭证暴露水平。授权服务器不应对客户端类型做出假设。

客户端可以实现为一组分布式组件，每个组件具有不同的客户端类型和安全上下文（例如，一个分布式客户端同时包含机密的服务器端组件和公共的浏览器端组件）。如果授权服务器不支持此类客户端或未提供关于其注册的指导，客户端应将每个组件注册为单独的客户端。


本规范围绕以下客户端配置文件进行设计：

**Web 应用 (web application)**
   Web 应用是在 [[Web 服务器]]上运行的机密客户端。资源所有者通过在资源所有者设备上的 user-agent 中渲染的 HTML 用户界面访问客户端。客户端凭证以及签发给客户端的任何 access token 都存储在 [[Web 服务器]]上，不会暴露给资源所有者或被其访问。

**基于 User-Agent 的应用 (user-agent-based application)**
   基于 user-agent 的应用是一种公共客户端，其客户端代码从 [[Web 服务器]]下载并在资源所有者设备上的 user-agent（如 Web 浏览器）中执行。协议数据和凭证容易被资源所有者访问（通常也是可见的）。由于此类应用驻留在 user-agent 内，它们可以在请求授权时无缝利用 user-agent 的功能。

**原生应用 (native application)**
   原生应用是在资源所有者设备上安装和执行的公共客户端。协议数据和凭证可被资源所有者访问。假设应用中包含的任何客户端认证凭证都可以被提取。另一方面，动态签发的凭证（如 access token 或 refresh token）可以获得可接受级别的保护。至少，这些凭证受到保护，免受应用可能交互的恶意服务器的攻击。在某些平台上，这些凭证可能受到保护，免受驻留在同一设备上的其他应用的攻击。

### 2.2. 客户端标识符

授权服务器向已注册的客户端签发一个客户端标识符——一个表示客户端提供的注册信息的唯一字符串。客户端标识符不是秘密；它会暴露给资源所有者，且不得单独用于客户端认证。客户端标识符在授权服务器范围内唯一。

客户端标识符字符串的大小本规范未作定义。客户端应避免对标识符大小做出假设。授权服务器应当记录其签发的任何标识符的大小。


### 2.3. 客户端认证

如果客户端类型是机密的，客户端和授权服务器需要建立适合授权服务器安全要求的客户端认证方法。授权服务器可以接受任何满足其安全要求的客户端认证形式。

机密客户端通常会获得（或建立）一组用于向授权服务器认证的客户端凭证（如密码、公私钥对）。

授权服务器可以与公共客户端建立客户端认证方法。然而，授权服务器不得依赖公共客户端认证来识别客户端。

客户端在每个请求中不得使用超过一种认证方法。

#### 2.3.1. 客户端密码

拥有客户端密码的客户端可以使用 [RFC2617] 定义的 HTTP Basic 认证方案向授权服务器进行认证。客户端标识符按照附录 B 使用"application/x-www-form-urlencoded"编码算法进行编码，编码值用作用户名；客户端密码使用相同算法编码并用作密码。授权服务器必须支持 HTTP Basic 认证方案来认证被签发了客户端密码的客户端。

例如（为了显示目的添加了额外换行）：


```http
Authorization: Basic czZCaGRSa3F0Mzo3RmpmcDBaQnIxS3REUmJuZlZkbUl3
```

或者，授权服务器可以支持在请求体中使用以下参数包含客户端凭证：

**client_id**
      必需。按照第 2.2 节描述的注册过程中签发给客户端的标识符。

**client_secret**
      必需。客户端密钥。如果客户端密钥是空字符串，客户端可以省略此参数。

不推荐使用这两个参数在请求体中包含客户端凭证，且应限于无法直接使用 HTTP Basic 认证方案（或其他基于密码的 HTTP 认证方案）的客户端。这些参数只能在请求体中传输，不得包含在请求 URI 中。

例如，使用 body 参数刷新 access token 的请求（第 6 节）（为了显示目的添加了额外换行）：

```http
POST /token HTTP/1.1
Host: server.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
&client_id=s6BhdRkqt3&client_secret=7Fjfp0ZBr1KtDRbnfVdmIw
```

授权服务器在发送使用密码认证的请求时，必须要求按照第 1.6 节使用 TLS。

由于此客户端认证方法涉及密码，授权服务器必须保护使用此方法的任何 endpoint 免受暴力攻击。

#### 2.3.2. 其他认证方法

授权服务器可以支持任何满足其安全要求的合适 HTTP 认证方案。使用其他认证方法时，授权服务器必须在客户端标识符（注册记录）和认证方案之间建立映射。

### 2.4. 未注册客户端

本规范不排除使用未注册客户端。然而，此类客户端的使用不在本规范范围内，需要额外的安全分析和互操作性影响审查。


## 3. 协议端点

授权过程使用两个授权服务器端点 (HTTP resources)：

- **Authorization endpoint（授权端点）** - 客户端通过 user-agent 重定向使用此端点从资源所有者获取授权。
- **Token endpoint（令牌端点）** - 客户端使用此端点将授权许可交换为 access token，通常需要客户端认证。

以及一个客户端端点：

- **Redirection endpoint（重定向端点）** - 授权服务器使用此端点通过资源所有者的 user-agent 将包含授权凭证的响应返回给客户端。

并非所有授权类型都使用两个端点。扩展授权类型可以根据需要定义额外的端点。

### 3.1. 授权端点

授权端点用于与资源所有者交互并获取授权许可。授权服务器必须首先验证资源所有者的身份。授权服务器认证资源所有者的方式（如用户名和密码登录、session cookie）不在本规范范围内。

客户端获取授权端点位置的方式不在本规范范围内，但该位置通常在服务文档中提供。

端点 URI 可以包含按照"application/x-www-form-urlencoded"格式化（按附录 B）的查询组件（[RFC[[3986]]] 第 3.4 节），在添加额外查询参数时必须保留该组件。端点 URI 不得包含片段组件。

由于对授权端点的请求会导致[[用户认证]]和明文凭据的传输（在 HTTP 响应中），授权服务器在发送请求到授权端点时必须按照第 1.6 节要求使用 TLS。

授权服务器必须支持在授权端点使用 HTTP "GET" 方法 [RFC2616]，也可以支持 "POST" 方法。


不带值的参数必须视为已从请求中省略。授权服务器必须忽略无法识别的请求参数。请求和响应参数不得多次包含。

#### 3.1.1. 响应类型

授权端点由授权码授权类型和隐式授权类型流程使用。客户端使用以下参数通知授权服务器所需的授权类型：

**response_type**
      必需。值必须为"code"（按第 4.1.1 节请求授权码）、"token"（按第 4.2.1 节请求 access token，隐式授权）或按第 8.4 节定义的已注册扩展值。

扩展响应类型可以包含空格分隔 (%x20) 的值列表，值的顺序无关紧要（例如，响应类型"a b"与"b a"相同）。此类复合响应类型的含义由各自的规范定义。

如果授权请求缺少"response_type"参数，或者响应类型未被理解，授权服务器必须如第 4.1.2.1 节所述返回错误响应。

#### 3.1.2. 重定向端点

在与资源所有者的交互完成后，授权服务器将资源所有者的 user-agent 引导回客户端。授权服务器将 user-agent 重定向到客户端的重定向端点，该端点在客户端注册过程中或在发起授权请求时与授权服务器建立。

重定向端点 URI 必须是由 [RFC[[3986]]] 第 4.3 节定义的绝对 URI。端点 URI 可以包含按照"application/x-www-form-urlencoded"格式化（按附录 B）的查询组件（[RFC[[3986]]] 第 3.4 节），在添加额外查询参数时必须保留该组件。端点 URI 不得包含片段组件。


##### 3.1.2.1. 端点请求机密性

当请求的响应类型为"code"或"token"，或当重定向请求会导致敏感凭证在开放网络上传输时，重定向端点应当按照第 1.6 节要求使用 TLS。本规范不强制使用 TLS，因为在撰写本文时，要求客户端部署 TLS 对许多客户端开发者来说是重大障碍。如果 TLS 不可用，授权服务器应当在重定向之前向资源所有者警告不安全的端点（例如，在授权请求期间显示消息）。

> [!warning] 安全警告
> 缺乏传输层安全可能对客户端安全及其被授权访问的受保护资源产生严重影响。当授权过程被客户端用作委托式终端[[用户认证]]的形式时（如第三方登录服务），传输层安全的使用尤为关键。

##### 3.1.2.2. 注册要求

授权服务器必须要求以下客户端注册其重定向端点：

- 公共客户端。
- 使用隐式授权类型的机密客户端。

授权服务器应当要求所有客户端在使用授权端点之前注册其重定向端点。

授权服务器应当要求客户端提供完整的重定向 URI（客户端可以使用"state"请求参数实现每次请求的自定义）。如果要求注册完整的重定向 URI 不可能，授权服务器应当要求注册 URI 的 scheme、authority 和 path（允许客户端在请求授权时仅动态变化重定向 URI 的查询组件）。

授权服务器可以允许客户端注册多个重定向端点。

缺乏重定向 URI 注册要求可能使攻击者能够将授权端点用作开放重定向器，如第 10.15 节所述。

##### 3.1.2.3. 动态配置

如果已注册多个重定向 URI，或仅注册了重定向 URI 的部分，或未注册重定向 URI，客户端必须在授权请求中使用"redirect_uri"请求参数包含重定向 URI。

当授权请求中包含重定向 URI 时，如果已注册了任何重定向 URI，授权服务器必须将接收到的值与至少一个已注册的重定向 URI（或 URI 组件）进行比较和匹配，如 [RFC[[3986]]] 第 6 节所定义。如果客户端注册包含完整的重定向 URI，授权服务器必须使用 [RFC[[3986]]] 第 6.2.1 节定义的简单字符串比较来比较两个 URI。

##### 3.1.2.4. 无效端点

如果授权请求因缺少、无效或不匹配的重定向 URI 而验证失败，授权服务器应当向资源所有者通知错误，并且不得自动将 user-agent 重定向到无效的重定向 URI。

##### 3.1.2.5. 端点内容

重定向请求到客户端的端点通常会导致 HTML 文档响应，由 user-agent 处理。如果 HTML 响应直接作为重定向请求的结果提供服务，HTML 文档中包含的任何脚本将拥有对重定向 URI 及其包含的凭证的完全访问权限。

客户端不应当在其重定向端点响应中包含任何第三方脚本（如第三方分析、社交插件、广告网络）。相反，应当从 URI 中提取凭证并再次将 user-agent 重定向到另一个端点，不暴露凭证（在 URI 或其他地方）。如果包含第三方脚本，客户端必须确保其自己的脚本（用于从 URI 中提取和移除凭证）将首先执行。

### 3.2. Token 端点

Token 端点由客户端用于通过出示其授权许可或 refresh token 来获取 access token。Token 端点用于除隐式授权类型之外的每种授权类型（因为隐式授权类型直接签发 access token）。


客户端获取 token 端点位置的方式不在本规范范围内，但该位置通常在服务文档中提供。

端点 URI 可以包含按照"application/x-www-form-urlencoded"格式化（按附录 B）的查询组件（[RFC[[3986]]] 第 3.4 节），在添加额外查询参数时必须保留该组件。端点 URI 不得包含片段组件。

由于对 token 端点的请求会导致明文凭据的传输（在 HTTP 请求和响应中），授权服务器在发送请求到 token 端点时必须按照第 1.6 节要求使用 TLS。

客户端在发起 access token 请求时必须使用 HTTP "POST" 方法。

不带值的参数必须视为已从请求中省略。授权服务器必须忽略无法识别的请求参数。请求和响应参数不得多次包含。

#### 3.2.1. 客户端认证

机密客户端或其他被签发了客户端凭证的客户端在向 token 端点发起请求时，必须按照第 2.3 节向授权服务器进行认证。客户端认证用于：

- 强制将 refresh token 和授权码绑定到它们被签发给的客户端。当授权码通过不安全通道传输到重定向端点，或重定向 URI 未完整注册时，客户端认证至关重要。
- 通过禁用客户端或更改其凭证来从被攻破的客户端中恢复，从而防止攻击者滥用被盗的 refresh token。更改一组客户端凭证比撤销整套 refresh token 显著更快。
- 实施认证管理最佳实践，要求定期轮换凭证。轮换整套 refresh token 可能具有挑战性，而轮换单组客户端凭证则显著更容易。


客户端在向 token 端点发送请求时可以使用"client_id"请求参数来标识自身。在向 token 端点的"authorization_code" "grant_type"请求中，未经认证的客户端必须发送其"client_id"，以防止自身无意中接受旨在给具有不同"client_id"的客户端的授权码。这保护客户端免受授权码替换攻击。（它不会为受保护资源提供额外的安全性。）

### 3.3. Access Token 范围

授权端点和 token 端点允许客户端使用"scope"请求参数指定访问请求的范围。相应地，授权服务器使用"scope"响应参数通知客户端所签发 access token 的范围。

scope 参数的值表示为以空格分隔的、区分大小写的字符串列表。这些字符串由授权服务器定义。如果值包含多个空格分隔的字符串，它们的顺序无关紧要，每个字符串为请求的 scope 增加额外的访问范围。

```
scope       = scope-token *( SP scope-token )
scope-token = 1*( %x21 / %x23-5B / %x5D-7E )
```

授权服务器可以基于授权服务器策略或资源所有者的指示，完全或部分忽略客户端请求的 scope。如果签发的 access token scope 与客户端请求的不同，授权服务器必须包含"scope"响应参数以通知客户端实际授予的 scope。

如果客户端在请求授权时省略了 scope 参数，授权服务器必须使用预定义的默认值处理请求，或者以无效 scope 为由拒绝请求。授权服务器应当记录其 scope 要求和默认值（如果已定义）。

## 4. 获取授权

为了请求 access token，客户端需要从资源所有者获取授权。授权以授权许可的形式表示，客户端用它来请求 access token。OAuth 定义了四种授权类型：authorization code、implicit、resource owner password credentials 和 client credentials。它还提供了定义额外授权类型的[[扩展机制]]。


### 4.1. 授权码授权 (Authorization Code Grant)

授权码授权类型用于获取 access token 和 refresh token，针对机密客户端进行了优化。由于这是基于重定向的流程，客户端必须能够与资源所有者的 user-agent（通常是 Web 浏览器）交互，并且能够接收来自授权服务器的传入请求（通过重定向）。

```
  +----------+
  | 资源所有者|
  |          |
  +----------+
       ^
       |
      (B)
  +----|-----+          客户端标识符            +---------------+
  |         -+---(A)--- & 重定向 URI ---------->|               |
  |  User-   |                                  |  授权服务器   |
  |  Agent  -+---(B)--- 用户认证 -------------->|               |
  |          |                                  |               |
  |         -+---(C)--- 授权码 ----------------<|               |
  +-|----|---+                                  +---------------+
    |    |                                         ^      v
   (A)  (C)                                        |      |
    |    |                                         |      |
    ^    v                                         |      |
  +---------+                                      |      |
  |         |>---(D)--- 授权码 --------------------'      |
  |  客户端 |          & 重定向 URI                         |
  |         |                                             |
  |         |<---(E)------ Access Token ------------------'
  +---------+       (w/ 可选 Refresh Token)

  注意：步骤 (A)、(B) 和 (C) 的连线在通过 user-agent 时分为两部分。

                     图 3：授权码流程
```


图 3 所示的流程包括以下步骤：

**(A)** 客户端通过将资源所有者的 user-agent 引导到授权端点来启动流程。客户端包含其客户端标识符、请求的 scope、本地状态和一个重定向 URI，授权服务器将在访问被授予（或拒绝）后通过该 URI 将 user-agent 引导回来。

**(B)** 授权服务器认证资源所有者（通过 user-agent）并确定资源所有者是授予还是拒绝客户端的访问请求。

**(C)** 假设资源所有者授予访问权限，授权服务器使用之前提供的重定向 URI（在请求中或在客户端注册期间）将 user-agent 重定向回客户端。重定向 URI 包含授权码和客户端之前提供的任何本地状态。

**(D)** 客户端通过包含上一步收到的授权码向授权服务器的 token 端点请求 access token。发起请求时，客户端向授权服务器进行认证。客户端包含用于获取授权码的重定向 URI 以供验证。

**(E)** 授权服务器认证客户端，验证授权码，并确保接收到的重定向 URI 与步骤 (C) 中用于重定向客户端的 URI 匹配。如果有效，授权服务器返回 access token 和可选的 refresh token。

#### 4.1.1. 授权请求

客户端通过按照"application/x-www-form-urlencoded"格式（按附录 B）将以下参数添加到授权端点 URI 的查询组件来构造请求 URI：

**response_type**
      必需。值必须设置为"code"。

**client_id**
      必需。如第 2.2 节所述的客户端标识符。

**redirect_uri**
      可选。如第 3.1.2 节所述。


**scope**
      可选。如第 3.3 节所述的访问请求范围。

**state**
      推荐。由客户端用于维护请求和回调之间状态的不透明值。授权服务器在将 user-agent 重定向回客户端时包含此值。该参数应当用于防止跨站请求伪造 (CSRF)，如第 10.12 节所述。

客户端使用 HTTP 重定向响应或通过 user-agent 可用的其他方式将资源所有者引导到构造的 URI。

例如，客户端使用 TLS 将 user-agent 引导到发起以下 HTTP 请求（为了显示目的添加了额外换行）：

```http
GET /authorize?response_type=code&client_id=s6BhdRkqt3&state=xyz
    &redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb HTTP/1.1
Host: server.example.com
```

授权服务器验证请求以确保所有必需参数都存在且有效。如果请求有效，授权服务器认证资源所有者并获取授权决定（通过询问资源所有者或通过其他方式建立批准）。

当决定确定后，授权服务器使用 HTTP 重定向响应或通过 user-agent 可用的其他方式将 user-agent 引导到提供的客户端重定向 URI。

#### 4.1.2. 授权响应

如果资源所有者授予访问请求，授权服务器签发授权码，通过按照"application/x-www-form-urlencoded"格式（按附录 B）将以下参数添加到重定向 URI 的查询组件来传递给客户端：

**code**
      必需。授权服务器生成的授权码。授权码必须在签发后不久过期以降低泄露风险。建议最大授权码生命周期为 10 分钟。客户端不得使用授权码超过一次。如果授权码被使用超过一次，授权服务器必须拒绝请求并应当（如果可能）撤销基于该授权码之前签发的所有 token。授权码绑定到客户端标识符和重定向 URI。

**state**
      如果客户端授权请求中存在"state"参数，则为必需。从客户端接收到的确切值。

例如，授权服务器通过发送以下 HTTP 响应来重定向 user-agent：

```http
HTTP/1.1 302 Found
Location: https://client.example.com/cb?code=SplxlOBeZQQYbYS6WxSbIA
          &state=xyz
```

客户端必须忽略无法识别的响应参数。授权码字符串大小本规范未作定义。客户端应避免对 code 值大小做出假设。授权服务器应当记录其签发的任何值的大小。

##### 4.1.2.1. 错误响应

如果请求因缺少、无效或不匹配的重定向 URI 而失败，或客户端标识符缺失或无效，授权服务器应当向资源所有者通知错误，并且不得自动将 user-agent 重定向到无效的重定向 URI。

如果资源所有者拒绝访问请求，或请求因缺少或无效的重定向 URI 以外的原因失败，授权服务器通过按照"application/x-www-form-urlencoded"格式（按附录 B）将以下参数添加到重定向 URI 的查询组件来通知客户端：

**error**
      必需。以下列表中的一个 ASCII [USASCII] 错误码：

      `invalid_request`
            请求缺少必需参数、包含无效参数值、多次包含同一参数，或格式错误。


      `unauthorized_client`
            客户端未被授权使用此方法请求授权码。

      `access_denied`
            资源所有者或授权服务器拒绝了请求。

      `unsupported_response_type`
            授权服务器不支持使用此方法获取授权码。

      `invalid_scope`
            请求的 scope 无效、未知或格式错误。

      `server_error`
            授权服务器遇到了阻止其完成请求的意外情况。（需要此错误码是因为无法通过 HTTP 重定向向客户端返回 500 Internal Server Error HTTP 状态码。）

      `temporarily_unavailable`
            授权服务器由于临时过载或维护目前无法处理请求。（需要此错误码是因为无法通过 HTTP 重定向向客户端返回 503 Service Unavailable HTTP 状态码。）

      "error"参数的值不得包含集合 %x20-21 / %x23-5B / %x5D-7E 之外的字符。

**error_description**
      可选。提供额外信息的人类可读 ASCII [USASCII] 文本，用于帮助客户端开发者理解发生的错误。
      "error_description"参数的值不得包含集合 %x20-21 / %x23-5B / %x5D-7E 之外的字符。

**error_uri**
      可选。标识包含错误信息的人类可读网页的 URI，用于向客户端开发者提供关于错误的额外信息。
      "error_uri"参数的值必须符合 URI-reference 语法，因此不得包含集合 %x21 / %x23-5B / %x5D-7E 之外的字符。


**state**
      如果客户端授权请求中存在"state"参数，则为必需。从客户端接收到的确切值。

例如，授权服务器通过发送以下 HTTP 响应来重定向 user-agent：

```http
HTTP/1.1 302 Found
Location: https://client.example.com/cb?error=access_denied&state=xyz
```

#### 4.1.3. Access Token 请求

客户端通过在 HTTP 请求实体主体中使用"application/x-www-form-urlencoded"格式（按附录 B）和 UTF-8 字符编码发送以下参数，向 token 端点发起请求：

**grant_type**
      必需。值必须设置为"authorization_code"。

**code**
      必需。从授权服务器收到的授权码。

**redirect_uri**
      如果第 4.1.1 节所述的授权请求中包含了"redirect_uri"参数，则为必需，且它们的值必须相同。

**client_id**
      如果客户端未按第 3.2.1 节向授权服务器进行认证，则为必需。

如果客户端类型是机密的或客户端被签发了客户端凭证（或被分配了其他认证要求），客户端必须按第 3.2.1 节向授权服务器进行认证。


例如，客户端使用 TLS 发起以下 HTTP 请求（为了显示目的添加了额外换行）：

```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb
```

授权服务器必须：

- 对机密客户端或任何被签发了客户端凭证（或具有其他认证要求）的客户端要求客户端认证，
- 如果包含客户端认证，则认证客户端，
- 确保授权码是签发给已认证的机密客户端的，或者如果客户端是公共的，确保授权码是签发给请求中的"client_id"的，
- 验证授权码是否有效，
- 确保如果第 4.1.1 节所述的初始授权请求中包含了"redirect_uri"参数，则"redirect_uri"参数存在，如果包含则确保其值相同。

#### 4.1.4. Access Token 响应

如果 access token 请求有效且已授权，授权服务器如第 5.1 节所述签发 access token 和可选的 refresh token。如果请求的客户端认证失败或无效，授权服务器如第 5.2 节所述返回错误响应。


成功响应示例：

```http
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "access_token":"2YotnFZFEjr1zCsicMWpAA",
  "token_type":"example",
  "expires_in":3600,
  "refresh_token":"tGzv3JOkF0XG5Qx2TlKWIA",
  "example_parameter":"example_value"
}
```

### 4.2. 隐式授权 (Implicit Grant)

隐式授权类型用于获取 access token（不支持签发 refresh token），针对已知使用特定重定向 URI 的公共客户端进行了优化。这些客户端通常使用 JavaScript 等脚本语言在浏览器中实现。

由于这是基于重定向的流程，客户端必须能够与资源所有者的 user-agent（通常是 Web 浏览器）交互，并且能够接收来自授权服务器的传入请求（通过重定向）。

与授权码授权类型（客户端分别发起授权请求和 access token 请求）不同，隐式授权中客户端在授权请求的结果中直接收到 access token。

隐式授权类型不包含客户端认证，依赖于资源所有者的存在和重定向 URI 的注册。由于 access token 被编码到重定向 URI 中，它可能会暴露给资源所有者和驻留在同一设备上的其他应用。


```
  +----------+
  | 资源所有者|
  |          |
  +----------+
       ^
       |
      (B)
  +----|-----+         客户端标识符             +---------------+
  |         -+---(A)--- & 重定向 URI ---------->|               |
  |  User-   |                                  |  授权服务器   |
  |  Agent  -+---(B)--- 用户认证 -------------->|               |
  |          |                                  |               |
  |          |<---(C)--- 重定向 URI -----------<|               |
  |          |           带 Access Token        +---------------+
  |          |            在 Fragment 中
  |          |                                  +---------------+
  |          |----(D)--- 重定向 URI ----------->|    Web 托管   |
  |          |             不带 Fragment        |     客户端    |
  |          |                                  |      资源     |
  |     (F)  |<---(E)---------- 脚本 ----------<|               |
  |          |                                  +---------------+
  +-|--------+
    |    |
   (A)  (G) Access Token
    |    |
    ^    v
  +---------+
  |         |
  |  客户端 |
  |         |
  +---------+

  注意：步骤 (A) 和 (B) 的连线在通过 user-agent 时分为两部分。

                     图 4：隐式授权流程
```


图 4 所示的流程包括以下步骤：

**(A)** 客户端通过将资源所有者的 user-agent 引导到授权端点来启动流程。客户端包含其客户端标识符、请求的 scope、本地状态和一个重定向 URI，授权服务器将在访问被授予（或拒绝）后通过该 URI 将 user-agent 引导回来。

**(B)** 授权服务器认证资源所有者（通过 user-agent）并确定资源所有者是授予还是拒绝客户端的访问请求。

**(C)** 假设资源所有者授予访问权限，授权服务器使用之前提供的重定向 URI 将 user-agent 重定向回客户端。重定向 URI 在 URI 片段中包含 access token。

**(D)** user-agent 按照重定向指令向 Web 托管的客户端资源发起请求（按 [RFC2616] 不包含片段）。user-agent 在本地保留片段信息。

**(E)** Web 托管的客户端资源返回一个网页（通常是包含嵌入脚本的 HTML 文档），能够访问包含 user-agent 保留片段的完整重定向 URI，并提取片段中包含的 access token（和其他参数）。

**(F)** user-agent 在本地执行 Web 托管客户端资源提供的脚本，提取 access token。

**(G)** user-agent 将 access token 传递给客户端。

关于使用隐式授权的背景信息参见第 1.3.2 和 9 节。使用隐式授权时的重要安全考虑参见第 10.3 和 10.16 节。

#### 4.2.1. 授权请求

客户端通过按照"application/x-www-form-urlencoded"格式（按附录 B）将以下参数添加到授权端点 URI 的查询组件来构造请求 URI：

**response_type**
      必需。值必须设置为"token"。

**client_id**
      必需。如第 2.2 节所述的客户端标识符。


**redirect_uri**
      可选。如第 3.1.2 节所述。

**scope**
      可选。如第 3.3 节所述的访问请求范围。

**state**
      推荐。由客户端用于维护请求和回调之间状态的不透明值。授权服务器在将 user-agent 重定向回客户端时包含此值。该参数应当用于防止跨站请求伪造 (CSRF)，如第 10.12 节所述。

客户端使用 HTTP 重定向响应或通过 user-agent 可用的其他方式将资源所有者引导到构造的 URI。

例如，客户端使用 TLS 将 user-agent 引导到发起以下 HTTP 请求（为了显示目的添加了额外换行）：

```http
GET /authorize?response_type=token&client_id=s6BhdRkqt3&state=xyz
    &redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb HTTP/1.1
Host: server.example.com
```

授权服务器验证请求以确保所有必需参数都存在且有效。授权服务器必须验证它将重定向 access token 的重定向 URI 与客户端注册的重定向 URI 匹配，如第 3.1.2 节所述。

如果请求有效，授权服务器认证资源所有者并获取授权决定（通过询问资源所有者或通过其他方式建立批准）。

当决定确定后，授权服务器使用 HTTP 重定向响应或通过 user-agent 可用的其他方式将 user-agent 引导到提供的客户端重定向 URI。


#### 4.2.2. Access Token 响应

如果资源所有者授予访问请求，授权服务器签发 access token，通过按照"application/x-www-form-urlencoded"格式（按附录 B）将以下参数添加到重定向 URI 的片段组件来传递给客户端：

**access_token**
      必需。授权服务器签发的 access token。

**token_type**
      必需。如第 7.1 节所述的已签发 token 类型。值不区分大小写。

**expires_in**
      推荐。access token 的生命周期（秒）。例如，值"3600"表示 access token 从响应生成时间起一小时后过期。如果省略，授权服务器应当通过其他方式提供过期时间或记录默认值。

**scope**
      可选，如果与客户端请求的 scope 相同；否则为必需。如第 3.3 节所述的 access token 的范围。

**state**
      如果客户端授权请求中存在"state"参数，则为必需。从客户端接收到的确切值。

授权服务器不得签发 refresh token。

例如，授权服务器通过发送以下 HTTP 响应来重定向 user-agent（为了显示目的添加了额外换行）：

```http
HTTP/1.1 302 Found
Location: http://example.com/cb#access_token=2YotnFZFEjr1zCsicMWpAA
          &state=xyz&token_type=example&expires_in=3600
```

开发者应注意，某些 user-agent 不支持在 HTTP "Location" 响应头字段中包含片段组件。此类客户端需要使用 3xx 重定向响应以外的方法来重定向客户端——例如，返回一个 HTML 页面，其中包含一个链接到重定向 URI 的"继续"按钮。


客户端必须忽略无法识别的响应参数。access token 字符串大小本规范未作定义。客户端应避免对值大小做出假设。授权服务器应当记录其签发的任何值的大小。

##### 4.2.2.1. 错误响应

如果请求因缺少、无效或不匹配的重定向 URI 而失败，或客户端标识符缺失或无效，授权服务器应当向资源所有者通知错误，并且不得自动将 user-agent 重定向到无效的重定向 URI。

如果资源所有者拒绝访问请求，或请求因缺少或无效的重定向 URI 以外的原因失败，授权服务器通过按照"application/x-www-form-urlencoded"格式（按附录 B）将以下参数添加到重定向 URI 的片段组件来通知客户端：

**error**
      必需。以下列表中的一个 ASCII [USASCII] 错误码：

      `invalid_request`
            请求缺少必需参数、包含无效参数值、多次包含同一参数，或格式错误。

      `unauthorized_client`
            客户端未被授权使用此方法请求 access token。

      `access_denied`
            资源所有者或授权服务器拒绝了请求。

      `unsupported_response_type`
            授权服务器不支持使用此方法获取 access token。

      `invalid_scope`
            请求的 scope 无效、未知或格式错误。


      `server_error`
            授权服务器遇到了阻止其完成请求的意外情况。（需要此错误码是因为无法通过 HTTP 重定向向客户端返回 500 Internal Server Error HTTP 状态码。）

      `temporarily_unavailable`
            授权服务器由于临时过载或维护目前无法处理请求。（需要此错误码是因为无法通过 HTTP 重定向向客户端返回 503 Service Unavailable HTTP 状态码。）

      "error"参数的值不得包含集合 %x20-21 / %x23-5B / %x5D-7E 之外的字符。

**error_description**
      可选。提供额外信息的人类可读 ASCII [USASCII] 文本，用于帮助客户端开发者理解发生的错误。
      "error_description"参数的值不得包含集合 %x20-21 / %x23-5B / %x5D-7E 之外的字符。

**error_uri**
      可选。标识包含错误信息的人类可读网页的 URI，用于向客户端开发者提供关于错误的额外信息。
      "error_uri"参数的值必须符合 URI-reference 语法，因此不得包含集合 %x21 / %x23-5B / %x5D-7E 之外的字符。

**state**
      如果客户端授权请求中存在"state"参数，则为必需。从客户端接收到的确切值。

例如，授权服务器通过发送以下 HTTP 响应来重定向 user-agent：

```http
HTTP/1.1 302 Found
Location: https://client.example.com/cb#error=access_denied&state=xyz
```

### 4.3. 资源所有者密码凭证授权 (Resource Owner Password Credentials Grant)

资源所有者密码凭证授权类型适用于资源所有者与客户端之间存在信任关系的场景，例如设备操作系统或高权限应用。授权服务器在启用此授权类型时应特别谨慎，仅在其他流程不可行时才允许使用。

此授权类型适用于能够获取资源所有者凭证（用户名和密码，通常通过交互式表单）的客户端。它也用于将使用直接认证方案（如 HTTP Basic 或 Digest 认证）的现有客户端迁移到 OAuth，通过将存储的凭证转换为 access token。

```
  +----------+
  | 资源所有者|
  |          |
  +----------+
       v
       |    资源所有者
      (A) 密码凭证
       |
       v
  +---------+                                  +---------------+
  |         |>--(B)---- 资源所有者 ----------->|               |
  |         |         密码凭证                 |  授权服务器   |
  |  客户端 |                                  |               |
  |         |<--(C)---- Access Token ---------<|               |
  |         |    (w/ 可选 Refresh Token)       |               |
  +---------+                                  +---------------+

          图 5：资源所有者密码凭证流程
```

图 5 所示的流程包括以下步骤：

**(A)** 资源所有者向客户端提供其用户名和密码。

**(B)** 客户端通过包含从资源所有者收到的凭证向授权服务器的 token 端点请求 access token。发起请求时，客户端向授权服务器进行认证。

**(C)** 授权服务器认证客户端并验证资源所有者凭证，如果有效，签发 access token。


#### 4.3.1. 授权请求和响应

客户端获取资源所有者凭证的方式不在本规范范围内。客户端在获取 access token 后必须丢弃凭证。

#### 4.3.2. Access Token 请求

客户端通过在 HTTP 请求实体主体中使用"application/x-www-form-urlencoded"格式（按附录 B）和 UTF-8 字符编码添加以下参数，向 token 端点发起请求：

**grant_type**
      必需。值必须设置为"password"。

**username**
      必需。资源所有者的用户名。

**password**
      必需。资源所有者的密码。

**scope**
      可选。如第 3.3 节所述的访问请求范围。

如果客户端类型是机密的或客户端被签发了客户端凭证（或被分配了其他认证要求），客户端必须按第 3.2.1 节向授权服务器进行认证。

例如，客户端使用传输层安全发起以下 HTTP 请求（为了显示目的添加了额外换行）：

```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=password&username=johndoe&password=A3ddj3w
```


授权服务器必须：

- 对机密客户端或任何被签发了客户端凭证（或具有其他认证要求）的客户端要求客户端认证，
- 如果包含客户端认证，则认证客户端，
- 使用其现有的密码验证算法验证资源所有者密码凭证。

由于此 access token 请求使用了资源所有者的密码，授权服务器必须保护端点免受暴力攻击（例如，使用速率限制或生成警报）。

#### 4.3.3. Access Token 响应

如果 access token 请求有效且已授权，授权服务器如第 5.1 节所述签发 access token 和可选的 refresh token。如果请求的客户端认证失败或无效，授权服务器如第 5.2 节所述返回错误响应。

成功响应示例：

```http
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "access_token":"2YotnFZFEjr1zCsicMWpAA",
  "token_type":"example",
  "expires_in":3600,
  "refresh_token":"tGzv3JOkF0XG5Qx2TlKWIA",
  "example_parameter":"example_value"
}
```

### 4.4. 客户端凭证授权 (Client Credentials Grant)

客户端可以仅使用其客户端凭证（或其他支持的认证方式）请求 access token，当客户端请求访问其控制的受保护资源，或与授权服务器预先约定的其他资源所有者的受保护资源时（约定方式不在本规范范围内）。

客户端凭证授权类型仅限机密客户端使用。

```
  +---------+                                  +---------------+
  |         |                                  |               |
  |         |>--(A)--- 客户端认证 ------------>|  授权服务器   |
  |  客户端 |                                  |               |
  |         |<--(B)---- Access Token ---------<|               |
  |         |                                  |               |
  +---------+                                  +---------------+

                     图 6：客户端凭证流程
```

图 6 所示的流程包括以下步骤：

**(A)** 客户端向授权服务器进行认证并从 token 端点请求 access token。

**(B)** 授权服务器认证客户端，如果有效，签发 access token。

#### 4.4.1. 授权请求和响应

由于客户端认证被用作授权许可，不需要额外的授权请求。

#### 4.4.2. Access Token 请求

客户端通过在 HTTP 请求实体主体中使用"application/x-www-form-urlencoded"格式（按附录 B）和 UTF-8 字符编码添加以下参数，向 token 端点发起请求：

**grant_type**
      必需。值必须设置为"client_credentials"。

**scope**
      可选。如第 3.3 节所述的访问请求范围。

客户端必须按第 3.2.1 节向授权服务器进行认证。


例如，客户端使用传输层安全发起以下 HTTP 请求（为了显示目的添加了额外换行）：

```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
```

授权服务器必须认证客户端。

#### 4.4.3. Access Token 响应

如果 access token 请求有效且已授权，授权服务器如第 5.1 节所述签发 access token。不应包含 refresh token。如果请求的客户端认证失败或无效，授权服务器如第 5.2 节所述返回错误响应。

成功响应示例：

```http
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "access_token":"2YotnFZFEjr1zCsicMWpAA",
  "token_type":"example",
  "expires_in":3600,
  "example_parameter":"example_value"
}
```

### 4.5. 扩展授权

客户端通过将授权类型指定为绝对 URI（由授权服务器定义）作为 token 端点的"grant_type"参数的值，并添加必要的额外参数来使用扩展授权类型。


例如，要使用 [OAuth-SAML2] 定义的 SAML 2.0 断言授权类型请求 access token，客户端可以使用 TLS 发起以下 HTTP 请求（为了显示目的添加了额外换行）：

```http
POST /token HTTP/1.1
Host: server.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=urn%3Aietf%3Aparams%3Aoauth%3Agrant-type%3Asaml2-
bearer&assertion=PEFzc2VydGlvbiBJc3N1ZUluc3RhbnQ9IjIwMTEtMDU
[...为简洁省略...]aG5TdGF0ZW1lbnQ-PC9Bc3NlcnRpb24-
```

如果 access token 请求有效且已授权，授权服务器如第 5.1 节所述签发 access token 和可选的 refresh token。如果请求的客户端认证失败或无效，授权服务器如第 5.2 节所述返回错误响应。

## 5. 签发 Access Token

如果 access token 请求有效且已授权，授权服务器如第 5.1 节所述签发 access token 和可选的 refresh token。如果请求的客户端认证失败或无效，授权服务器如第 5.2 节所述返回错误响应。

### 5.1. 成功响应

授权服务器签发 access token 和可选的 refresh token，通过将以下参数添加到 HTTP 响应的实体主体中并使用 200 (OK) 状态码来构造响应：

**access_token**
      必需。授权服务器签发的 access token。

**token_type**
      必需。如第 7.1 节所述的已签发 token 类型。值不区分大小写。

**expires_in**
      推荐。access token 的生命周期（秒）。例如，值"3600"表示 access token 从响应生成时间起一小时后过期。如果省略，授权服务器应当通过其他方式提供过期时间或记录默认值。


**refresh_token**
      可选。refresh token，可用于使用相同的授权许可获取新的 access token，如第 6 节所述。

**scope**
      可选，如果与客户端请求的 scope 相同；否则为必需。如第 3.3 节所述的 access token 的范围。

参数使用 [RFC4627] 定义的"application/json"媒体类型包含在 HTTP 响应的实体主体中。参数通过在每个参数添加到最高结构层级来序列化为 JSON 结构。参数名称和字符串值作为 JSON 字符串包含。数值作为 JSON 数字包含。参数的顺序无关紧要，可以变化。

授权服务器必须在任何包含 token、凭证或其他敏感信息的响应中包含值为"no-store"的 HTTP "Cache-Control"响应头字段 [RFC2616]，以及值为"no-cache"的"Pragma"响应头字段 [RFC2616]。

例如：

```http
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "access_token":"2YotnFZFEjr1zCsicMWpAA",
  "token_type":"example",
  "expires_in":3600,
  "refresh_token":"tGzv3JOkF0XG5Qx2TlKWIA",
  "example_parameter":"example_value"
}
```

客户端必须忽略响应中无法识别的值名称。从授权服务器接收的 token 和其他值的大小未作定义。客户端应避免对值大小做出假设。授权服务器应当记录其签发的任何值的大小。


### 5.2. 错误响应

授权服务器以 HTTP 400 (Bad Request) 状态码响应（除非另有规定），并在响应中包含以下参数：

**error**
      必需。以下列表中的一个 ASCII [USASCII] 错误码：

      `invalid_request`
            请求缺少必需参数、包含不支持的参数值（授权类型除外）、重复参数、包含多个凭证、使用多种机制认证客户端，或格式错误。

      `invalid_client`
            客户端认证失败（如未知的客户端、未包含客户端认证或不支持的认证方法）。授权服务器可以返回 HTTP 401 (Unauthorized) 状态码来指示支持的 HTTP 认证方案。如果客户端通过"Authorization"请求头字段尝试认证，授权服务器必须以 HTTP 401 (Unauthorized) 状态码响应，并包含匹配客户端所用认证方案的"WWW-Authenticate"响应头字段。

      `invalid_grant`
            提供的授权许可（如授权码、资源所有者凭证）或 refresh token 无效、已过期、已被撤销、与授权请求中使用的重定向 URI 不匹配，或签发给了其他客户端。

      `unauthorized_client`
            已认证的客户端未被授权使用此授权许可类型。

      `unsupported_grant_type`
            授权服务器不支持此授权许可类型。


      `invalid_scope`
            请求的 scope 无效、未知、格式错误，或超出了资源所有者授予的范围。

      "error"参数的值不得包含集合 %x20-21 / %x23-5B / %x5D-7E 之外的字符。

**error_description**
      可选。提供额外信息的人类可读 ASCII [USASCII] 文本，用于帮助客户端开发者理解发生的错误。
      "error_description"参数的值不得包含集合 %x20-21 / %x23-5B / %x5D-7E 之外的字符。

**error_uri**
      可选。标识包含错误信息的人类可读网页的 URI，用于向客户端开发者提供关于错误的额外信息。
      "error_uri"参数的值必须符合 URI-reference 语法，因此不得包含集合 %x21 / %x23-5B / %x5D-7E 之外的字符。

参数使用 [RFC4627] 定义的"application/json"媒体类型包含在 HTTP 响应的实体主体中。参数通过在每个参数添加到最高结构层级来序列化为 JSON 结构。参数名称和字符串值作为 JSON 字符串包含。数值作为 JSON 数字包含。参数的顺序无关紧要，可以变化。

例如：

```http
HTTP/1.1 400 Bad Request
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "error":"invalid_request"
}
```


## 6. 刷新 Access Token

如果授权服务器向客户端签发了 refresh token，客户端通过在 HTTP 请求实体主体中使用"application/x-www-form-urlencoded"格式（按附录 B）和 UTF-8 字符编码添加以下参数，向 token 端点发起刷新请求：

**grant_type**
      必需。值必须设置为"refresh_token"。

**refresh_token**
      必需。签发给客户端的 refresh token。

**scope**
      可选。如第 3.3 节所述的访问请求范围。请求的 scope 不得包含资源所有者最初未授予的任何 scope，如果省略则视为与资源所有者最初授予的 scope 相同。

由于 refresh token 通常是长期有效的凭证，用于请求额外的 access token，refresh token 绑定到其被签发给的客户端。如果客户端类型是机密的或客户端被签发了客户端凭证（或被分配了其他认证要求），客户端必须按第 3.2.1 节向授权服务器进行认证。

例如，客户端使用传输层安全发起以下 HTTP 请求（为了显示目的添加了额外换行）：

```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
```


授权服务器必须：

- 对机密客户端或任何被签发了客户端凭证（或具有其他认证要求）的客户端要求客户端认证，
- 如果包含客户端认证，则认证客户端并确保 refresh token 是签发给已认证客户端的，
- 验证 refresh token。

如果有效且已授权，授权服务器如第 5.1 节所述签发 access token。如果请求验证失败或无效，授权服务器如第 5.2 节所述返回错误响应。

授权服务器可以签发新的 refresh token，在这种情况下客户端必须丢弃旧的 refresh token 并用新的 refresh token 替换。授权服务器可以在签发新的 refresh token 后撤销旧的 refresh token。如果签发了新的 refresh token，新的 refresh token 的 scope 必须与客户端在请求中包含的 refresh token 的 scope 相同。

## 7. 访问受保护资源

客户端通过向资源服务器出示 access token 来访问受保护资源。资源服务器必须验证 access token 并确保其未过期且其 scope 涵盖了请求的资源。资源服务器验证 access token 的方法（以及任何错误响应）不在本规范范围内，但通常涉及资源服务器与授权服务器之间的交互或协调。

客户端使用 access token 向资源服务器认证的方式取决于授权服务器签发的 access token 类型。通常涉及使用 HTTP "Authorization"请求头字段 [RFC2617] 和由所用 access token 类型的规范定义的认证方案，如 [RFC[[6750]]]。


### 7.1. Access Token 类型

access token 类型为客户端提供了成功使用 access token 发起受保护资源请求所需的信息（连同类型特定的属性）。如果客户端不理解 token 类型，则不得使用 access token。

例如，[RFC[[6750]]] 中定义的"bearer" token 类型通过简单地在请求中包含 access token 字符串来使用：

```http
GET /resource/1 HTTP/1.1
Host: example.com
Authorization: Bearer mF_9.B5f-4.1JqM
```

而 [OAuth-HTTP-MAC] 中定义的"mac" token 类型通过连同 access token 一起签发消息认证码 (MAC) 密钥来使用，该密钥用于对 HTTP 请求的某些组件进行签名：

```http
GET /resource/1 HTTP/1.1
Host: example.com
Authorization: MAC id="h480djs93hd8",
                   nonce="274312:dj83hs9s",
                   mac="kDZvddkndxvhGRXZhvuDjEWhGeE="
```

以上示例仅供说明之用。开发者在使用前应查阅 [RFC[[6750]]] 和 [OAuth-HTTP-MAC] 规范。

每个 access token 类型定义指定了与"access_token"响应参数一起发送给客户端的额外属性（如果有）。它还定义了在发起受保护资源请求时用于包含 access token 的 HTTP 认证方法。

### 7.2. 错误响应

如果资源访问请求失败，资源服务器应当通知客户端错误。虽然此类错误响应的细节不在本规范范围内，但本文档在第 11.4 节建立了一个公共注册表，用于在 OAuth token 认证方案之间共享错误值。

主要为 OAuth token 认证设计的新认证方案应当定义向客户端提供错误状态码的机制，其中允许的错误值在本规范建立的错误注册表中注册。


此类方案可以将有效的错误码集合限制为注册值的子集。如果错误码使用命名参数返回，参数名称应当为"error"。

其他能够用于 OAuth token 认证但不是主要为此目的设计的方案，可以以相同方式将其错误值绑定到注册表。

新认证方案可以选择也指定使用"error_description"和"error_uri"参数以与本规范中它们的使用方式并行的方式返回错误信息。

## 8. 可扩展性

### 8.1. 定义 Access Token 类型

access token 类型可以通过两种方式定义：在 Access Token Types 注册表中注册（遵循第 11.1 节的程序），或使用唯一的绝对 URI 作为其名称。

使用 URI 名称的类型应当限于不常用的供应商特定实现，且特定于使用它们的资源服务器的实现细节。

所有其他类型必须注册。类型名称必须符合 type-name ABNF。如果类型定义包含新的 HTTP 认证方案，类型名称应当与 HTTP 认证方案名称相同（如 [RFC2617] 所定义）。token 类型"example"保留用于示例中。

```
type-name  = 1*name-char
name-char  = "-" / "." / "_" / DIGIT / ALPHA
```

### 8.2. 定义新的端点参数

用于授权端点或 token 端点的新请求或响应参数在 OAuth Parameters 注册表中定义和注册，遵循第 11.2 节的程序。

参数名称必须符合 param-name ABNF，参数值语法必须明确定义（例如，使用 ABNF 或引用现有参数的语法）。

```
param-name  = 1*name-char
name-char   = "-" / "." / "_" / DIGIT / ALPHA
```


未注册的供应商特定参数扩展，如果不常用且特定于使用它们的授权服务器的实现细节，应当使用不太可能与其他注册值冲突的供应商特定前缀（例如，以"companyname_"开头）。

### 8.3. 定义新的授权许可类型

新的授权许可类型可以通过为"grant_type"参数分配唯一的绝对 URI 来定义。如果扩展授权类型需要额外的 token 端点参数，它们必须按第 11.2 节所述在 OAuth Parameters 注册表中注册。

### 8.4. 定义新的授权端点响应类型

用于授权端点的新响应类型在 Authorization Endpoint Response Types 注册表中定义和注册，遵循第 11.3 节的程序。响应类型名称必须符合 response-type ABNF。

```
response-type  = response-name *( SP response-name )
response-name  = 1*response-char
response-char  = "_" / DIGIT / ALPHA
```

如果响应类型包含一个或多个空格字符 (%x20)，则作为空格分隔的值列表进行比较，值的顺序无关紧要。只能注册一种值的顺序，该注册涵盖同一组值的所有其他排列。

例如，响应类型"token code"本规范未作定义。然而，扩展可以定义并注册"token code"响应类型。一旦注册，相同的组合不能再注册为"code token"，但两个值都可以用于表示相同的响应类型。

### 8.5. 定义额外的错误码

在协议扩展（即 access token 类型、扩展参数或扩展授权类型）需要额外的错误码用于授权码授权错误响应（第 4.1.2.1 节）、隐式授权错误响应（第 4.2.2.1 节）、token 错误响应（第 5.2 节）或资源访问错误响应（第 7.2 节）时，可以定义此类错误码。


如果扩展错误码所配合的扩展是已注册的 access token 类型、已注册的端点参数或扩展授权类型，则扩展错误码必须注册（遵循第 11.4 节的程序）。与未注册扩展一起使用的错误码可以选择注册。

错误码必须符合 error ABNF，并且应当尽可能以标识名称为前缀。例如，标识扩展参数"example"的无效值的错误应当命名为"example_invalid"。

```
error      = 1*error-char
error-char = %x20-21 / %x23-5B / %x5D-7E
```

## 9. 原生应用

原生应用是在资源所有者使用的设备上安装和执行的客户端（即桌面应用、原生移动应用）。原生应用在安全性、平台功能和整体终端用户体验方面需要特别考虑。

授权端点要求客户端与资源所有者的 user-agent 之间进行交互。原生应用可以调用外部 user-agent 或在应用内嵌入 user-agent。例如：

- **外部 user-agent** - 原生应用可以使用向操作系统注册了 scheme 的重定向 URI 来捕获授权服务器的响应，将客户端调用为处理器、手动复制粘贴凭证、运行本地 [[Web 服务器]]、安装 user-agent 扩展，或提供标识客户端控制的服务器托管资源的重定向 URI，该资源随后使响应对原生应用可用。

- **嵌入式 user-agent** - 原生应用通过直接与嵌入式 user-agent 通信来获取响应，通过监控资源加载期间发出的状态变化或访问 user-agent 的 cookie 存储来获取响应。

在选择外部还是嵌入式 user-agent 时，开发者应考虑以下因素：

- 外部 user-agent 可以提高完成率，因为资源所有者可能与授权服务器已有活跃的 session，无需重新认证。它提供了熟悉的终端用户体验和功能。在资源所有者还可以依赖 user-agent 的功能或扩展来辅助认证（如密码管理器、双因素设备读取器）。

- 嵌入式 user-agent 可以提供更好的可用性，因为它消除了切换上下文和打开新窗口的需要。

- 嵌入式 user-agent 带来安全挑战，因为资源所有者在一个无法识别的窗口中进行认证，无法获得大多数外部 user-agent 中的视觉保护。嵌入式 user-agent 教育终端用户信任无法识别的认证请求（使钓鱼攻击更容易执行）。

在选择隐式授权类型和授权码授权类型时，应考虑以下因素：

- 使用授权码授权类型的原生应用应当不使用客户端凭证，因为原生应用无法保持客户端凭证的机密性。

- 使用隐式授权类型流程时，不返回 refresh token，一旦 access token 过期就需要重复授权过程。

## 10. 安全考虑

作为一个灵活且可扩展的框架，OAuth 的安全考虑取决于许多因素。以下各节为实现者提供了专注于第 2.1 节描述的三种客户端配置文件的安全指南：Web 应用、基于 user-agent 的应用和原生应用。

全面的 OAuth 安全模型和分析以及协议设计的背景由 [OAuth-THREATMODEL] 提供。

### 10.1. 客户端认证

授权服务器与 Web 应用客户端建立客户端凭证用于客户端认证。授权服务器被鼓励考虑比客户端密码更强的客户端认证方式。Web 应用客户端必须确保客户端密码和其他客户端凭证的机密性。


授权服务器不得向原生应用或基于 user-agent 的应用客户端签发客户端密码或其他客户端凭证用于客户端认证。授权服务器可以向特定设备上的原生应用客户端的特定安装签发客户端密码或其他凭证。

当客户端认证不可能时，授权服务器应当采用其他方式验证客户端身份——例如，通过要求注册客户端重定向 URI 或请求资源所有者确认身份。在请求资源所有者授权时，有效的重定向 URI 不足以验证客户端身份，但可以在获得资源所有者授权后用于防止将凭证传递给伪造客户端。

授权服务器必须考虑与未经认证客户端交互的安全影响，并采取措施限制签发给此类客户端的其他凭证（如 refresh token）的潜在暴露。

### 10.2. 客户端冒充

如果被冒充的客户端未能或无法保持其客户端凭证的机密性，恶意客户端可以冒充另一个客户端并获取对受保护资源的访问。

授权服务器必须在可能时认证客户端。如果授权服务器由于客户端的性质无法认证客户端，授权服务器必须要求注册任何用于接收授权响应的重定向 URI，并应当采用其他方式保护资源所有者免受此类潜在恶意客户端的攻击。例如，授权服务器可以让资源所有者协助识别客户端及其来源。

授权服务器应当强制执行显式的资源所有者认证，并向资源所有者提供有关客户端和请求的授权 scope 及生命周期的信息。由资源所有者在当前客户端上下文中审查信息并授权或拒绝请求。

授权服务器不应当自动处理重复的授权请求（不进行主动的资源所有者交互），除非认证客户端或依赖其他措施确保重复请求来自原始客户端而非冒充者。


### 10.3. Access Token

access token 凭证（以及任何机密的 access token 属性）在传输和存储中必须保持机密，仅在授权服务器、access token 对其有效的资源服务器以及被签发 access token 的客户端之间共享。access token 凭证必须按照第 1.6 节使用具有 [RFC2818] 定义的服务器认证的 TLS 进行传输。

使用隐式授权类型时，access token 在 URI 片段中传输，可能会将其暴露给未授权方。

授权服务器必须确保 access token 不能被未授权方生成、修改或猜测以产生有效的 access token。

客户端应当请求具有最小必要 scope 的 access token。授权服务器在选择如何满足请求的 scope 时应当考虑客户端身份，可以签发权限少于请求的 access token。

> [!warning] 安全限制
> 本规范未提供任何方法供资源服务器确保给定客户端向其出示的 access token 是由授权服务器签发给该客户端的。

### 10.4. Refresh Token

授权服务器可以向 Web 应用客户端和原生应用客户端签发 refresh token。

refresh token 在传输和存储中必须保持机密，仅在授权服务器和被签发 refresh token 的客户端之间共享。授权服务器必须维护 refresh token 与其被签发给的客户端之间的绑定。refresh token 必须按照第 1.6 节使用具有 [RFC2818] 定义的服务器认证的 TLS 进行传输。

授权服务器在客户端身份可以认证时必须验证 refresh token 与客户端身份之间的绑定。当客户端认证不可能时，授权服务器应当部署其他方式检测 refresh token 滥用。

例如，授权服务器可以采用 refresh token 轮换 (Rotation)，在每次 access token 刷新响应中签发新的 refresh token。之前的 refresh token 被无效化但由授权服务器保留。如果 refresh token 被攻破并随后被攻击者和合法客户端同时使用，其中一个将出示已无效化的 refresh token，这将通知授权服务器发生了违规。

授权服务器必须确保 refresh token 不能被未授权方生成、修改或猜测以产生有效的 refresh token。

### 10.5. 授权码

授权码的传输应当通过安全通道进行，如果重定向 URI 标识了网络资源，客户端应当要求其重定向 URI 使用 TLS。由于授权码通过 user-agent 重定向传输，它们可能通过 user-agent 历史记录和 HTTP referrer 头字段被泄露。

授权码作为明文持有者 (Bearer) 凭证运行，用于验证在授权服务器授权的资源所有者与返回到客户端完成流程的资源所有者是同一人。因此，如果客户端依赖授权码进行自身的资源所有者认证，客户端重定向端点必须要求使用 TLS。

授权码必须是短生命周期且单次使用的。如果授权服务器观察到多次尝试交换授权码获取 access token，授权服务器应当尝试撤销基于已泄露授权码已授予的所有 access token。

如果客户端可以被认证，授权服务器必须认证客户端并确保授权码签发给了同一客户端。

### 10.6. 授权码重定向 URI 操纵

使用授权码授权类型请求授权时，客户端可以通过"redirect_uri"参数指定重定向 URI。如果攻击者可以操纵重定向 URI 的值，就可以使授权服务器将资源所有者的 user-agent 重定向到攻击者控制的 URI 并携带授权码。

攻击者可以在合法客户端创建账户并启动[[授权流程]]。当攻击者的 user-agent 被发送到授权服务器授予访问权限时，攻击者获取合法客户端提供的授权 URI 并替换客户端的重定向 URI 为攻击者控制的 URI。攻击者然后诱骗受害者跟随被操纵的链接以授权对合法客户端的访问。

一旦到达授权服务器，受害者收到的是一个代表合法可信客户端的正常有效请求，并授权该请求。受害者随后被重定向到攻击者控制的端点并携带授权码。攻击者通过使用客户端提供的原始重定向 URI 将授权码发送给客户端来完成[[授权流程]]。客户端将授权码交换为 access token 并将其关联到攻击者的客户端账户，该账户现在可以访问受害者授权的受保护资源（通过客户端）。

> [!danger] 防御措施
> 为了防止此类攻击，授权服务器必须确保用于获取授权码的重定向 URI 与交换授权码获取 access token 时提供的重定向 URI 完全相同。授权服务器必须要求公共客户端并应当要求机密客户端注册其重定向 URI。如果请求中提供了重定向 URI，授权服务器必须根据注册值进行验证。

### 10.7. 资源所有者密码凭证

资源所有者密码凭证授权类型通常用于遗留系统或迁移原因。它减少了客户端存储用户名和密码的总体风险，但并未消除将高权限凭证暴露给客户端的需要。

此授权类型比其他授权类型风险更高，因为它维持了本协议试图避免的密码反模式。客户端可能滥用密码，或者密码可能无意中泄露给攻击者（例如，通过日志文件或客户端保留的其他记录）。

此外，由于资源所有者无法控制授权过程（资源所有者的参与在将凭证交给客户端时就结束了），客户端可以获取比资源所有者期望的更广泛 scope 的 access token。授权服务器应当考虑通过此授权类型签发的 access token 的 scope 和生命周期。

授权服务器和客户端应当尽量减少使用此授权类型，并尽可能使用其他授权类型。


### 10.8. 请求机密性

access token、refresh token、资源所有者密码和客户端凭证不得以明文传输。授权码不应当以明文传输。

"state"和"scope"参数不应当以明文包含敏感的客户端或资源所有者信息，因为它们可能通过不安全通道传输或不安全地存储。

### 10.9. 确保端点真实性

为了防止中间人攻击 (Man-in-the-Middle)，授权服务器必须要求对发送到授权端点和 token 端点的任何请求使用具有 [RFC2818] 定义的服务器认证的 TLS。客户端必须按照 [RFC6125] 及其服务器身份认证要求验证授权服务器的 TLS 证书。

### 10.10. 凭证猜测攻击

授权服务器必须防止攻击者猜测 access token、授权码、refresh token、资源所有者密码和客户端凭证。

攻击者猜测生成的 token（以及其他非面向终端用户处理的凭证）的概率必须小于或等于 2^(-128)，应当小于或等于 2^(-160)。

授权服务器必须利用其他方式保护面向终端用户使用的凭证。

### 10.11. 钓鱼攻击

此协议及类似协议的广泛部署可能导致终端用户对被重定向到要求输入密码的网站这种做法变得麻木。如果终端用户在输入凭证前不仔细验证这些网站的真实性，攻击者就可能利用这种做法窃取资源所有者的密码。

服务提供商应当尝试教育终端用户钓鱼攻击的风险，并应当提供使终端用户能够轻松确认其网站真实性的机制。客户端开发者应当考虑其与 user-agent 交互方式（如外部、嵌入式）的安全影响，以及终端用户验证授权服务器真实性的能力。


为了降低钓鱼攻击的风险，授权服务器必须在每个用于终端用户交互的端点上要求使用 TLS。

### 10.12. 跨站请求伪造 (CSRF)

跨站请求伪造 (Cross-Site Request Forgery, CSRF) 是一种攻击，攻击者使受害终端用户的 user-agent 跟随恶意 URI（例如，作为误导链接、图片或重定向提供给 user-agent）到信任的服务器（通常通过有效 session cookie 的存在来建立）。

针对客户端重定向 URI 的 CSRF 攻击允许攻击者注入自己的授权码或 access token，这可能导致客户端使用与攻击者受保护资源而非受害者相关联的 access token（例如，将受害者的银行账户信息保存到攻击者控制的受保护资源）。

客户端必须为其重定向 URI 实现 CSRF 保护。这通常通过要求发送到重定向 URI 端点的任何请求包含将请求绑定到 user-agent 已认证状态的值（例如，用于认证 user-agent 的 session cookie 的哈希值）来实现。客户端应当使用"state"请求参数在发起授权请求时将此值传递给授权服务器。

一旦从终端用户获得授权，授权服务器将终端用户的 user-agent 重定向回客户端，并在"state"参数中包含所需的绑定值。绑定值使客户端能够通过将绑定值与 user-agent 的已认证状态进行匹配来验证请求的有效性。用于 CSRF 保护的绑定值必须包含不可猜测的值（如第 10.10 节所述），且 user-agent 的已认证状态（如 session cookie、HTML5 本地存储）必须保存在仅客户端和 user-agent 可访问的位置（即受同源策略保护）。

针对授权服务器授权端点的 CSRF 攻击可能导致攻击者在未涉及或通知终端用户的情况下获取终端用户对恶意客户端的授权。

授权服务器必须为其授权端点实现 CSRF 保护，并确保恶意客户端无法在资源所有者不知情和未明确同意的情况下获取授权。


### 10.13. 点击劫持 (Clickjacking)

在点击劫持攻击中，攻击者注册一个合法的客户端，然后构建一个恶意网站，在该网站中，它在透明的 iframe 中加载授权服务器的授权端点网页，覆盖在一组虚拟按钮之上，这些按钮被精心构造以直接对齐授权页面上的重要按钮下方。当终端用户点击误导性的可见按钮时，实际上点击的是授权页面上的不可见按钮（如"授权"按钮）。这允许攻击者诱骗资源所有者在终端用户不知情的情况下授予其客户端访问权限。

为了防止此类攻击，原生应用在请求终端用户授权时应当使用外部浏览器而非在应用内嵌入浏览器。对于大多数较新的浏览器，可以通过授权服务器使用（非标准的）"x-frame-options"头字段来避免 iframe。该头字段可以有两个值："deny"和"sameorigin"，分别会阻止任何框架嵌入或仅阻止不同源的框架嵌入。对于较旧的浏览器，可以使用 JavaScript frame-busting 技术，但可能并非在所有浏览器中都有效。

### 10.14. 代码注入和输入验证

代码注入攻击发生在应用未经过滤地使用输入或其他外部变量并导致应用逻辑被修改时。这可能允许攻击者获取对应用设备或其数据的访问、导致拒绝服务或引入各种恶意副作用。

授权服务器和客户端必须对（在可能时验证）接收到的任何值进行过滤——特别是"state"和"redirect_uri"参数的值。

### 10.15. 开放重定向器

授权服务器、授权端点和客户端重定向端点可能被配置不当并作为开放重定向器运行。开放重定向器是使用参数自动将 user-agent 重定向到参数值指定的位置而不进行任何验证的端点。

开放重定向器可用于钓鱼攻击，或被攻击者用于利用熟悉且信任的目的地的 URI authority 组件使终端用户访问恶意网站。此外，如果授权服务器允许客户端仅注册重定向 URI 的部分，攻击者可以利用客户端运营的开放重定向器构造一个能通过授权服务器验证但会将授权码或 access token 发送到攻击者控制端点的重定向 URI。

### 10.16. 隐式流程中滥用 Access Token 冒充资源所有者

对于使用隐式流程的公共客户端，本规范未提供任何方法供客户端确定 access token 是签发给哪个客户端的。

资源所有者可能自愿委托对资源的访问，通过向攻击者的恶意客户端授予 access token。这可能是由于钓鱼或其他借口。攻击者还可能通过其他机制窃取 token。攻击者然后可以尝试通过向合法的公共客户端提供 access token 来冒充资源所有者。

在隐式流程（response_type=token）中，攻击者可以轻松地替换来自授权服务器的响应中的 token，将真实的 access token 替换为之前签发给攻击者的 access token。

与依赖在后端通道传递 access token 来识别客户端用户的原生应用通信的服务器，同样可能被攻击者创建的能够注入任意被盗 access token 的被攻破应用所侵害。

任何假设只有资源所有者才能向其出示有效 access token 的公共客户端都容易受到此类攻击。

此类攻击可能将合法客户端处的资源所有者信息暴露给攻击者（恶意客户端）。这也将允许攻击者以最初授予 access token 或授权码的资源所有者的相同权限在合法客户端上执行操作。

对客户端进行资源所有者认证不在本规范范围内。任何将授权过程用作委托式终端[[用户认证]]的规范（如第三方登录服务）不得在没有额外安全机制使客户端能够确定 access token 是否为其使用而签发的情况下使用隐式流程（如对 access token 进行受众限制）。

### 11.1. OAuth Access Token Types 注册表

## 附录 B. application/x-www-form-urlencoded 媒体类型的使用

在本规范发布时，"application/x-www-form-urlencoded"媒体类型在 [W3C.REC-html401-19991224] 第 17.13.4 节中定义，但未在 IANA MIME 媒体类型注册表 (<http://www.iana.org/assignments/media-types>) 中注册。此外，该定义不完整，因为它未考虑非 US-ASCII 字符。

为了在使用此媒体类型生成有效载荷时解决这一不足，名称和值必须首先使用 UTF-8 字符编码方案 [RFC3629] 进行编码；然后得到的八位字节序列需要进一步使用 [W3C.REC-html401-19991224] 中定义的转义规则进行编码。

在解析使用此媒体类型的有效载荷中的数据时，从反转名称/值编码得到的名称和值需要被视为八位字节序列，使用 UTF-8 字符编码方案进行解码。

例如，由六个 Unicode 码点 (1) U+0020 (SPACE)、(2) U+0025 (PERCENT SIGN)、(3) U+0026 (AMPERSAND)、(4) U+002B (PLUS SIGN)、(5) U+00A3 (POUND SIGN) 和 (6) U+20AC (EURO SIGN) 组成的值将被编码为以下八位字节序列（使用十六进制表示法）：

```
20 25 26 2B C2 A3 E2 82 AC
```

然后在有效载荷中表示为：

```
+%25%26%2B%C2%A3%E2%82%AC
```
