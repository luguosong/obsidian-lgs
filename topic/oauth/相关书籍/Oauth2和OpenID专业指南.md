## 前言

我是在 2019 年加入 Auth0 时认识 Vittorio 的。那时我只知道他是 Identity 领域的权威专家，而我对这个话题知之甚少。通常他会负责技术团队的入职培训，但当我去 Bellevue 参加入职培训时，分配了另一位培训师。不过他还是来办公室跟我打了招呼，因为他听说新员工中有一位意大利人。这清楚地表明他不只是一位 Identity 专家——他是一个充满同理心的人。

在 Auth0 职业生涯初期，我有幸与 Vittorio 密切合作，部分是协助他编写这本书，部分是审阅我和团队创建的其他开发者内容。这让我有机会学到很多东西，不仅是 Identity 的技术层面，还有如何针对特定受众解释复杂概念。他的风格充满隐喻和轶事（你会在本书中找到不少），一直令我着迷。他那戏剧性的表达方式、明显的口音和典型的意大利手势，在我心中树立了一个伟大故事讲述者的形象。

我们在 2019 年底启动了这个书籍项目，并以增量方式出版了两版。Vittorio 早已制定了本书的路线图，并与我分享了图表和笔记，但他繁重的事务使他一再推迟完成。

他在 2023 年 10 月的意外离世令我震惊，整个 Identity 专业社区同样如此。我多次收到开发者关于本书完成情况的询问。因此，不辜负读者的期望，同时纪念 Vittorio，按照原始路线图和他的笔记完成这本书，似乎是我应做的事。我尽可能保留他的风格，尽管那是独特且不可复制的。自这个项目启动以来，一些情况发生了变化，因此我不得不更新一些引用，甚至包括已出版的部分。在这项改编、更新和修订的工作中，Aaron Parecki 和 Filip Skokan 的帮助至关重要，我衷心感谢他们。

在让你开始阅读之前，最后一点说明。这不是一本关于 OAuth 2.0 和 OpenID Connect 的技术手册。本书不会为你提供在应用中使用 OAuth 2.0 和 OpenID Connect 的详细指南。或者说，你也会学到这些，但更重要的是，本指南将解释这些协议中那些细节背后的原因——为什么我们需要它们，以及它们是如何产生的。

理解技术选择背后的原因，以及由此而来的历史背景，将帮助你更好地理解 OAuth 2.0 和 OpenID Connect，并获得对它们的专业认知。言归正传，希望你享受阅读本书！

Andrea Chiarelli

## 引言

本书将帮助你理清 OAuth、OpenID Connect 以及实现认证和委托授权所需的众多组件。

你将了解认证和授权需求在过去几年中的演变，以及当今的标准协议如何在其前身的基础上发展和增强，以应对那些挑战——问题与解决方案在不断升级的竞赛中相互博弈。

你将同时学习 OAuth 2.0 和 OpenID Connect 的原理和实践。你将了解协议的哪些部分适用于各种经典场景和应用类型（传统 Web 应用的单点登录 (Single Sign-On)、单页应用、从桌面端、移动端和 Web 应用调用 API 等）。我们将详细审视每一个交互和参数——将一切置于上下文中，始终致力于理解每个实现选择在宏观图景背后的原因。

阅读本书后，你将对认证和委托授权中的经典问题有清晰的认识，了解开放协议提供的现代工具如何解决这些问题，并获得 OAuth 2.0 和 OpenID Connect 的实操知识。所有这些将使你能够做出明智的设计决策——甚至能够在故障排查和网络抓包分析中游刃有余。 

## 第 1 章 数字身份入门

在本章中，你将掌握 identity 的一些核心要点，包括相关概念和我们在这一领域常用的术语。同时你将深入了解 Identity 领域中那些经典难题——那些我们要征服的"恶龙"——这恰好也是 Auth0 by Okta 为客户所能解决的问题。

闲话少说，identity 到底是怎么回事？为什么人们总说"哦，这很复杂"？为什么？先看看下面这张图。它简单得不能再简单了：这里只有两个主体，就像你基础物理课上遇到的问题。这应该是那种简单的问题：

我拥有某种资源，还有一个用户——某种实体想要以某种方式访问该资源。就两个主体做一个动作，为什么这么复杂？

首先，因为这是关键任务。

当这种场景出问题时，会灾难性地出错。因此，和所有关键任务场景一样，它值得我们尊重、关注和准备。大量的精力投入在防止灾难场景成为现实。但在开发领域的这个特定方向上，让事情变得复杂的是所有决定你"需要做什么才能得到可行方案"的因素的笛卡尔积。考虑以下因素： 

**资源类型**

想一想你可能拥有的所有资源类型。就在几年前，如果你走进一家银行，你会看到一台主机，它有一个中央数据库，仅此而已。而今天，几乎所有东西都可以通过编程方式访问。于是有了 API 经济，有了 serverless——这些热词实际上指向的是暴露资源的不同方式，当然还有网站、应用和你日常生活中使用的各种东西。每当你与计算机系统交互时，都有某种你需要连接的资源。从开发者的角度来看，实现这种连接实际上是大量的工作。 

**开发技术栈**

不同的开发技术栈之间存在微小差异，但这些差异会在实现资源访问和交互方式的代码上产生巨大的不同。这是一层复杂性。 

**身份来源**

另一层复杂性在于当今可用的身份来源数量之庞大。

想一想你自己的身份在网络上被表达的所有方式。你可以是一个社交网络的成员、一家公司的员工、一个国家的公民。所有这些身份都以某种方式存在于某个数据库中，而那个"某处"决定了你如何提取这些信息。

你以一种方式连接 Facebook，以另一种方式连接 Active Directory，在向国家纳税时又以另一种方式被识别。所以，我们再次遇到了另一个复杂因素：如果你想从这些仓库中提取身份信息，就必须按照每个仓库的要求和特性找到相应的方式。 

**客户端类型**

最后，还有更多复杂因素，但我只想再提一个：当今我们消费信息的途径之丰富令人难以置信。想一想你可以使用的所有可能的客户端：从手机和应用，到网站、手表。你几乎可以用任何东西来访问数据。再一次，这些因素与你想访问的资源类型、你从中提取信息的地方叠加在一起，使复杂性倍增。所以，这张图看起来可能很简单，但实际上绝非如此。

那么，Okta 能做些什么让这一切更易于管理呢？我们提供了很多不同的东西，但最核心的组件是 Auth0。它是一个服务，你可以用它来外包解决方案中所需的大部分认证功能——这样你就不必面对那些复杂性。具体来说，我们提供：

• 抽象掉连接多个身份来源细节的方式。每个身份提供者（Identity Provider）进行身份事务的方式各不相同，我们将所有这些复杂性从你面前抽象掉了。

• 处理用户管理生命周期的方式。我们有用户表示和用于处理用户生命周期等功能。

• 大量的 SDK 和示例，帮助你走完最后一公里——当你使用特定的开发技术栈时，你可以使用这些组件以与该技术栈习惯一致的方式连接到 Auth0。

• 业内绝对前所未有的定制化能力。目前没有其他服务能提供与 Auth0 相同的定制自由度。

现在，当你需要将应用连接到 Auth0 时，你需要做些什么来告诉我们"Auth0，请执行认证"。而 Auth0 中这个"什么"是使用开放标准实现的。

开放标准是由行业内不同参与者组成的联盟制定的广泛共识性协议。我们这些 Identity 专业人士之所以决定制定开放标准，是因为我们意识到，如果把认证等场景中必需的一些事务用通用标准、通用消息、通用协议固化下来，所有人——用户、客户和厂商——都会从中受益。当时发生的事情是：我们去世界各地那些不算便宜的酒店，与业界同行会面，争论在身份事务上下文中应用应该如何展示自己的服务。我们对身份提供者也有类似的讨论。应该交换什么样的消息？我们甚至为了消息中分号的细节争得面红耳赤。制定标准就是这么有趣，但这一切都是值得的：现在我们有了开放标准，所有厂商都实现了这些开放标准，作为客户，你可以自由选择想用的厂商，无需担心被特定技术或厂商锁定。最重要的是，你可以规划之后引入不同的技术，而不必担心不兼容的问题。

当然，这主要是理论上的——有点像那些忽略摩擦力或月球引力影响潮汐的简化物理题。现实中总有一些细节需要处理。但大体来说，如果你在过去几十年里在我们这个行业工作过，你就会知道有了这些可以依赖的开放标准，我们的处境好了太多。

在身份管理中，你将接触到许多协议，其中许多可能还未被发明。如今日常使用的主要有：

OpenID Connect，用于登录。

OAuth 2.0，是 OpenID Connect 的基础，它是一个委托协议，旨在帮助你访问第三方 API。

JSON Web Token（JWT），一种标准的 token 格式。你将要使用的大部分 token 都采用这种格式。

SAML，某种程度上算是遗留协议（但仍然非常活跃），用于跨域浏览器的单点登录 (Single Sign-On)。SAML 也定义了一种标准的 token 格式，过去非常流行，至今仍被广泛使用。 

### 从每个应用都有用户密码说起......

让我们花几分钟快速回顾认证技术的演变历程。我希望通过回到基础、重温这个简化的时间线，向你展示为什么今天的事情是现在这个样子。在此过程中，我也会在恰当的时机引入正确的术语。在正确的时机接触新术语——也就是说，当某个术语首次出现时——你会以最一般化的方式理解对应概念的含义。与此形成对比的是，如果你只在解决特定问题的上下文中接触某个术语，你对它的理解会更狭窄。你可能会认为你正在解决的问题就是这个概念唯一的用途，从而忽略了全局，并在将来陷入各种误解。我们不会让这种事发生！

让我们回到最基础的地方，想一想前面图 1.1 中描述的场景——我有某种资源（比如一个 Web 应用），还有一个用户，我们想把两者连接起来。那么，在这个上下文中，identity 是什么？

我们不会纠结于哲学之类的东西。这里的 identity 可以用一种非常操作化、非常精确的方式来定义。我们将数字身份（digital identity）定义为：在特定应用所交付功能的上下文中，定义特定用户的一组属性。

这是什么意思？这意味着如果我是一个书商，我需要了解用户的有关信息主要是他们的信用卡号、配送地址和最近购买的十本书。这就是该上下文中他们的数字身份。如果我是税务部门，那么用户的数字身份同样是一个物理地址、一个标识符（在美国就是社会安全号码）以及与从公民那里收税相关的任何其他信息。如果我是一个做 DNA 测序的服务，我用户的身份就是他们用于登录的用户名、用于通知的电子邮件地址，可能还有他们的整个基因组。

你可以看到，对于我们想要实现的各种功能，我们实际上拥有完全不同或几乎完全不同的属性集。这些属性可能对应同一个自然人，也可能不对应。这无关紧要。从系统设计的角度来看，数字身份就是这样。所以你可以说，这个用户的数字身份就是我们可以放入应用存储中的这组属性。那么，身份问题就变成了：我何时将这些特定属性引入上下文？世界上最古老的技巧是让资源和用户就某件事达成一致，比如某种共享密钥。于是，当用户再次访问网站并出示该密钥或证明自己知道该密钥时，网站就会说：”好的，我知道你是谁了。你就是我昨天见过的那个用户。”这是你的属性集，欢迎回来。我认证了这个用户。总结来说，就是获取一组凭据（credentials），发送过去，并假设这些凭据之前已保存在数据库中。如果匹配，用户就通过了认证。这个场景总结在下面的图中：

关于用户名和密码，你听到了很多负面评价......它们都是真的。这很不幸，但确是事实。然而，这是一个极其简单的方案，正因如此，它非常、非常、非常有韧性。

即使我们拥有更先进的技术来完成大致相同的工作，密码仍然非常流行。我预测今年——就像每年一样——会有人说这是密码消亡之年。但我认为密码还会存在一段时间。我最喜欢的比喻是自然界中发生的事情。人类据说是进化的巅峰。然而，海里仍然有大量的水母。它们如此简单，没错，我们更先进，但我敢打赌水母的个体数量比人类多。它们身体结构的简单并不意味着它不成功。你会看到，随着我们回顾这段历史，密码在某种程度上是更高级协议赖以构建的基础模块。再次说明，我并非否定消除密码和使用更好方案的努力，我只是想设定一个合理的预期：这还需要一些时间。 

### ...到目录服务

让事情变得更有趣一些。想象一下我们有一个用户和一个应用的场景。现在，将这个场景扩展到该用户是某家公司员工的情况。有一系列应用被该用户在公司业务中使用。大多数应用都是用户在受雇期间工作的一部分。想象一个应用是报销系统，另一个是财务系统，还有一个是仓储管理系统。任何你能想到的都可以。几年前的情况是，我们在一台计算机上有一堆应用。然后，有人拿着同轴电缆出现了，安装了令牌环网络，把所有这些计算机连入了网络。但这并没有让环境——特别是应用——自动具备网络就绪能力。发生的事情是你会遇到这样的局面——关键问题在于——一个用户访问不同的、互不相知的独立应用，这些应用重复实现了本可以轻松集中的功能。具体来说，每个用户有不同的用户名和密码——或者我应该说是不同的用户名，因为人们当然会重复使用密码。每次用户进入一个新应用，都必须输入凭据。

每当用户不得不离开公司——无论是否自愿——管理员都必须在所有这些不同的应用中进行一场"朝圣之旅"，逐一追踪用户的条目，手动取消配置（deprovision），这当然是一个既繁琐又容易出错的流程。这很困难。你经常听到恐怖故事：心怀不满的员工利用采购系统大量购买物品来报复前老板，而他们之所以能做到，是因为他们在采购系统中的凭据没有被及时撤销。

至少可以说，这不是一个好的局面。

业界的回应是引入了一种新的实体，我们称之为目录（directory）。目录至今仍然极其流行。它是一个软件组件、一个服务，集中了你在图 1.3 中看到的许多功能。 

![image](https://cdn-mineru.openxlab.org.cn/result/2026-05-15/58666355-8904-417a-b7b5-474c257ad5bc/a4a42078ad465eb3f6e52ed5e0198eb4b5816b745bfd33c46486a5a2ba113634.jpg)


基本上，目录集中了凭据和属性，使应用不再需要实现自己的身份管理逻辑。此时，用户只需用中央目录登录一次，从那以后就可以单点登录 (Single Sign-On) 访问所有其他应用。应用开发者实际上不需要

为身份功能编写任何代码就能实现这一效果。事实上，既然网络基础设施本身提供了身份信息，管理员就可以利用这个集中化的地方来处理用户生命周期。可以说，目录的引入真正创造了身份管理员这一专业角色类别。目录的普遍可用性创造了一个工具生态系统，帮助人们运维身份等事务。所以，这是一次巨大的改进——但它建立在边界（perimeter）的前提之上。为了让一切按预期工作，你必须让所有参与者都在那个边界之内。这个边界通常就是办公楼本身，用户实际走进大楼，坐在特定的物理设备前，与企业中心这座”大教堂”有直接的”视线”——目录，一个了解所有人所有事的中央枢纽。 

### 跨域 SSO

当然，从当今的商业实践我们知道，这种方式无法扩展。它在一家公司内部运作良好，但有太多业务流程需要多家公司参与。

想想经典的供应商或经销商关系。这些关系中的每一种都需要跨越多个组织。于是，当一个组织中的用户需要访问另一个组织中不同的资源时，问题就出现了。事实上，这个用户在资源方的目录中并不存在。

业界最初试图解决这个问题的方法是引入我们所说的影子账户（shadow account），即将用户配置（provision）到资源方的目录中。这是完全不可持续的，因为它呈现出我们之前提到的每个应用自行处理身份时同样的问题，只是规模不同。假设我们有一个用户，其生命周期在一个地方管理——他们自己的主目录——但同时也在资源方目录中被配置为条目。当用户从主目录被取消配置时，可能还有一系列在其他目录（如我们场景中的资源方目录）中配置的用户账户仍然存在，需要手动取消配置。这当然是一个大问题，因为取消配置不太可能及时发生，或者像所有变更一样，在非集中管理的分布式系统中更难反映。再加上，想象一下这家公司作为许多其他公司的经销商，却需要在一定程度上重复其客户公司已经在各自目录中为管理自己用户所做的工作。这简直不可持续。

于是，就像计算机科学中的经典做法一样，我们通过添加一层抽象来解决这个问题。我们提取了在本地目录场景中看到的能力，并将其抽象化。我们提供了相同的事务，但以一种不依赖网络基础设施的方式来描述。例如，Active Directory 以及目录服务通常依赖一种叫做 Kerberos 的认证协议，它与网络层高度集成，因此有特定的网络硬件要求。而在跨公司场景中，我们必须跨越公共互联网的鸿沟，无法施加任何要求，因为请求将穿越未知的网络硬件。

当时的巨头——Sun、IBM 等公司坐在一起，制定了一个叫做 SAML 的协议，即安全断言标记语言（Security Assertion Markup Language）。简而言之，该协议描述了一种事务：用户可以在一个地方登录，然后在另一个地方出示登录证明并获得访问权限。下面是它的工作方式。我们需要一种东西来为实际资源加上门面——一些能够用该协议通信的软件，在这个特定情况下就是我们所说的中间件（middleware）：一个位于你的应用和调用者之间的组件，拦截流量并在请求到达实际应用之前执行逻辑。身份提供者一侧也会暴露类似的协议能力。在图 1.3 所示的拓扑中，我们有一台已经承担本地目录职责的机器（目录术语中称为域控制器）。我们只需教这台机器说另一种语言——SAML，它可以被视为一种用于公司边界之外通信的"通用语言"。

为了完成这个事务，我们需要引入另一个概念：信任（trust）。回想一下我们之前描述的场景——在单个目录内的场景：其中每个应用和每个用户都隐式地相信和信任域控制器。网络软件本身在你需要认证时会将你发送回域控制器，域控制器执行认证。这一切是隐式的，就像你呼吸的空气一样自然，因为整个网络中只有一个地方可以执行认证职责。

现在，看看这个特定场景：

公司 2 边界内的应用可以被其任何业务伙伴访问：现在有了关于从哪里获取用户身份的选择，不再有显而易见的默认用户来源。我们说一个资源信任一个身份提供者（Identity Provider）或权威机构，是指该资源愿意相信该权威机构关于其用户所说的内容。如果权威机构说”这个用户是我的用户，并且五分钟前成功通过了认证”，资源就会相信它。这就是信任的全部含义。

当你在应用前面设置中间件时，通常会配置你信任的身份提供者的坐标。这在实际执行事务时如何发挥作用？让我们通过描述下图中每个编号步骤的细节来看看这个流程是如何运作的： 

![image](https://cdn-mineru.openxlab.org.cn/result/2026-05-15/58666355-8904-417a-b7b5-474c257ad5bc/270a89fec0692c585bd57c332e3fa9a66d81fb014834088e958579b3eb2bee1a.jpg)

在图的第一段，用户将浏览器指向应用并尝试 GET 一个页面（1）。应用前面的中间件拦截了请求，发现用户尚未认证，于是将请求转换为向身份提供者（IdP）发起的认证请求，因为该 IdP 已被配置为受信任的 IdP 之一（2）。

具体来说，中间件会构造某种消息——可能是一个带有特定查询字符串参数的 URL——并将浏览器重定向到与身份提供者关联的某个特定 endpoint（3）。

在这个特定场景中，目标 endpoint 属于一个本地身份提供者。你可以看到，对 IdP 认证 endpoint 的调用发生在企业边界之内。这意味着该调用将像本地网络上的任何其他调用一样使用 Kerberos 进行认证。你已经可以看到这些协议的分层，一个叠在另一个上面。得益于 Kerberos 的使用以及用户已经在本地目录中通过认证这一事实，用户在此调用期间不需要输入任何凭据。

接下来，身份提供者确认用户已经正确通过认证，并确定该资源是已记录且已批准的资源之一。由于这些检查都通过了，IdP 向用户颁发我们称之为安全令牌（security token）的东西（4）。安全令牌是一个制品（artifact），一堆比特位，用于携带用户已成功认证的有形证明。安全令牌是经过数字签名的。这是什么意思？数字签名是保护比特位不被篡改的机制。假设有人在传输过程中修改了其中的任何比特位：当预期的接收者尝试检查签名时，会发现签名无法通过验证。接收者就能确定这些比特位在传输过程中被修改了。

这个特性之所以有用，有两个原因。一个原因是，既然我们使用公钥密码学，我们预期用于执行签名的私钥只能由该令牌的预期来源访问。宇宙中没有其他人能产生那个签名，只有那一方。回想一下我们刚才说的信任：这个特性可以用作令牌来自特定实体的证明，特别是来自受信任实体的证明。第二个原因是，既然令牌内容在不破坏签名的情况下无法在传输中被修改，我就可以将令牌作为一种即时提供用户数字身份的机制。作为应用，我不需要提前协商获取定义用户的属性（根据我们的定义，即用户身份），而是可以在收到令牌的同时及时接收这些属性。这可能是该特定用户第一次也是最后一次访问这个应用，但由于两个组织之间存在信任关系，我不需要做任何预先配置的步骤。

令牌中携带的属性称为声明（claim）。声明仅仅是在特定上下文中打包的属性，该上下文允许接收者决定是否相信用户确实拥有该属性。想想登机时发生的事情。如果我向登机口工作人员出示护照，他们能够将我的名字（由护照所声明）与登机牌上印刷的名字进行比对，然后决定是否让我通过。登机口工作人员之所以能得出这个结论，是因为他们信任政府——签发我护照的实体。如果我掏出一张上面用我歪歪扭扭的字迹写着名字的便利贴，用它代替护照出示给登机口工作人员，我大概是无法登机的——事实上，我很可能会惹上麻烦。在这种情况下，媒介本身就是信息。令牌确实承载了决定你是否信任特定信息的能力。令牌中的属性变成了声明。这是一个重要的区别。

一旦身份提供者颁发了 SAML 令牌，它通常将其放在 HTML 表单中返回给浏览器，同时附带一些在页面加载后立即触发的 JavaScript——将令牌 POST 到应用，在那里它会被中间件拦截（5）。中间件查看令牌，确认它是否来自受信任的来源，确认签名是否未被破坏等等，如果一切都没问题，它就发出我们所说的会话 cookie（session cookie）（6）。会话 cookie 代表成功完成了认证。通过设置 cookie 来表示会话，应用就不必在每次后续请求时再次执行令牌交互。会话 cookie 的作用仅仅是让应用在每次收到回传请求时将用户视为已认证。

这就是 SAML 解决跨域单点登录这个特定问题的方式。我们会看到，这种用令牌换取 cookie 的模式在 OpenID Connect 中同样会出现。 

### 密码共享反模式

所有这些都发生在商业世界中，但消费世界从身份的角度来看也没有停滞不前。发生的一件事是，随着我们越来越多的生活搬到线上，我们越来越频繁地需要从一个应用访问我们在另一个应用中管理的资源。

让我举一个非常具体的例子。我猜你们很多人都有 LinkedIn 账户，很多人也有 Gmail。想象以下场景：一个用户已经以某种方式登录了 LinkedIn。在这个场景中，他们是如何登录 LinkedIn 的并不重要。假设 LinkedIn 想建议你邀请所有 Gmail 联系人成为你的 LinkedIn 人脉。

我们使用 LinkedIn 和 Gmail 只是因为它们是大家熟悉的名字和熟悉的用例，但我们绝不暗示它们真的以这种方式实现，也不暗示它们在编写本书中扮演了任何直接角色。

那么，LinkedIn 过去是怎么做的呢？我在这里以 LinkedIn 为例，但在委托授权兴起之前，基本上任何类似服务都是这种行为。让我们按照下图中的步骤来看看这个流程： 

![image](https://cdn-mineru.openxlab.org.cn/result/2026-05-15/58666355-8904-417a-b7b5-474c257ad5bc/b54538147bf92c6f8f1ec4e31c910d9689f25b3549d23b5ecafda39004a79e09.jpg)

LinkedIn 实际上会要求你提供 Gmail 用户名和密码，这些凭据通常由 Gmail 存储和验证（1）。你将 Gmail 凭据提供给 LinkedIn（2），然后 LinkedIn 使用它们来访问 Gmail 应用自身用于程序化访问其服务的 Gmail API（3）。这就实现了 LinkedIn 想要的功能：调用 Gmail API 来列出你的联系人（4）并代表你发送电子邮件。

这个场景有什么问题？很多问题，但有两个特别不可忽视。

第一个问题是，向任何非凭据保管方的实体提供你的凭据访问权始终是一个糟糕的主意。这主要是因为这些不同的实体不会像凭据原始保管方那样有足够的利益约束。如果 LinkedIn 没有尽到审慎义务，将那些凭据保存在不安全的地方......当然，他们会有负面公关，但这不会像对 Gmail 那样成为灾难——因为 Gmail 的用户访问会受到直接影响。例如，Gmail 用户需要更改密码，这会导致他们极有可能流失，或至少对服务产生较低的满意度。

第二个糟糕的事情是：虽然 LinkedIn 在这个事务中的意图是好的（对作为用户的我来说，扩展人脉是互利的，对作为服务的 LinkedIn 也一样），但他们实现这个功能的方式赋予了他们过大的权力。LinkedIn 实际上可以使用这个用户名和密码对我的 Gmail 做任何事。他们可以阅读我的邮件，可以有选择地删除邮件，可以发送其他邮件，可以做超出原始意图范围的任何事情——这显然是不好的。 

### 委托授权：OAuth 2.0

为了应对上一节末尾描述的挑战，业界想出了一种方法来解决赋予应用过多权力的问题。

OAuth 2.0 的设计目的正是实现前面描述的委托访问场景，但不带有暴力破解方法中的那些不良特性。OAuth 2.0 方法的决定性特征在于引入了一个新的实体——授权服务器（authorization server），它明确处理与委托授权相关的操作。我现在不会讲太多细节，因为后面我会让你在这方面听得不厌其烦。

这里只需说明授权服务器有两个 endpoint：

• 授权端点（authorization endpoint），用于处理与最终用户的交互。它的设计目的是让用户表达是否希望某个服务以某种方式访问他们的资源。授权端点处理委托授权事务中的交互部分。

• 令牌端点（token endpoint），用于处理软件与软件之间的通信，负责实际执行用户在权限、同意、委托等方面表达的意图。稍后会有更多细节。

请注意，在以下讨论中，我们假设用户在所描述的场景发生之前已经登录了 LinkedIn。我们不关心登录是如何发生的，只是假设它已经发生了。OAuth 2.0，正如你将反复听到的，不是一个登录协议。

假设在 LinkedIn 会话中，用户到达了一个点，LinkedIn 想要代表他们访问 Gmail API，就像上一节中类似场景所描述的那样。

在 OAuth 2.0 的方法中，这意味着 LinkedIn 会引导用户前往 Gmail，并向 LinkedIn 授予查看联系人和代表他们发送邮件的权限。让我们通过下图来跟踪这个新流程： 

![image](https://cdn-mineru.openxlab.org.cn/result/2026-05-15/58666355-8904-417a-b7b5-474c257ad5bc/816bae49170b29d87f19d8f3c0252b139fa940811799ca745a63066d2e257195.jpg)



图 1.7


LinkedIn 按照 OAuth 2.0 规范构造授权请求，并将用户的浏览器重定向到 Gmail 的授权服务器，具体来说是授权端点（1）。

Gmail 使用授权端点来提示用户（2）输入凭据（如果他们当前未通过 Gmail Web 应用的认证）。

这一切都在情理之中。事实上，是 Gmail 在向 Gmail 用户索要 Gmail 凭据。所以没有违规操作；一切正常。一旦用户通过认证，Gmail 授权服务器将提示最终用户，大意是："嘿，我有一个已知的客户端 LinkedIn，需要使用你的权限访问我的 API。具体来说，它想查看你的联系人，并代表你发送电子邮件。你同意吗？"

一旦用户表示同意，授权服务器会颁发一个授权码（authorization code）（3）。授权码只是一个不透明的字符串，作为授权服务器的提醒，表明用户已为该特定客户端授予了那些权限的同意。授权码通过浏览器返回给 LinkedIn（4）。从现在起，事务的其余部分在服务端完成。

请注意：在任何所描述的事务发生之前，LinkedIn 必须先到授权服务器注册自己为已知客户端（client）。作为客户端注册操作的一部分，LinkedIn 收到了一个标识符（称为 client id），更重要的是，还有一个 client secret。client id 和 client secret 将用于在发送到 Gmail 授权服务器（特别是其 token endpoint）的请求中证明 LinkedIn 作为应用的身份。

图解的剩余部分将为你展示这是如何发生的。既然已获得授权码，LinkedIn 将联系授权服务器的 token endpoint（5），出示自己的凭据（client id 和 client secret）以及授权码，实质上是在说："嘿，这个用户对此授予了同意，我是 LinkedIn。我能获得我想要的资源访问权限吗？"

作为结果，授权服务器将颁发一种新的令牌，我们称之为访问令牌（access token）（6）。access token 是一种制品，用于授予 LinkedIn 代表用户访问 Gmail API（7）的能力，但仅限于用户同意的权限范围（8）。

这解决了"密码共享反模式"一节中描述的权限过大问题。事实上，只要 LinkedIn 仅通过尝试用户同意的操作来访问 Gmail API，对 API 的请求就会成功。一旦 LinkedIn 尝试做超出同意操作范围的事情，比如删除电子邮件，endpoint 就会拒绝 LinkedIn 的访问，因为伴随 API 调用的 access token 被限定在用户同意的权限范围内（在我们的例子中是读取联系人和发送电子邮件）。Scope（范围）是我们在这里用来表示客户端代表用户请求的权限的关键概念。这种机制有效地解决了权限过大的问题，提供了一种表达和执行委托授权的方式。

到目前为止我们所描述的是 OAuth 2.0 的典型用例，即协议最初设计的目标场景。然而在实践中，OAuth 2.0 被到处使用，并且遭受各种滥用——也就是以 OAuth 2.0 未被设计的方式来使用。请注意这些有问题的场景：每次你听到某个解决方案使用 OAuth 2.0 时，请先想到这里描述的典型用例。OAuth 2.0 支持许多其他场景，我们将在本书中讨论大部分。然而，核心意图正如我们在本节用例中所表达的。思考一个解决方案是按照这里表达的意图使用 OAuth 2.0，还是显著偏离了它，是一个有用的思维工具——帮助你验证是在处理典型场景还是需要为非标准方法做好准备。 

### 在 OAuth 2.0 之上叠加登录：OpenID Connect

让我给你演示一种特别常见的 OAuth 2.0 滥用方式。随着 OAuth 2.0 和委托授权场景开始获得关注，许多应用开发者决定他们想做的不只是调用 API。他们想在消费领域实现我们用 SAML 实现的功能。他们想让用户使用完全不同系统中的账户来登录他们的应用。将这个新需求落实到我们一直在讨论的场景中，LinkedIn 可能希望拥有 Gmail 账户的用户能够直接用它登录 LinkedIn，而无需创建 LinkedIn 账户。换句话说，LinkedIn 只是希望用户能够通过复用他们的 Gmail 账户来在 LinkedIn 上注册。

这是一个合理的想法，因为在很多情况下，人们通常不太愿意创建新账户、新密码等。所以，使账户复用成为可能本身不是坏主意。

然而，OAuth 2.0 并非为实现登录操作而设计。大多数提供商只暴露 OAuth 2.0 作为支持其 API 委托授权的方式。他们没有暴露任何正式的登录机制，因为这不是他们追求的场景。这并没有阻止应用开发者，他们直接搭 OAuth 2.0 流程的便车，来实现某种"穷人版"的登录。想象一下典型 OAuth 2.0 流程中描述的委托授权场景，再想象它在用户之前未登录 LinkedIn 的情况下发生。下图描述了这个流程：

LinkedIn 可以在没有任何已认证用户登录的情况下执行获取 Gmail API 访问权限的流程（1）。一旦 LinkedIn 成功访问 Gmail API（2），它可能会推断：”好吧，这证明了与我应用交互的人拥有合法的 Gmail 账户。”于是 LinkedIn 可能对此感到满意，并认为该用户已通过认证，在实践中可以通过创建和保存会话 cookie（3）来实现，就像我们之前讨论 SAML 方法时的登录流程一样。

这里提醒一下，我们使用 LinkedIn 和 Gmail 只是因为它们是大家熟悉的名字和用例，但我们绝不暗示它们真的以这种方式实现。

这种实现登录的模式至今仍然是常见做法。很多人这样做。这通常不是好主意，主要原因是 access token 对请求它的客户端来说是不透明的（opaque），这使得许多重要细节无法被验证。例如，access token 能成功调用 API 这一事实，并不能说明该 access token 是为你的客户端颁发的还是为其他应用颁发的。有人可能通过另一个应用合法地获得了 access token（在我们的场景中，不是以 LinkedIn 的身份，而是以其他应用的身份），然后以某种方式将该令牌注入到请求中。如果 LinkedIn 只是使用该令牌调用 API，并推理说”好的，只要我可以用这个令牌调用 API 而不出错，我就认为当前用户已认证”，那么 LinkedIn 就会被欺骗，创建一个已认证的会话。

access token 对客户端不透明的另一个后果是，攻击者可以从一个用户那里获取令牌，然后以某种方式将其注入到完全不同用户的注册操作中。LinkedIn 再次无从察觉，因为除非被调用的 API 返回可用于识别调用用户的信息，否则 API 调用成功这一简单事实不会提供任何客户端可用来确定发生了身份替换的信息。

我所描述的这些攻击被称为”混淆代理人攻击”（Confused Deputy attack），它们是在 OAuth 2.0 之上搭便车实现登录操作的经典缺陷。

更糟糕的是：用这种方法，无法标准化基于 OAuth 2.0 的登录流程。在我们的模型场景中，最后一公里是成功调用 Gmail API。如果我想将同样的模式应用于 Facebook，最后一公里就是成功调用 Facebook Graph API，这与 Gmail API 截然不同。这使得无法将这种模式固化到单一 SDK 中——一个可以用于实现行业所有提供商登录的 SDK，即使它们都正确支持 OAuth 2.0。

正是在这里，业界的主要参与者再次聚在一起，决定引入一个叫做 OpenID Connect 的新规范，它正式化了如何在 OAuth 2.0 之上叠加登录功能。我会在本书其余部分详细讲述这一努力的细节，但简而言之，该方法的核心是引入了一种新的制品，我们称之为 ID 令牌（ID token）。ID token 可以由授权服务器通过 OAuth 2.0 定义的所有流程颁发。OpenID Connect 描述了应用如何请求 ID token——而不是请求 access token（或与 access token 请求一起）。下图总结了这样的流程之一：

ID token 是一种旨在被客户端自身消费的令牌，而不是由客户端用来访问资源。ID token 的特征是它有一个固定的格式，客户端可以解析和验证。使用已知格式以及令牌是为客户端本身颁发的这一事实，意味着当客户端请求并获得 ID token 时，客户端可以检查和验证它——就像通过 SAML 保护的 Web 应用检查和验证 SAML 令牌一样。这也意味着能够从中提取身份信息——再次，就像我们学到的那样，这是 SAML 中的常见做法。这些属性使得使用 OAuth 2.0 实现真正的登录成为可能。OpenID Connect 带来的新特性不止于此：新规范引入了请求令牌的新方式，包括一种 ID token 可以直接通过前端通道（front channel）——在浏览器和应用之间——呈现给客户端的方式。这使得实现登录变得非常容易，就像我们在 SAML 案例中学到的那样，无需像典型 OAuth 2.0 API 调用模式那样使用密钥和后端集成流程。

我们在本章中看到的内容可以被视为一系列事件的粗略时间线，最终以 OpenID Connect 的创建为高潮。在接下来的章节中，我们将扩展这里描述的高层流程，深入探讨协议的细节。 

### Auth0：将复杂性挡在门外的中间人

Auth0 在这一切中的角色是什么？你可以把 Auth0 想象成一个中间人，它拥有各种协议能力，可以与几乎所有支持你所使用协议的应用对话，比如 OAuth 2.0、OpenID Connect、SAML、WS Federation。 

![image](https://cdn-mineru.openxlab.org.cn/result/2026-05-15/58666355-8904-417a-b7b5-474c257ad5bc/acfc083ad69317af1067f1c1e3b84b4af5af382f47a5d6f31b8948bb59d408ae.jpg)



图 1.10


你可以使用本章描述的任何标准协议流程将你的应用与 Auth0 集成——简而言之，Auth0 就是一个超级授权服务器。从那时起，Auth0 就可以接管认证功能。当需要认证时，你的应用可以将用户重定向到 Auth0，而 Auth0 会使用每个身份提供者所需的协议与你想要集成的不同身份提供者进行对话。如果所选的身份提供者使用我提到过的开放协议之一，Auth0 需要执行的集成就非常容易。但如果它们使用任何专有方式，对应用开发者来说也无所谓。一旦应用重定向到 Auth0，Auth0 就会处理集成细节。对你来说，只需拨动一个开关，说”我想与这个特定的身份提供者对话”——由 Auth0 中介的结果将始终以你选择用于与 Auth0 集成的开放协议所确定的格式呈现。具体来说，这就是我们之前所说的 Auth0 将问题从你面前抽象掉的意思。

此外，Auth0 提供了一种管理用户生命周期的方式。Auth0 维护自己的用户存储；它与外部用户存储集成，并暴露各种可用于管理用户的操作。例如，你可以让来自不同身份提供者的多个账户汇聚到 Auth0 和你应用中的同一个账户。你可以规范化从不同身份提供者收到的声明集合，使你的应用不必包含任何特定于身份提供者的逻辑。

我们还提供了在认证时注入你自己的代码的方式，这样你可以轻松执行自定义逻辑，如订阅、计费，或任何在你的场景中与认证同时发生才有意义的功能。

你对用户将经历的体验拥有完全的控制权，因为 Auth0 允许你定制认证用户体验的每个方面。Auth0 使你非常容易地使用这一系列功能，主要通过提供一个带有简单点击界面的仪表盘。你也可以使用 Auth0 的管理 API 以编程方式访问仪表盘所做的一切以及更多。

Identity 入门就到这里。这是对数字身份领域过去 15 到 20 年演变的一次相当快速的回顾。在接下来的章节中，我们将花更多时间深入细节。 

## 第 2 章 OAuth 2.0 与 OpenID Connect

让我们深入一步，把注意力转向 OAuth 和 OpenID Connect (OIDC) 这两个协议本身。

你读过这些协议的规范吗？我在这个领域算是个老兵了——当 CORBA、WS-Trust 以及各种老牌协议还大行其道的时候，我就已经在做身份认证相关的工作了。过去的身份协议往往异常复杂：基于 XML，具备高保证（high-assurance）特性，理解和实现的难度都很高。例如，它们使用的密码学支持所谓基于消息的安全（message-based security）——即使在普通 HTTP 上也能实现安全通信。这是一个有趣的特性，但代价是极其复杂的消息格式规则，使得实现成本高到只有最大的行业参与者才能承担。

如今的新一代协议——OAuth、OpenID Connect 等——则基于简单的 HTTP 和 JSON（一种相当简洁的格式），并且重度依赖一个前提：所有通信都发生在安全信道上。这个简单的假设极大地简化了事情——配合其他精简和裁剪，使得新协议更容易上手，至少是可读的。

然而，这毕竟不是在读《哈利·波特》。通读八十六页密集的技术语言——比如 OpenID Connect Core 规范——即使对专业从业者来说也是一项艰巨的任务。如果你从事身份领域的工作，你会发现自己需要反复、详细地查阅这些规范，像律师一样对每个字都穷究其义——这些文档信息密度极高。你还会发现这些规范有相当高的圈复杂度（cyclomatic complexity）。也就是说，规范之间存在大量交叉引用来提供上下文，而且通常冗余很少。如果当前文档中的某个概念有一个链接指向另一份规范的定义，你就必须跟随那个链接，先搞清楚那个概念，才能继续往下读。这样的规范数量非常庞大，即使你把范围限制在 OpenID Connect 和 OAuth 2.0 核心规范一两跳之内。你在 OAuth、OpenID、JWT、JWS 等周围看到的所有规范都是核心规范，描述了处理这些规范所要解决的主要场景时涉及的最基本的方面。还有一整圈最佳实践和新能力没有在这里展示。完整的图景实际上要大得多。

### OAuth 2.0 角色

让我们先从场景驱动的规范学习之旅开始之前所需的几个定义说起。OAuth 2.0 和 OpenID Connect 定义了一些原语（primitive），用于描述身份事务中发生的事情。

具体来说，OAuth 2.0 引入了几个标准角色（role），不同的参与方在身份事务中可以扮演这些角色。由于 OpenID Connect 构建在 OAuth 2.0 之上，它也继承了这些角色： 

• 第一个是 resource owner（资源所有者）。resource owner 简单来说就是用户。回想上一章中 LinkedIn 和 Gmail 的场景：LinkedIn 想要访问的资源是用户的 Gmail 收件箱，因此该场景中的用户就是 resource owner。 

• 然后是 resource server（资源服务器），它是资源的守护者，是你必须通过的门卫。通常是一个 API。在我们的模型场景中，resource server 是保护 LinkedIn 代表 resource owner 调用的那个 API 的一切——用于枚举联系人和用 Gmail 发送邮件。 

• 接下来是 client（客户端），对开发者来说可能是最显眼的实体。从 OAuth 2.0 的角度来看，client 是需要获取资源访问权限的应用程序。在我们的例子中，就是 LinkedIn 的 Web 应用。 

对于 OAuth 2.0——一个委托授权协议和资源访问协议来说，每个应用都被建模为 client。然而，我们会看到，当我们在 OAuth 2.0 之上叠加更多功能时（例如使用 OpenID Connect 进行登录），按照规范的术语，被称为 client 的实体，实际上往往就是我们要访问的资源。注意这句话中我用的”资源”不是 OAuth 意义上的，而是日常英语中的含义。你可以想象，把你要访问的资源叫做”client”有多么容易让人困惑！ 

既然你在第 1 章已经看到了 OpenID Connect 如何构建在 OAuth 2.0 场景之上，你就知道原因了。那是因为在 OpenID Connect 中，登录意味着请求一个 ID token，这是一个特殊的语义化 access token，旨在由请求者自身消费，而非用于访问外部资源。你的应用既是 client（因为它请求 ID token），又是资源本身（因为它消费 ID token，而不是用它去调用 API），但在协议术语中，我们最终用来描述应用的术语就只有 client。这对初学者来说可能令人困惑，但事情就是这样。在本书中我会经常指出这个不一致之处。 

• 最后是 authorization server（授权服务器），正如第 1 章"数字身份简介"中所定义的，它是一组 endpoint 的集合，用于驱动其中描述的（以及更多的）委托认证场景。 

authorization server 暴露了 authorization endpoint（授权端点），这是用户进行任何需要交互操作时访问的地方。实际上，authorization endpoint 返回的是网页。情况并非总是如此（我们将在 SPA 相关章节中看到），但在 authorization endpoint 上不展示 UI 的情况只是例外。authorization server 还提供 token endpoint（令牌端点），应用通常通过编程方式与之通信，执行实际获取 token 的操作。 

Authorization endpoint 和 token endpoint 在 OAuth 2.0 核心规范中定义。 

OpenID Connect 在此基础上增加了 discovery endpoint（发现端点）。这是一个标准端点，以机器可读的格式发布 authorization server 的能力信息。例如，它会列出刚才描述的两个 endpoint 的地址。discovery endpoint 提供的另一项重要信息是 OIDC client 应该用于验证该 authorization server 签发的 token 的密钥，等等。 

### OAuth 2.0 Grant 与 OIDC Flow

在 OAuth 2.0 和 OpenID Connect 的语境中，最复杂的概念通常是我们所说的 grant。简而言之，grant 就是 client 用来从 authorization server 获取某种凭据的一组步骤，目的是访问某个资源。就这么简单。OAuth 2.0 定义了大量的 grant，因为每种 grant 都针对不同 client 类型的连接能力和安全保证做了最优设计。grant 还用于解决不同的场景，比如代表用户执行访问与使用 client 自身权限执行访问等。 

我不会在这里详述各种 grant，因为本书将会把几乎所有 grant 都从里到外剖析一遍。目前只需知道 OAuth 2.0 原本定义了一组核心 grant：Authorization Code、Implicit、Resource Owner Credentials、Client Credentials 和 Refresh Token。OpenID Connect 引入了一个新的 grant——Hybrid，它将两个特定的 OAuth 2.0 grant 合并为一个流程。 

除了核心 OAuth 2.0 和 OpenID Connect 规范定义的 grant 之外，IETF 的 OAuth 2.0 工作组和 OpenID 基金会还在持续制定独立的扩展，用于解决核心规范未曾涵盖的场景，或被认为过于特定而不宜纳入核心的场景。通过添加新规范来扩展和专门化核心规范的能力是一种强大的机制，帮助社区在新场景出现时获得所需的指导。 

本书将详细考察每一个重要的 grant，特别强调特定 grant 最适合的场景、每种 grant 主要特征背后的原因，以及选择用特定 grant 解决场景时需要考虑的最重要因素。 

## 第 3 章 Web 登录

从本章开始，我们将深入探讨具体场景。让我们从最常见的场景开始：Web 登录。 

### Confidential Client

在深入其机制之前，我需要先概要介绍一些 OAuth 2.0 和 OpenID Connect 中使用的制品和术语。特别是，我想和你聊聊 client 类型。 

OAuth 2.0 中的 confidential client 是一种能够证明自身应用编程身份的 client。它可以是任何应用——authorization server 可以为其分配某种类型的凭据，使该应用在向 authorization server 发起的任何请求中都能证明自己是已注册的 client。 

这通常适用于任何单实例（singleton）应用。想想一个运行在特定机器集群上的网站。即使运行在集群上，那里运行的也是一个逻辑实体。当我在 authorization server 上注册并配置 client 时，它有明确的身份。我有确定这个 client 位于何处的 URL，还有获取双方约定的密钥的流程，可以将其在本地保存和保护。 

理论上，如果应用运行在服务器上，只有服务器管理员才能访问那个密钥。将这一切与运行在你设备上的应用对比一下：那些应用完全不是单实例的。例如，每部手机上都会有不同的 Slack 实例。当你从应用商店下载应用时，没有简单的方法能获得一个代表该特定 client 实例的唯一密钥。 

你当然不能把这样的密钥嵌入代码中，因为代码瞬间就会被反编译，你就麻烦了。而且设备始终在使用者的口袋里，你无法控制它，所以无法长时间保护密钥。有动机的黑客有无限的时间来挖掘设备中的信息——而服务器则必须先被攻破，才能泄露秘密。 

总而言之，confidential client 是适合分配密钥的 client。经典场景是运行在服务器上的网站，但也可以是 IoT 场景——你想标识设备本身而非其用户。 

另一个场景涉及长时间运行的进程。 

例如，考虑一个使用 Jenkins 的持续集成系统，它在夜间编译产品、运行测试以及类似的长时间任务。你可能希望该守护进程以其自身身份运行，而非某个用户的身份。事实上，如果使用某个用户的身份，而该用户后来离开了公司，一切都可能停摆，而且没人知道原因。这种情况经常发生，因为人们很容易忘记某个用户身份曾被用于运行这些脚本。所以，为守护进程分配独立的身份是更好的选择。 

这里有一个微妙之处：即使应用是 confidential client，也不是每一次 grant 操作都需要使用 client 凭据。这是应用具备的一种能力，但不必每次都使用。事实上，在我们即将探索的场景中，就不需要使用密钥。通常，密钥用于在你请求 token 以访问其他资源时证明你的 client 身份。而在 Web 登录的场景中，你会发现你自己就是资源。 

### 带 Form Post 的 Implicit Grant

我们在这里要使用的 grant 是带 Form Post 的 Implicit grant。这个名字有点拗口，但遗憾的是协议就是这么定义的。这在 OpenID Connect 之前是无法实现的。这是使用 OpenID Connect 实现 Web 登录最简单的方式，而且与 SAML 非常相似。实际上，它基本上遵循了我在第 1 章"数字身份简介"中演示第一个 SAML 流程时所描述的相同步骤。 

这个 grant 构成了只有 OpenID Connect 才能实现的功能基础：将网站登录与授予该网站访问 API 的委托权限结合起来。我们现在要做的是研究这个事务的一半——只看登录部分。当我们讨论 API 时，再看另一半。这两半可以组合起来，让用户体验真正流畅。同时，从设计角度来看，组合登录和 API 调用能力使得一个应用可以扮演多个角色。这是 OpenID Connect 之前无法实现的强大场景。 

由于我们使用的是 front channel，不需要使用应用凭据。这里那里会有一些安全考量，但正如刚才所说，它和 SAML 是一样的。 

从开发者的角度来看，设置这一切非常优雅。只需在应用前面安装中间件。然后，在配置中指向 discovery endpoint（如第 2 章所述），指定注册应用时获得的 client 标识符。在 authorization server 中，指定你希望接收 token 的地址，就完成了。

我们有一个使用浏览器的用户、一个由实现 OpenID Connect 的中间件保护的 Web 应用，以及一个 authorization server。 

你可能注意到，在这个 authorization server 中，我只展示了 authorization endpoint 和 discovery endpoint。我没有展示 token endpoint，因为在这个特定的流程中我们不会用到它。 

核心思路是：Web 应用一启动，中间件就会访问 discovery endpoint，获取关于 authorization server 的所有必要信息。具体来说，它会获取 authorization endpoint 的地址和用于验证签名的密钥。稍后我们会详细展示这些步骤是如何发生的（参见"元数据与发现"一节）。现在，我们先聚焦于认证阶段本身。

#### 1. 请求 Web 应用的受保护路由

在第一步中，浏览器访问应用的某个路由，该路由恰好受保护，因此不允许匿名请求访问。 

#### 2. Authorization Request 重定向

中间件拦截这个调用，并作为响应发出一个发往 authorization server 的 authorization request。HTTP 响应的状态码为 302，即重定向。它包含多个参数，用于向 authorization server 传达执行所需认证操作的全部信息。 

![image](https://cdn-mineru.openxlab.org.cn/result/2026-05-15/58666355-8904-417a-b7b5-474c257ad5bc/4e7fa72d1e6a6d655cf9887088f3aee91e0104e5d8e589ce0181dda47efc32db.jpg)



图 3.2


理解这个消息的结构非常重要，因为后续我们将看到的所有其他消息都是它的变体。这里，我们将涉及所有最相关的参数。 

**Authorization endpoint**

第一个元素是 authorization endpoint。这是我们期望 authorization server 上 authorization endpoint 功能所在的地址。

**Client ID**

`client_id` 参数是你的应用在 authorization server 上的标识符。authorization server 有一组与你的应用关联的配置设置，当收到这个特定的 client ID 时，就会调出这些配置。 

**Response type**

`response_type` 参数表示我想要的制品。在这个特定场景中，我要登录，所以需要一个 ID token。因此 `response_type` 参数的值将是 `id_token`。我可以请求各种各样的制品，包括制品的组合；我们将在后面详细介绍这些组合。 

**Response mode**

Response mode 是指我希望以何种方式接收这些制品。HTTP 提供的所有选择都可用。可以通过 query string 获取，但这通常不是好主意，因为制品会留在浏览器历史记录中。可以通过 fragment 获取，它仍然是 URL 的一部分但不会发送到服务器。可以通过 form post (`form_post`) 获取，这正是我们在这里使用的方式。在这种情况下，我们只想确保 token 通过 POST 发送到我们的 client。这样我们就不会把东西放在 query string 中——如前所述，从安全角度来看这通常是不好的做法。使用 POST 还允许我们处理大型 token。事实上，如果你把东西放在 form post 以外的任何地方，可能会遇到大小限制。 

**Redirect URI**

`redirect_uri` 参数扮演着非常重要的角色。它代表我的应用中期望接收 token 和制品的地址。我需要指定这个参数，因为在此上下文中我们使用的 token 是所谓的 bearer token（持有者令牌）。Bearer token 是仅凭持有就能使用的 token。换句话说，我可以直接使用它们，而不需要做任何其他事情——其他类型的 token 可能需要额外的操作。例如，其他类型的 token 可能要求我同时知道某个密钥并使用它。但 bearer token 不需要。关于 bearer token 你会在 token 验证一节（参见"Token 验证原则"）中听到更多。因此，我必须使用 HTTPS，以确保没有中间人可以介入并拦截流量。 

同时，指定确切的响应返回地址也非常重要。如果我不这样做——例如，不与提供的地址做严格匹配，而是允许调用方附加额外参数——就会危及通信安全。可能发生（而且过去确实发生过）的情况是：我使用的开发栈中可能存在缺陷，导致我的请求被重定向到别处。那将意味着把我的 bearer token 发送给了恶意方，而他们只需要这些 token 就能冒充我。OAuth 2.0 和 OpenID Connect 对此非常严格：你在请求中指定的 redirect URI 必须与你期望的地址精确匹配。 

**Scope**

`scope` 参数表示我请求这些制品的原因。在上面的例子中，我指定了 `openid`、`profile` 和 `email`，这些 scope 会让 authorization server 签发一个具有特定结构的 ID token。这与前面的 response type 有些冗余，但我同时还要求用用户的个人资料和邮箱信息（如果有的话）来丰富这个 ID token。 

简而言之，通过 scope，我指定了请求这些制品的原因。我们会看到，在使用 API 时，我们会请求想要获取的特定委托权限。 

**Nonce**

`nonce` 参数主要是一个防止 token 注入的技巧。在请求时，我生成一个唯一标识符并保存在某处（比如 cookie 中）。这个标识符被发送到 authorization server，最终我收到的 ID token 中会包含一个持有相同标识符的 claim。那时，我就可以将该 claim 与我保存的标识符进行比较，确信收到的 token 就是我请求的那个。如果我收到一个带有不同标识符（或没有标识符）的 token，就必须认定响应是伪造的，token 是被注入的。 

值得一提的是，我将 `form_post` 指定为 `response_mode` 的值，因为 ID token 的默认 response mode 是不同的（默认是 fragment），因此我必须显式覆盖它。下表显示了 OAuth 2.0 和 OpenID Connect 定义的每种 response type 对应的默认 response mode。如果我在请求中省略 `response_mode`，authorization server 将使用默认值： 

<table><tr><td>response_type</td><td>default response_mode</td></tr><tr><td>code</td><td>query</td></tr><tr><td>token</td><td>fragment</td></tr><tr><td>id_token</td><td>fragment (query disallowed)</td></tr><tr><td>none</td><td>query</td></tr><tr><td>code token</td><td>fragment (query disallowed)</td></tr><tr><td>code id_token</td><td>fragment (query disallowed)</td></tr><tr><td>id_token token</td><td>fragment (query disallowed)</td></tr><tr><td>code id_token token</td><td>fragment (query disallowed)</td></tr></table>

注意，即将推出的 OAuth 2.1 规范将弃用 `token` response type，因此所有包含 `token` 的 response type 也将被弃用。 

#### 3. Authorization Request

浏览器的下一步是遵循 302 重定向，执行一个 GET 请求，携带我刚才描述的所有参数到达 authorization endpoint。 

从现在开始，authorization server 会执行它认为必要的任何操作来认证用户并提示同意。这个过程如何发生并不由 OAuth 2.0 或 OpenID Connect 规定。用户认证的机制、凭据收集等完全是 authorization server 的私有事务，只要最终的响应符合标准格式即可。你可以有多因素认证、多个页面，或者只有一个页面。只要你产出的结果是标准的，怎么做都行。

#### 4. Authorization Response

一切顺利后，你会收到一个状态码为 200 的 HTTP 响应。这意味着你已经成功通过了 authorization server 的认证。authorization server 会设置一个 cookie 来代表你与它的会话。因此，如果之后你需要再次访问 authorization endpoint，就不必再显式输入凭据登录了。你可能需要授予更多同意，但不需要重新输入凭据。 

另一个值得注意的重要部分是我们请求的 ID token。它作为我们接收到的 form post 中的一个参数被返回。你可以在返回的 HTML body 中看到，JavaScript 的 onload 事件被绑定以自动提交表单。 

#### 5. 将 Token 发送到应用

当 authorization server 返回的页面被渲染后，它会将表单 POST 到我们的应用。这意味着请求的 ID token 终于被发送到了我的 Web 应用。

#### 6. Token 验证与 Web 应用会话创建

接下来发生的事情与我们在第 1 章"数字身份简介"中 Web 单点登录场景里研究的几乎一样。应用接收 ID token，根据各种信任规则以及从 discovery endpoint 获取的信息来决定是否接受它。如果接受，应用会发出一个 HTTP 302 响应，附带自己的 cookie。有了这个代表与应用已认证会话的 cookie，只要 cookie 有效，我就不需要再次获取 ID token。创建 cookie 的同时，应用发出 HTTP 302 响应，将浏览器重定向到最初请求的路由。 

#### 7. 携带授权信息请求受保护路由

当浏览器遵循重定向时，我们回到了起点：请求一个受保护的路由，但这次携带了会话 cookie。 

如果你将原始请求与这次重定向进行比较，会发现它们是完全相同的请求，只是这次多了一个 cookie。 

#### 8. 访问受保护路由

终于，经过这番来回之后，我们得到了响应：一个 HTTP 200 响应，body 中包含页面内容。 

从此以后，对应用的每个后续请求都会携带会话 cookie，证明已认证会话已经建立。 

### ID Token 的结构

如前所述，ID token 是证明成功认证发生了的制品。我们有两种方式请求它：使用 `response_type` 参数并设值为 `id_token`，以及使用 `scope` 参数并包含 `openid` 值。 

之所以有两种机制，是因为规范的作者希望即使你的 SDK 仅基于 OAuth 2.0，也能使用 OpenID Connect。事实上，在 OAuth 2.0 时代，response type 的枚举中并没有 ID token。由于 scope 作为参数是完全通用的，使用一个特定的 scope 来让 authorization server 返回 ID token 就成了一种很好的向后兼容方式。如今，这倒成了一种很容易让人困惑的设计，但既然你现在知道了，就不会再有这个困惑了。 

OpenID Connect 将 ID token 定义为固定格式——JSON Web Token (JWT) 格式。规范实际上不仅定义了格式，还定义了 ID token 中必须存在的 claim 列表。此外，它甚至以规范性语言告诉你验证某些 claim 时需要做什么。正如我们所说，如果在请求的 scope 中包含 `profile` 或 `email` 值，就会使 ID token 的内容有所不同。

### Token 验证原则

我们一直在大量讨论 token 验证，依赖于直觉认为它涉及验证签名和执行元数据发现。让我们更详细地探讨这个问题，更系统地讨论 token 验证到底意味着什么。 

我们已经看到了 token 在几个场景中发挥的功能。我们看到了用 SAML 登录，看到了用于调用 API 的 access token，特别是现在我们看到了如何用 ID token 登录。所有这些场景都涉及一个实体——资源——接收 token 并决定它是否授权调用方执行其尝试的操作。资源如何做出这个决定？ 

#### 主体确认（Subject Confirmation）

Subject confirmation 是一个从 SAML 继承来的概念。具体来说，subject confirmation method 决定了资源如何判断一个 token 是否被正确使用。 

**Bearer**

最简单的方式。类似于在地上捡到 20 美元。你捡起钱，去任何想用这笔钱的地方，用它付款，就能得到你购买的商品或服务。不会被多问任何问题，因为使用 20 美元的唯一条件就是你拥有这 20 美元并让它易手。这就是 bearer subject confirmation method 的本质。如果你拥有一个 token 的比特串，你就有权使用这个 token。 

**Proof of Possession**

更高级的方式。在 proof of possession 中，你有一个 token，它的某个加密段中包含某种密钥。这个加密是专门为 token 的预期接收者做的。其思路是：当 client 获取这样的 token 时，它还会收到一个独立的会话密钥，与嵌入在 token 加密段中的密钥相同。当 client 向预期接收者发送消息时，它像 bearer 方式一样附上 token，但同时使用这个会话密钥做些事情——比如对消息的某部分签名。 

当资源接收到 token 和消息时，它会像 bearer 方法中描述的那样以常规方式验证 token。完成之后，它会从为其加密的部分中提取会话密钥，然后用会话密钥验证消息中的签名。如果验证通过，接收者就能确定调用者就是最初获取 token 的那个请求者。否则，调用者不可能使用会话密钥。 

这种机制比 bearer 更安全：截获消息的攻击者可以重放 token，但不知道会话密钥，就无法执行额外的签名来提供 proof of possession。 

直到最近，在 OAuth 2.0 或 OpenID Connect 中几乎没有人使用 proof of possession。但现在 proof of possession 正在回归。一个名为 Demonstrating Proof of Possession (DPoP) 的最新规范展示了如何在 OAuth 2.0 和 OpenID Connect 中使用我刚才描述的机制，不过距离广泛采用还需要一些时间。因此，就实际目的而言，你可以认为 Bearer token 就是目前的通行规则。还有一个概念叫 sender constraint（发送方约束），我会在讨论原生客户端时（第 5 章"桌面和移动应用"）详细介绍。 

#### 基于格式的验证检查

在 OAuth 2.0 中，access token 没有格式。标准没有规定任何格式，主要是因为最初的设计意图是 authorization server 和 resource server 部署在一起、可以共享内存的场景。 

例如，回想第 1 章中描述的场景：Gmail 既是拥有自身 API 的 resource server，同时也是 authorization server。 

在那个特定场景中，这两个实体可以共享内存。例如，它们可以有一个共享数据库。所以，当 client 请求 access token 时，这个 access token 可以只是一个不透明字符串——authorization server 在特定表中保存用户授予 client 的权限时使用的主键。 

当 client 调用 resource server 并出示这个 token 时，resource server 拿到 token，用它查找数据库中正确的行，从而找到已授权的权限。resource server 用这些信息做出授权决策。 

这个场景既符合规范的精神，也符合规范的字面要求，而我们不需要强制规定任何特定格式。 

然而，在 OpenID Connect 的情况下，我们确实为 ID token 定义了格式。我们期望接收者真正查看 token 内部并执行验证步骤。这通常发生在 resource server 和 authorization server 不在同一位置、因此无法使用共享内存通信的情况下。在这些情况下，通常（但不总是）会依赖一种约定好的格式。 

同样，在 SAML 的情况下，我们也定义了格式——一组关于如何编码 token 的指令。 

在基于格式的验证检查中，某些约束几乎适用于所有格式，特别是 JWT： 

**用于完整性的签名**

你的 token 是经过签名的，我们已经看到了签名 token 的原因：确保 token 的来源，并防止在传输过程中被篡改。token 必须提供关于所用密钥和算法的指示信息，以便接收者可以验证其签名。 

**基础设施 claim**

Token 格式通常包含基础设施 claim，它们提供了 token 接收者必须验证的信息，以决定是否接受传入的 token。这类 claim 的一个典型例子是 issuer——即签发（并签名）token 的实体的标识符，它应该与预期接收者信任的 issuer 之一对应。另一个常见的基础设施 claim 是 audience，说明 token 是为谁签发的。你需要 audience claim 来验证 token 确实是给特定接收者的。你还需要过期时间 claim：token 通常有受限的有效期，以便有机会撤销它们。 

这些都是你期望 token 具有的 claim，中间件通常会负责验证它们。 

#### 替代验证策略：内省（Introspection）

还有一种不同的 token 验证方式，叫做 introspection（内省）。使用这种方式时，接收 token 的资源将其视为不透明的。可能是因为它没有能力验证 token。在 JWT 的情况下这应该很少见，因为检查 JWT 相当简单，任何开发技术栈都能做到。 

然而，想象一下，出于某种原因，你不能假设传入的 token 是你知道如何验证的格式。你可以拿传入的 token 发送到 introspection endpoint——这是 authorization server 可以暴露的一个额外 endpoint。既然你通过 HTTPS 连接到 introspection endpoint，你实际上可以验证服务器本身的身份。你可以确信你将 token 发送到了该去的地方，而不是恶意站点。authorization server 检查 token，判断其是否有效，如果有效，就通过同一通道返回 token 的内容（例如 claim）。 

简而言之，resource server 将 token 发回给 authorization server，说"请告诉我它是否有效"。authorization server 做出判断，将结果连同 token 的内容一起返回给 client，这样资源就可以查看 token 内部。 

就我个人而言，我不太喜欢 introspection，主要是因为它比较脆弱。你需要保证 authorization server 持续运行且可用，如果你的应用非常频繁地调用，可能会被限流。而且，这种方式下，你必须等额外的网络往返完成后才能对所调用的资源做出访问控制决策。你还可能耗尽出站 HTTP 连接——它们通常来自一个连接池。这增加了不少工作量。 

有时候确实没有其他选择。但一般来说，对于 Auth0，鉴于我们总是使用 JWT 和公钥密码学，通常更好的做法是在你自己的 API 处验证 token。 

### 元数据与发现

Token 验证中间件通过 discovery endpoint 发现有效 token 中预期的值。中间件只需访问 OpenID Connect 定义的 `./well-known/openid-configuration` URL，即可按照规范获取验证信息。 

该 URL 发布的文档通常包含我们需要直接获取的信息，比如 issuer 值、authorization endpoint 的地址等。它还链接到包含实际密钥的另一个文件，这些密钥可能就是 X.509 公钥证书的原始比特。

## 第 4 章 从 Web 应用调用 API

本章我们将注意力转向调用 API。这是 OAuth 2.0 所解决的核心场景：对 API 的委托访问正是 OAuth 存在的主要理由。

大部分讨论将围绕 OAuth 2.0 为委托 API 访问场景提供的标准 grant——Authorization Code grant（授权码授权）展开。我们还会考察其他 grant，例如 Hybrid grant（混合授权）和 Client Credentials grant（客户端凭据授权），它们适用于略有不同的 API 调用场景。

### Authorization Code grant

从宏观上看，从 Web 应用调用 API 的方式与从任何类型的客户端调用 API 大致相同。细节会有所不同，我们将在全书逐步看到。

根据客户端的类型，我们会使用具有不同属性的不同 grant。本章中，我们重点关注 Web 应用从其服务端代码调用 API 的场景。为此，我们使用 OAuth 2.0 的 Authorization Code grant。Authorization Code grant（为简洁起见，后文简称 Code grant）使一个 Web 应用能够代表用户、在用户授权的范围内访问 API。这正是我们在第 1 章介绍 OAuth 2.0 时遇到的那个 grant。

在第 1 章"在 OAuth 2.0 之上叠加登录：OpenID Connect"一节中，我们看到有些人试图将此 grant 扩展用于实现登录，而非调用 API。同一节中我们也看到，如果仅用此 grant 获取和使用 access token 来实现登录，效果并不理想。我们了解了 OpenID Connect 如何在此 grant 之上实现正确的登录方式，本章还将对此做更多讨论。此刻我只想强调：本章所关注的是调用 API，而非登录。

另一个需要事先理解的重要概念是：Code grant 只能让应用程序执行用户自身已经能够执行的操作，不能更多。实际上，应用程序通常最终拥有的访问权限会更少。用户不能通过 Code grant 授予应用程序访问用户自身不拥有或无权访问的资源的权限。人们在思考 OAuth 2.0（尤其是 Code grant）时容易产生混淆。他们观察到 API 根据 token 中 scope 的存在与否来决定是否允许调用。这容易让人误以为 scope 本身赋予了客户端访问资源的权限。实际上，scope 只是选择了用户已拥有的、并委托给客户端的权限子集。

我再次强调，Authorization Code grant 是一个委托流程。它允许客户端代表用户执行操作，这意味着用户的能力是应用程序代表用户所能做的事情的硬性上限。换言之，通过 Code grant 获取 token 的客户端不能做超出用户能力范围的事情。如果你需要客户端做超出用户能力的事情（这是常见场景），就需要切换到另一种流程：权限直接授予需要它的应用程序，无需用户参与。还不清楚？别担心，我们会在本章后面重新讨论这些要点。

在上一章中，我们探讨了如何通过 front channel（前端通道）执行 Web 登录，这使我们在无需任何密钥的情况下就能实现完整场景。正如你在流程和网络跟踪的详细描述中所见，没有涉及任何密钥。然而，在 Authorization Code grant 中，使用应用程序凭据（如 client secret）是不可避免的。每当 Web 应用兑换 authorization code 时，它都需要向 authorization server 进行客户端身份验证。

我们将根据不同的需求采取不同的方式来处理委托 API 调用场景：是只在用户在线且已登录时才需要访问 API，还是需要获取对 API 的持久访问权限，即使在用户不在场时也能调用这些 API。

我最喜欢的例子是一个可以在任意时间发布推文的应用。就我个人而言，我不喜欢早起；我真的很讨厌。然而，推文在相当早的时候发布效果最好。我在西海岸这件事让情况更糟：如果要在整个北美都算作早晨的时间手动发布推文，我必须起得非常早。幸运的是，有一些应用可以在我预先安排的任意时间代我发推。这些应用是典型的需要随时可用的 access token 来代表我调用 Twitter API 的客户端示例——无论我当前是否处于活跃登录会话中，还是在 blissfully（幸福地）熟睡。这是一个经典场景——离线访问（offline access）——展示了 refresh token 这一重要产物的需求和使用方式。我们将在本章详细探讨这一场景。

正如我们在第 1 章"委托授权：OAuth 2.0"一节中首次解释 OAuth 2.0 流程时所做的那样，我们假设用户已经登录到 Web 应用。我们不知道那次登录是如何实现的，在这个上下文中也不关心——API 调用操作可以独立于登录执行（虽然我们稍后会在 Hybrid grant 一节中看到，这里存在潜在的协同效应）。让我们详细分析消息序列。 

#### 1. 路由请求

在我们的示例场景中，用户访问了一个允许预约的 Web 应用路由。预约操作需要代表用户访问预约 API，这促使 Web 应用生成一个委托访问请求。注意，如果你将此与第 3 章"The Implicit Grant with Form Post"一节中登录操作的等效步骤进行比较，会发现 Web 应用没有前置中间件来拦截路由请求。在这种情况下，路由并不是我们要保护的资产：请求该路由恰好触发了获取 token 来调用 API 的需求。生成相应委托授权请求所需的逻辑实际上位于应用代码库内部（虽然通常由 SDK 实现而非从头编写）。 

#### 2. Authorization Request

应用对该请求的响应有些眼熟：一个包含发往 authorization server 消息的 302 HTTP 状态码响应。不过，你可以看到与第 3 章”The Implicit Grant with form_post”一节中等效步骤 2 的一些不同之处。

首先，我们设置了一个 cookie 来跟踪 nonce 值（详见第 3 章”Authorization Request Redirect”一节），因为除了访问 API 所需的 access token 外，我们还将请求一个 ID token。ID token 在此流程中很有用，它提供了关于事务的额外信息，因为 access token 本身对客户端是不透明的。本章后面会有更多细节。

接下来，在捕获的跟踪消息中，我们有 authorization endpoint。暂时先忽略 audience 参数。下一个条目是 `client_id`，代表在 authorization server 上标识该 Web 应用的客户端 ID。

此 grant 的 `response_type` 为 `code`。我们希望从 authorization endpoint 获取一个 code，Web 应用稍后将通过 token endpoint 用它换取 access token。

我们不需要指定 response mode，因为默认值即可——在 `response_type` 为 `code` 的情况下，默认 response mode 是 query，即我们期望 authorization server 在查询字符串参数中返回 authorization code。

接下来是 `scope` 参数。此消息包含之前遇到过的所有 scope 值——`openid`、`profile` 和 `email`——表示我们在 code 之外还需要一个 ID token。不过这次请求 ID token 不是为了登录目的；我们只是想了解在本次事务中授予权限的 resource owner（资源所有者）是谁。如果没有 ID token，即客户端自身可以消费的东西，我们将无从得知。我们只会盲目地获取一个 access token 并使用它，而没有任何关于获取它的用户身份的信息。

scope 集合中包含一个我们尚未遇到的 scope 值：`read:appointment`。该 scope 值代表我们要调用的 API 所暴露的一个权限；换言之，使用该 API 时可以执行的操作之一，可以通过授权检查来管控。在 authorization request 中呈现该 scope 值，客户端等于向 authorization server 声明：”此 Web 应用希望代表用户行使 `read:appointment` 权限”。这是 authorization server 需要知道的信息。它将决定请求处理方式中的重要细节，例如呈现给用户的同意提示内容，以及授予委托权限的实际结果。

下一个参数是 redirect URI，你已经很熟悉了。捕获的消息中最后一个参数是 nonce，这是我们在书中前面遇到的 token 注入防护机制。

现在我们已经详细讨论了每一个消息参数，让我们回过头来看 audience 参数。当请求受 Auth0 保护的 API 的 access token 时，客户端需要指定一个额外参数 `audience`，用于指示客户端请求访问的资源身份。

OAuth 2.0 核心规范不包含执行此功能的参数，主要是因为存在一个潜在假设（虽然不是要求）：resource server 和 authorization server 是共置的。这个假设使得标识请求针对哪个 resource server 变得不必要。以 Facebook 使用 OAuth 2.0 来管控其 Graph API 访问为例，可以具体说明这种情况。Facebook 的 authorization server 只能为 Facebook Graph API 颁发 access token；场景中没有其他 resource server。客户端唯一的自由度是为那一个 resource server（Facebook Graph）指定不同的 scope。不同的 scope 表达不同的权限和我打算执行的操作，但它们都指向同一个 resource server，无需在 authorization request 中显式命名。Google、Dropbox 和其他流行服务也有类似的情况。每当客户端从这些服务获取 token 时，它们总是在调用提供商自己的 API，其身份从上下文中不言自明，不需要在请求中提供标识符。

当解决方案包含第三方 authorization server 时——例如 Auth0 客户利用 Auth0 authorization server 来保护自己的自定义 API——拓扑结构允许同一个 authorization server 管控对众多资源的访问，这些资源可以部署在不同位置。在这种场景下，客户端需要能够指定它打算请求访问哪个资源。

构造包含特定 resource server 显式引用的消息有多种方式。例如，API 可以将 resource server 标识符嵌入各个 scope 字符串本身中。但这种方式有问题：scope 字符串可能变得非常长且难以阅读。而且，在同一请求中包含引用不同资源的多个 scope 可能会导致歧义，不清楚生成的 access token 可以用于哪些资源。

鉴于这些复杂性，Auth0 和其他身份供应商决定引入一个专用参数来标识资源。例如 Azure AD 有一个 `resource` 参数，其语义与 Auth0 的 `audience` 等价。

在这些个别供应商做出决策之后，IETF OAuth 2.0 工作组正式认可了此类原语的实用性，并发布了新规范 OAuth 2.0 Resource Indicators。该规范通过一个 `resource` 参数扩展了 OAuth 2.0，其作用与 Auth0 的 `audience` 完全等价。我们计划在未来的更新中也开始接受这些标准参数。 

#### 3. 302 重定向执行

接下来，浏览器执行 302 HTTP 状态码重定向，将我们分析的消息发送到 authorization endpoint。

#### 4. Authorization Response

authorization server 收到 authorization request 后，负责处理流程中的交互部分。

authorization endpoint 判断需要什么来验证用户身份，并完成该过程。然后向用户呈现一个同意提示："客户端 X 想要代表你读取预约。"用户同意后，authorization endpoint 按照我们请求的 `response_type`，在查询字符串中返回所请求的 authorization code。此外，响应还包含常规的 set-cookie 指令，authorization server 借此在浏览器中记录已建立认证会话。

#### 5. 将 Authorization Code 提交给 Web 应用

此时，浏览器只需执行重定向，将 authorization code 分派给 Web 应用。从这一刻起，Web 应用将在服务端继续流程。 

#### 6. 兑换 Authorization Code

Web 应用将 authorization code 与其自身的 client 凭据组合，通过消息发送给 token endpoint。

![image](https://cdn-mineru.openxlab.org.cn/result/2026-05-15/58666355-8904-417a-b7b5-474c257ad5bc/cce27e7ba865b8147df2ee9d282dace154f3dbd628853def64199d05b93e9ca0.jpg)



图 4.2


发往 token endpoint 的消息是一个 HTTP POST 请求，应用在其中提交其 `client_id` 和 `client_secret`、从 front channel 接收的 authorization code，以及一个新参数 `grant_type`。消息布局如图 4.2 所示（带标注）。

每次应用与 token endpoint 通信时，都必须指定所需的 grant type，让 authorization server 知道如何解读请求。在本例中，所需的流程是 `authorization_code` grant。这告诉 authorization server 在消息中查找 authorization code，并在该特定 grant 的上下文中考虑 client ID 和 secret。例如，如果请求将 `client_credentials` 指定为 grant type（我们稍后讨论的流程），authorization server 就会忽略 authorization code，只查看 client ID 和 client secret，且只考虑客户端应用自身的身份，而非 authorization code 所隐含的 resource owner 的同意选项。换言之，`grant_type` 参数用于消除歧义，明确客户端期望 authorization server 执行的流程。

请求也可以包含 audience。不过在这种情况下它是冗余的。authorization code 已在该 audience 的上下文中颁发，authorization server 知道这一点，因此无需在此请求中再次提供。在请求中加入 audience 有助于提高清晰度：例如，在检查网络跟踪时，无需关联回溯到之前的消息即可了解此请求的目的。

最后，消息包含一个 `redirect_uri` 参数。在这个阶段，由于客户端通过直接通道与 authorization server 通信，authorization server 实际上没有机会执行重定向。`redirect_uri` 在此处是作为防止 redirect URI 篡改的安全措施——authorization server 会验证此处呈现的 `redirect_uri` 与流程中 authorization code 请求阶段提供的 `redirect_uri` 是否一致，防止攻击者执行 URI 替换（参见 https://tools.ietf.org/html/rfc6749#section-10.6）。 

#### 7. 在 Token Endpoint 响应中接收 Access Token

假设请求被 authorization server 接受且处理无误，grant 以一个响应消息结束，携带步骤 2（Authorization Request）中 `response_type` 原本指定的产物——在本例中是一个 access token。以下是响应消息内容的分解：

- 所请求的 access token。

- 一个 ID token，这是对请求的 scope 值列表中包含 `openid` 的响应。

- Token 类型，目前始终为 Bearer——如 token 验证一节中所讨论的。

- `expires_in` 参数，表示 access token 应被视为有效的时间长度。虽然 access token 本身有时可能包含该信息且恰巧采用可检查的格式，但客户端应始终将 access token 视为不透明的。因此，`expires_in` 需要作为响应中的参数提供，以便客户端使用该信息（例如，决定 access token 应被缓存多长时间）。

#### 重要提示

客户端应用应始终将 access token 视为不透明的，因为其内容和格式是 authorization server 和 resource server 之间的私事。authorization server 和 resource server 之间的约定条款可能随时变更：如果客户端应用包含依赖解析 access token 内容能力的代码，即使是微小的变更也会导致代码失效——通常无法补救。

想象一下这种情况：access token 最初以明文发送，后来开始被加密，使得只有预期的资源接收者才能解密。任何客户端都将失去对 token 内容的访问能力。依赖访问 token 内容能力的客户端代码将不可挽回地崩溃。总之，避免在客户端应用中编写检查 access token 内容的逻辑。在网络跟踪中检查 token 内容用于故障排除是完全没问题的，因为信息将通过调试工具消费，不会产生未来可能失效的代码。 

#### 8. 使用 Access Token 调用 API

一旦客户端获得所请求的 access token，就可以调用 API 了：只需在经典的 REST 调用中包含 access token 的比特位即可。在本例中，调用是 GET 方法，但任何 REST 调用风格都是可以的。该消息的关键特征是 Authorization HTTP 头，它展示了 Bearer 认证方案并携带 access token 的比特位。

OAuth 2.0 Bearer Token Usage 规范——描述如何使用通过 OAuth 2.0 获取的 bearer token 来访问资源的文档——指出可以将 token 放在传出请求的其他位置，例如调用体中甚至请求链接中作为查询参数。遇到在请求体中发送 token 的客户端非常罕见。使用查询字符串发送 access token 是被积极反对的做法，因为它有重大的安全隐患。考虑客户端在浏览器中运行的情况：每当 token 包含在查询字符串中时，其比特位就会出现在浏览器历史记录中。任何能导出浏览器历史记录的攻击也会暴露 token。此外，如果 API 调用之后紧跟一个重定向，查询字符串将在 referer 头中对重定向目标主机可见：这同样会将 token 暴露在正常的客户端-资源交换之外。

基于这些及其他原因，可以合理预期，在依赖 OAuth 2.0 的实际 API 调用中，几乎所有调用都将使用 Authorization HTTP 头。 

### Authorization Code grant 与 PKCE

最新的 OAuth 2.0 Security Best Current Practice (BCP) 文档建议每个 Authorization Code 流程都应使用 Proof Key for Code Exchange (PKCE，RFC 7636)，这是 authorization code grant 的一个扩展，旨在保护 Authorization Code 不在传输中被窃取。PKCE 最初是为公共客户端（public client）设计的，在那里它执行关键的安全功能，我们将在下一章详细描述。我们选择在本节保持简略，将 PKCE 的介绍推迟到下一章，因为届时你对原始 grant 将更加熟悉，将 PKCE 作为增量步骤添加会更容易理解。不过，我们希望在此就指出 BCP 的指导，这样如果你在其他地方读到相关内容，就知道是怎么回事。 

#### 侧栏：授权的核心概念与术语

OAuth 2.0 提供了一个委托授权框架。遗憾的是，开发者常常忽视”委托”这一部分，试图用 OAuth 原语和流程来解决协议并未明确设计来处理的纯授权场景。结果是在简单示例场景中看似可行的方案，一旦应用到更真实的环境中就会捉襟见肘。

因此，花几个段落讨论授权中的核心概念和术语是一项值得的投资，明确阐述它们与 OAuth 的关系——特别是，哪些是 OAuth 的一部分，哪些是我们所暴露的底层资源的属性。

**权限（Permissions）**

假设你想为一个已有资源提供编程访问。根据资源的性质，可以在其上或利用它执行不同的操作集。在文档编辑系统的上下文中，用户可以查看、阅读、评论或修改文档。封装打印机的 API 可能暴露黑白打印或彩色打印的能力。任何类型的资源都会有一组对该特定资源有意义的权限，这些权限可以对特定调用者允许或拒绝。权限就是这样的东西：描述可以对资源执行的操作类型的声明——`document:read`、`document:write`、`print:bw`、`print:color`、`mail:read`、`mail:send` 等等。

权限描述的是资源的内在属性，无论这些资源如何暴露都存在。如果权限在涉及这些资源的委托授权场景中有用，OAuth 2.0 解决方案可能会将它们呈现出来。但在一般情况下，权限自身独立存在，也将在 OAuth 之外使用。

**特权（Privileges）**

特权是被分配的权限：它声明某个特定主体（principal，比如 John）可以对给定资源执行特定操作（比如调用打印机 API 进行全彩打印）。

与权限一样，特权的概念独立于 OAuth（或任何其他更高层协议）而存在。例如，描述特权所需的框架需要用于主体（可能被分配权限的用户和应用）的原语，而 OAuth 2.0 并未定义这些。

应用于一组资源的权限和特权的存在将影响基于这些资源的 OAuth 2.0 解决方案的行为，但这在 OAuth 2.0 规范定义的协议和消息中并没有直接描述。

**Scope**

终于可以讨论 OAuth 原语了。在资源需要在委托授权解决方案的上下文中被暴露的情况下，scope 是使客户端应用能够请求对给定资源的特定权限行使用户特权（privilege）的原语。客户端向 authorization server 表达这一意图的机制是在 authorization request 中包含与所请求权限对应的 scope。当以这种语义使用时——即给定资源的权限列表——scope 用于定义客户端应用希望代表用户行使的用户特权子集。注意 scope 也可用于其他目的：我们已看到 `openid`（请求额外产物的存在，即 ID token）或 `profile`、`email`（影响返回内容）的例子。 

**有效权限（Effective Permissions）**

我们终于可以把所有这些概念如何相互关联拼凑起来了。

考虑一个经典的委托授权流程，客户端请求 authorization server 访问某个资源。具体来说，客户端指定在其打算对资源执行的操作中需要哪些权限。authorization server 收到请求并验证用户身份后，通常会提示用户授予应用对相应权限的委托访问。用户通过该提示授予同意，实际上是在说："是的，我同意这个特定客户端代表我行使所请求的特权。"

例如，客户端实现了一个邮件解决方案，它请求的权限是 `mail.read`。请求的 scope 是 `mail.read`，返回的 access token 将包含（按值或按引用，取决于格式）`mail.read`。

一旦客户端获得 access token，它将用它来调用 API 并请求读取邮件列表。API 收到并验证 access token 后，保护 API 的中间件将验证 token 携带的 scope 是否包含 `mail.read`——即 API 执行所请求的读取操作所需的权限——然后允许请求继续执行。

但授权检查还未结束！假设客户端请求的邮件列表来自与授予同意并获取 access token 的用户不同的用户的收件箱。API 应该允许请求成功吗？当然不能！Scope 不会在不存在特权的地方创建特权。Scope 只能授予客户端 resource owner 对资源所拥有的特权的子集，永远不能添加 resource owner 不具备的特权。有效权限是 resource owner 拥有的特权与授予客户端的 scope 的交集。有效权限代表客户端实际能做的事情，它可能是 scope 中声明内容的子集。你始终需要在运行时检查 scope 是否代表 resource owner 对被访问资源实际能做的事情。

另外请注意，resource owner 在授予同意时拥有的特权并不保证会永久保留。因此，即使你的 authorization server 将 scope 和特权混为一谈（例如，只在用户拥有相应特权时才允许用户同意），也没有什么能阻止这些特权在之后被撤销。这使 API 必须进行检查，而不能仅仅依赖传入的 access token 中的 scope。这是 OAuth 语境中经常被误解的一个微妙要点。

注意 OAuth 也可用于应用对应用的流程，其中不涉及用户。客户端仅通过自身的 client 凭据从 authorization server 获取资源的 access token，而不是代表 resource owner 请求访问。你可以说在这些场景中，客户端应用本身就是 resource owner：没有委托，因此不需要 scope 来限制涉及的特权。我们将在本章后面的小节学习相应的 OAuth 2.0 grant——Client Credentials grant。在这种情况下，权限如何表达并不完全明确，因为 OAuth 2.0 核心规范没有提供表达已分配特权的机制（不过有一个新规范——JWT Profile for OAuth 2.0 Access Tokens——确实引入了一些相关指导）。无论这些特权如何表达的实现细节如何，这是一种特权实际被携带在 token 中的情况。也可能存在其他情况，authorization server 在 access token 中包含用户特权、角色、组成员关系和其他授权信息。这些情况都是有效的，代表了真实且重要的场景。但它们不是我们本书所研究的规范所描述的，因此我们不再展开。

最后，请注意虽然 scope 通常映射到权限，但这并非总是如此。还记得 `openid` scope 吗？它在请求中的存在只是导致 authorization server 的响应中包含一个 ID token。或者想想 `profile` scope，添加到请求中时会导致 ID token 包含原本不会出现的 claim。因此很容易在权限和 scope 之间建立映射。Scope 在许多常见场景中确实对应权限，这可能错误地造成 scope 和权限是同一概念的印象，但事实上，记住它们不是同一概念非常重要。 

### Refresh Token grant

现在让我们回到 grant 的话题。我之前顺带提过：token 通常有过期时间。token 之所以有过期时间，是因为 token 缓存了一些事实和用户属性，而这些事实可能在 token 颁发后发生变化。

此外，客户端在某一时刻能够获取 token 并不能保证同一客户端将来能获取相同的 token。例如，resource owner 可能访问 authorization server 并撤销该客户端获取之前授予的 scope 对应 token 的同意。这使得任何先前颁发的 token 的内容变得过时，因为它们不再反映当前状况。

其思路是，通过赋予 token 较短的有效期，确保客户端不能真正使用它们（因此也不能使用它们缓存的信息）太长时间。token 过期后，客户端将被强制回调并重复请求以获取新 token。这个新请求为 authorization server 创造了机会，使其可以颁发包含最新信息的新 token，或者在条件已变更时拒绝颁发新 token（例如，用户账户已从系统中删除）。

token 有效期越短，颁发的信息就越新。解决方案通常寻求在 token 有效期与性能和流量考量之间取得平衡。

当然，这带来了另一个挑战：虽然我们确实需要最新信息，但我们不希望为此给用户带来糟糕的体验。用户应该对幕后为实现这些更新而展开的所有低层机制毫不知情。我们需要让客户端能够以不影响用户体验的方式续期 token。OAuth 通过引入一个新产物——refresh token——以及相关的 grant 解决了这个问题，这些 grant 用于在不显示提示的情况下处理 token 续期。

使用 refresh token 的第一步是请求一个。OAuth 2.0 核心规范没有定义请求 refresh token 的机制，将是否颁发 refresh token 的决定留给各个 authorization server。然而，OpenID Connect 确实定义了请求 refresh token 的机制，结果是大量 OAuth 2.0 authorization server 采用该机制作为请求 refresh token 的主要（甚至唯一）方式。

#### 1. Refresh Token 兑换请求

该 grant 的第一步是一个典型的 token endpoint 请求，类似于本章前面描述的 code 兑换请求。

检查请求，你会遇到以下参数：

- 常规的 `client_id`

- `client_secret`。这是一个 confidential client（机密客户端），因此对 token endpoint 的请求要求客户端应用表明自身身份。

#### 2. Refresh Token 响应

authorization server 的响应返回一个新的 access token、一个新的 ID token（因为原始请求包含 `openid`），以及获取 refresh token 时被授予的 scope 列表——在本例中是在 Authorization Code grant 期间。

authorization server 返回已授予的 scope 列表，因为客户端可能并不真正知道这个特定 refresh token 最初被授予了什么，或者 authorization server 的条件自原始颁发以来是否发生了变化。此外，客户端可以请求特定的 scope 列表，但 authorization server 始终可以决定只返回这些 scope 的一个子集。在这种情况下，如果 authorization server 不返回在此特定 refresh token 兑换上下文中已授予的 scope 列表，客户端将无从得知。

即使客户端记住了最初请求的 scope，也不能保证该列表是准确的。请记住，客户端必须将 access token 视为不透明的，因此不能简单地查看 access token 来了解。

在本例中，authorization server 没有在 access token 和 ID token 之外返回新的 refresh token。

客户端应保留在第一次流程中收到的 refresh token 比特位，并持续使用直到过期。

在各种场景中，authorization server 确实会在每次 refresh token grant 时包含一个新的 refresh token。最值得注意的是一种称为 token rotation（令牌轮换）的安全措施上下文中的情况。

Token rotation 保证每当你使用一个 refresh token 时，该特定 refresh token 的比特位将不再用于任何未来的兑换尝试。每次使用 refresh token 都会导致 authorization server 使其失效并颁发一个新的，与续期后的 access token 一起返回。客户端需要准备好丢弃旧的 refresh token，并在每次续期操作时预期存储新的。

任何使用旧 refresh token 的尝试都会导致 authorization server 断定请求发起者窃取了它。这可能触发保护措施，例如在泄露表明应用被入侵的情况下，使同一认证会话中创建的所有其他 token 失效。注意对于 confidential client 而言，这种措施可能过度，因为对合法客户端的使用是通过要求应用在兑换 refresh token 时使用其 `client_secret` 来强制执行的。然而，这对公共客户端（public client）极为有用，因为公共应用可以在不展示任何应用凭据的情况下兑换 refresh token。更多细节将在下一章关于原生和移动客户端中讨论。

#### 3. 调用 API

新的 access token 将以与旧的完全相同的方式使用：所有关于按照 OAuth 2.0 Bearer Token Usage 规范调用 API 的讨论都适用。 

#### 关于 Refresh Token 的一些考量

客户端通过在 scope 中包含 `offline_access` 来请求 refresh token，这一事实向 authorization server 表明，所生成的 refresh token 的生命周期将与执行 grant 所在的认证用户会话的生命周期解耦。换言之，用户是否通过 front channel 登录到应用，与同一应用能否兑换 refresh token 并无关联。

同样，应用仍然能够使用有效的 refresh token 这一事实，并不说明最初帮助获取该 refresh token 的用户是否有活跃的登录会话。这两件事完全分离。

`offline_access` 旨在支持的场景正是我在本章开头描述的那个：用户希望安排一条推文在未来某个时间发布，无论届时用户是否登录。更一般地说，它解决了应用可能需要在没有用户在场处理交互请求的情况下获取有效 access token 来调用 API 的场景。开发者常犯的一个错误是将应用后端兑换 refresh token 的能力解读为用户仍有会话的证明。根据上述解释，这是一个危险的错误，可能导致通过退出登录重新复活已经过期或终止的会话，使 front channel 会话管理失效。

在开发即使没有活跃用户会话也需要调用 API 的应用时，应用显然需要持久化 refresh token，使其在交互会话不存在时仍然可用。即使 API 调用被限定在交互会话的生命周期内，如果你想避免在 Web 服务器内存回收时让用户重新经历 token 获取流程，也需要将 token 保存在内存之外的地方。当然，持久化 refresh token（以及 token 的通用情况）需要谨慎。重要的是确保 token 按用户存储，防止某个用户最终访问并使用与另一个用户关联的 refresh token 的可能性。这与执行会话隔离所需的基本卫生要求相同，但当涉及 token 时，遵循最佳实践更为关键，因为身份混淆的影响巨大，且在交互会话生命周期之外持久化用户数据会带来复杂性。

结束本章关于 refresh token 的话题之前，给出最后一条建议。即使你知道与 refresh token 关联的过期时间，也不应在代码中依赖它。refresh token 可能因多种原因停止工作，与其预期的过期时间无关。例如，用户可能撤销同意，立即使基于先前同意颁发的 refresh token 失效。另一个例子：resource server 可能更改策略，规定从那时起只接受通过多因素认证获取的 access token。这使得任何通过单因素会话获取的 refresh token 无法获得有效的 access token，并迫使客户端通过多因素认证重新获取新的 refresh token。同样，所有这些可能发生在原始 refresh token 的声明过期时间之前。基于所有这些原因，审慎的做法是在开发客户端代码时假设 refresh token 可能随时停止工作，并在前期嵌入适当的错误管理和补救逻辑。 

#### 侧栏：Access Token 与 ID Token 的对比

你现在已经有机会看到 access token 和 ID token 的实际应用。同样重要的是，你了解了 OAuth 2.0 和 OpenID Connect 引入这两种产物的根本原因。值得退后一步总结两种 token 类型的差异，因为混淆何时使用哪种 token 是你作为身份从业者将遇到的最常见挑战之一。

**Access Token 回顾**

Access token 是旨在使客户端应用能够访问资源（通常代表 resource owner）的产物，授予客户端应用委托授权。如前所述，OAuth 2.0 没有规定 token 格式。

前面我们讨论了 authorization server 和 resource server 共置的常见拓扑的影响。这种拓扑允许它们访问共享内存，使得为 access token 使用特定格式变得不必要。

相反，考虑 authorization server 与 resource server 分离的情况，例如 Auth0 这样的身份即服务产品，同一 authorization server 被多家公司拥有的多个 resource server 共享。这种场景确实可以从商定格式并使用它来验证传入 token 中受益，即使协议本身没有提供开箱即用的方案。使用 JWT 作为 access token 的格式如此普遍，以至于我推动了一项标准化工作来定义其互操作性配置文件。

不惜啰嗦也要强调的是：作为客户端应用开发者，你绝不应该编写检查 access token 内容的代码。在某些情况下你可能知道正在使用特定的 token 格式，这一事实并不能改变这一点。之所以这不是好主意，更多是关于客户端、资源和 authorization server 之间的契约。实际上，你经常有机会查看 access token 内部，但情况可能随时改变。Access token 中使用的格式是 resource server 和 authorization server 商定的事项，细节可以随时由它们自行决定更改，而无需通知客户端。任何基于对 access token 内容的假设编写的代码，在这些假设不再成立时就会失效，有时甚至无法补救。想象一下信息被移除或内容被加密，使得只有 access token 的预期接收者才能检查它。虽然开发者在故障排除期间读取可用的任何信息（包括捕获的 token 内容）是合理的，但日常执行此操作的代码往往会导致停机和严重的生产问题。 

**ID Token 回顾**

ID token 旨在支持登录操作，并可选地将认证信息提供给客户端。它们不包含任何委托授权信息（尽管没有什么能阻止实现者用自定义值扩展规范中描述的默认 claim 集）。ID token 在用户登录时发挥作用，客户端可以使用它们了解认证流程中发生了什么。与客户端确实不应该检查 access token（如前几段详细讨论的）不同，客户端必须查看 ID token 内部——这是 Web 登录一章中描述的验证步骤的一部分，也是 OpenID Connect 核心规范所要求的。

关于 ID token 最常见的困惑之一是它们能否用于调用 API。简短回答是：不应该。让我们花点时间理解为什么人们会尝试这样做，以及为什么这通常不是个好主意。

ID token 旨在支持登录操作。客户端应用同时是 ID token 的请求者和接收者：一旦客户端收到 token，它就到达了目的地，不应再继续传递。客户端需要做的只是验证它并在存在用户属性时提取出来。两者都是可以本地完成的操作，因为 ID token 具有固定格式，而 OpenID Connect 规范详细说明了如何执行验证。ID token 不应离开客户端应用的最终证明在于 `aud` claim（受众声明），它通过携带客户端的 `client_id` 值来形式化客户端应用是预期接收者。我们在第 3 章”ID Token 解剖”中讨论了所有这些。

尽管如此，在实际情况中确实存在客户端应用使用 ID token 调用 API 的情形。通常这是因为设计者没有完全理解底层协议，特别是 audience claim 的作用。对他们来说，JWT 就是 JWT，而 ID token 通常更容易获取，因为它不需要注册 API、定义 scope 以及根据每个特定 authorization server 的要求调整验证技术。例如，有些 authorization server 不使用 JWT 作为 access token 的格式，而需要支持 introspection 调用。另一些可能根本不是为保护第三方 API 而设计的；因此不提供 API 注册和 access token 颁发与验证功能，但仍会为登录目的颁发 ID token。

一般来说，使用 ID token 调用 API 存在问题。主要问题触及了我们之所以有 audience 的根本原因。接收 ID token 的 API 只能验证该 token 是为那个特定客户端颁发的：token 中没有任何内容表明它是为了调用这个特定 API 而颁发的。除了无法为特定 API 插入临时 claim 的实际问题外，还有严重的安全隐患：泄露的 ID token 现在不仅可以用于访问客户端，还可以用于调用此 API 以及遵循相同策略的所有其他 API。

正确设置 scope 的 token 可以限制泄露事件的影响范围（scope 为 API A 的 access token 只能与 A 一起使用），而多个 API 接受 ID token 意味着它们将全部同时被入侵。这也使得在 API 之间维持隔离变得非常困难：如果 A 和 B 都接受 ID token，那么当客户端调用 A 时，A 可以转而使用从客户端收到的同一 token 来调用 B。虽然这在某些情况下可能是可接受的，但在一般情况下，这绝不应作为副作用发生。

最后，我要提到使用 ID token 调用 API 无法通过发送方约束（sender constraint）来保护，因为支持它的协议不会提供任何机制将 ID token 关联到客户端和 API 之间的通道。

为了全面起见，我想承认一种特定情况：使用 ID token 调用 API 可能不会导致灾难性后果——尽管它永远不会像使用 access token 那样好。考虑客户端应用和 API 本身恰好是同一逻辑应用的情况。这就是通常描述为”第一方应用”（first-party app）的场景，两端拥有相同所有者并紧密耦合以实现特定解决方案。以社交网络 API 及其客户端应用为例。在这种情况下，解决方案不严格要求委托，传入的 token 可能被期望用于识别用户，颁发给该客户端的 token 不会被任何 API 接受——除了第一方 API（排除个别应用所有者决定接受它们的情形，这不在第一方解决方案开发者的控制范围内）。

从最终用户的角度看，构成解决方案的客户端+API 整体是一个逻辑整体——我通过 Twitter 应用使用 Twitter 账户的体验通常不需要任何明确提及 API 的特殊同意。在这种情况下，可以认为请求 token 的应用组件和实现 API 的组件实际上是同一实体，可以用同一标识符表示——因此，关键步骤来了，被具有相同 audience 的 token 所指向……就像 ID token 一样。

有一次，在喝啤酒的时候，OpenID Connect 规范的一位作者告诉我，ID token 不过是具有专门语义的 access token。话虽如此，使用 ID token 调用 API 通常仍然不值得。虽然范围狭窄的第一方场景确实存在，但用 access token 实现它们仍然更好（想想前面提到的发送方约束限制），而且越界使用 ID token 并使你暴露于严重安全风险的风险太大了。我在这里提到这个特定情况是因为，如果你在这个领域工作足够长时间，你很可能会在实际中遇到这种做法，我想让你有能力理解其中细微差别和遵循该做法的人的观点：然而，最佳实践仍然是使用 access token 调用 API。如果你需要 JWT 格式的 access token，请使用前面提到的 JWT Profile for OAuth 2.0 Access Tokens。 

### ID Token 与 Back Channel

OpenID Connect 提供多种不同的登录方式。我们在上一章学习的方式利用 front channel。它依赖 Implicit flow（即直接从 authorization endpoint 颁发 ID token）加上 form post（将 token 传输到后端托管的逻辑，这是基于重定向的应用的惯例）。该流程恰好拥有最少的组件，因为它不需要客户端应用获取、管理和使用 client secret。该流程与传统协议（如 SAML 或 WS-Federation）具有大致相同的安全特性，这些协议仍广泛用于任务关键型、高价值场景。

我们在本章中刚刚学习的用于调用 API 的 Authorization Code grant 也可以并且通常用于执行登录操作——通过按照我们学习请求 access token 的相同步骤来获取 ID token。假设你处于这样的场景：出于某种原因，你不希望将 ID token 的比特位暴露给用户浏览器。使用 Authorization Code grant，你可以让一切在服务端进行。你可以像获取调用 API 的 token 那样执行 Authorization Code grant：只需同时请求 ID token。注意，这正是我们在 API 调用场景中所做的——在初始请求中包含 `openid` scope。要使该操作算作登录，我们只需验证该 ID token 并基于其内容创建 front channel 会话。

与 front channel 的显著区别在于，由于客户端通过直接 HTTPS 连接从 token endpoint 获取 ID token，ID token 比特位的来源是确定的。客户端确切知道 ID token 直接来自 authorization server，没有可能在传输中篡改内容的中间人。来源和完整性得到验证后，就不需要验证 ID token 的签名了。想想看：如果你要验证签名，你会使用从发现文档中获取的密钥。那你为什么相信那是正确的密钥？因为你是通过 HTTPS 直接通道获取发现端点的！同样的假设适用于从与 token endpoint 的直接连接获取 ID token 的情况，这就是为什么客户端可以跳过签名验证。

非常重要需要理解的是：不需要验证签名并不意味着客户端可以跳过 token 验证！客户端仍需要验证 audience、issuer、过期时间以及 OpenID Connect 规范描述的 ID token 验证的所有其他检查。签名只是接收者为验证传入 token 应执行的众多检查之一，即使在 front channel 情况下也是如此。

然而，请记住，虽然与 token endpoint 的直接 HTTPS 连接向你保证了 token 的来源，但它并不确保你收到的 token 就是你请求的那个。在初始请求和与 authorization server 交换 authorization code 之间可能发生了 authorization code 注入，而你的应用无法察觉。这就是为什么 OAuth 2.1 也建议对 confidential client 使用 PKCE。

通过 Authorization Code grant 获取 ID token 在技术上比通过 front channel 接收更安全。然而，这种技术更为繁重，因为它要求客户端获取、保护和使用应用凭据——这有管理成本、相关风险（比如在源代码控制中遗忘密钥）、性能和可用性挑战（额外的服务器调用）。如果你的应用只需要登录用户，并且对 token 通过浏览器传输没有特别限制，front channel 技术就能正常工作——多年成功的 SAML 部署使用类似技术保护高价值场景就是证明。如果你确实处于需要更高安全性的情况，或者已经在执行需要 authorization code flow 的 API 调用，你可能考虑按照本节所述通过 back channel 实现登录。 

### UserInfo Endpoint

客户端在请求 ID token 时如果不指定 `profile` 和 `email` scope 值，将收到一个骨架 token，表明用户 X（通常由不透明标识符表示）成功通过了颁发者 Y 的认证。Token 还指定了时间和可能的认证模式，但没有其他信息——特别是没有用户属性。

客户端可能出于多种原因选择这种最简 ID token 内容。例如，客户端可能想要这样的 token 来使用易于设置的 front channel 登录流程，同时避免向浏览器泄露个人身份信息（PII）。或者，客户端可能只是想在带宽有限的网络上减少数据传输量，或者在按流量计费的连接上，较大的 ID token 可能导致用户支付更多数据费用。

好消息是，客户端可以选择使用最简 ID token，并在需要时仍然获取用户属性。OpenID Connect 引入了一个新的 API 端点，称为 UserInfo endpoint，可以通过出示适当的 access token 来检索用户信息——遵循本章前面学习的 OAuth 2.0 Bearer Token API 调用技术。每当客户端需要了解用户信息时，无论是因为没有保存初始 ID token 还是收到了一个最简 token，都可以使用先前获取的 access token 向 UserInfo endpoint 发起请求。它将收到的内容实质上等同于客户端在请求带有 `profile` 和 `email` scope 的 ID token 时会获得的内容。

第一章描述了从 OAuth 2.0 到 OpenID Connect 的演进。一个关键段落是关于一种滥用 OAuth 模拟登录的方式，其中成功用 access token 调用 API 的能力被视为客户端认定用户已登录的充分证明。这有几个问题：access token 不能与特定用户绑定（如果你试图认证，即登录，这非常重要），不能证明是作为该应用的登录操作的一部分而颁发的，而且由于每个提供商保护不同形态的 API（Facebook Graph、Twitter API 等），无法标准化。

UserInfo endpoint 解决了第一个和第三个问题。UserInfo 响应确实提供了获取用于保护调用的 access token 的用户信息——而且由于它是标准的，可以构建通用 SDK 来使用它。这使得客户端可以以标准化的方式实现纯 OAuth 2.0 来检索用户信息。

然而，非常重要的是要认识到：成功调用 UserInfo endpoint 并不等同于验证 ID token，仅凭它本身不能用于实现登录，它不算作登录验证。调用 UserInfo endpoint 只证明相应的 access token 是有效的，并且与返回属性的用户身份相关联：它不能证明 access token 是为那个特定客户端颁发的。OpenID Connect 登录操作始终要求验证 ID token，尽管正如我们在某些情况下看到的，签名检查可以从验证清单中跳过。

在考虑从 confidential client 使用 UserInfo endpoint 时还需要记住的另一件事是，所有关于使用密钥负担的讨论在这里都适用，因为那是获取 access token 的一部分。

### Hybrid grant

Hybrid grant（混合授权）顾名思义，是将多个流程混合为一个。它结合了登录操作（从 front channel 获取 ID token）和从客户端后端获取调用 API 的 access token（通过请求和兑换 authorization code）。这节省了网络往返次数，合并了提示和同意请求，总体来说是一种非常高效的方式，可以在执行登录操作的同时准备好调用 API。没有为 Hybrid grant 展示单独的流程图，因为你可以通过组合上一章的 Web 登录流程图和本章展示的 Authorization Code 流程轻松自行拼凑。OpenID Connect 在这种混合搭配登录和调用 API、让实体同时扮演两种角色方面的能力是独一无二的：一个”资源”（即作为登录访问的一部分被访问的东西），以及一个客户端（消费 API 等其他资源）。OpenID Connect 中的应用始终被称为”客户端”（client），强调后者角色而省略前者，这是对其

OAuth 2.0 起源的致意（也因为 OAuth 2.0 中”resource”一词保留给 API）。

Hybrid grant 是一个非常强大的工具，在应用中被广泛使用。实际上，如今一个应用永远只需要登录或只需要调用 API 的情况相当罕见。通常是一个连续谱，而这种 grant 的可用性使得通过简单修改 Implicit plus Form Post grant 或 Authorization Code grant 就能轻松添加其中一种功能。 

### Client Credentials grant

在专注于调用 API 的本章最后一节，我们将学习 Client Credentials grant（客户端凭据授权），这是 OAuth 2.0 为客户端需要使用自身程序化身份而非代表用户获取 access token 的场景定义的流程。与我们目前考察的所有 grant 不同，Client Credentials grant 没有公共客户端变体——它只能由 confidential client 执行。

目前为 API 考察的所有流程都旨在授予客户端对资源的委托访问，即让客户端在访问资源时”借用”用户的一些特权。

在许多情况下，客户端需要以自身身份而非代表用户进行操作。这些是应用拥有自身身份和直接资源特权的场景。这类场景不要求用户登录或在场。即使用户碰巧在访问时已登录，其特权可能也不是客户端需要行使的。这种场景的经典例子是应用需要执行当前登录用户没有权限的操作。例如，一个持续集成（CI）Web 应用，其构建过程的最后一步是将编译产品的二进制文件保存到没有任何用户有权访问的特定共享目录中。

解决该问题的一种方式是打开闸门，给每个用户访问该共享目录的权限。这样可以保持 CI 以委托访问模式调用共享目录的能力。然而，滥用的风险将非常高：用户可能选择在 CI 流程之外行使他们对文件共享目录的特权。

另一种方式是将文件共享目录访问的特权授予应用本身。进而，应用可以实现逻辑来决定哪些用户应该能够写入共享目录。因此，它可以使用自身的写入特权，仅对适当的用户会话执行写入操作，且仅在 CI 逻辑要求的范围内执行。换言之，通过将访问资源的必要特权授予应用本身，决定谁可以做什么的责任从 authorization server 转移到应用本身，后者成为资源的守门人。

上述模式的一种常见说法是：应用及其访问的下游 API 被定义为一个受信子系统（trusted subsystem）。

用一个现实世界的类比，考虑经典游乐园如何处理游客访问。在入口处，游客支付门票并获得手环或等效的可见标志，表明该个人已付费获得访问权。这个标志不需要带任何佩戴者身份的指示。一旦游客进入，他们可以享受每个游乐设施，除了手环之外不需要任何进一步的访问控制检查，手环广播了他们在场内的权利。

类似地，一旦用户登录到 CI Web 应用，所有对下游 API 的后续调用都将以 Web 应用自身的身份执行，仅凭用户成功登录这一事实。在某种程度上，你可以把这看作边界概念的复兴。然而，与传统网络边界的重大区别在于，这里的边界主要是逻辑上的（API 接受颁发给 CI 应用客户端的 token 的意愿）而非物理上的（实际的网络边界）。

这类模式在微服务上下文中非常常见，其中网关验证调用者的身份。一旦该检查成功完成，从网关发出的所有后续调用都可以携带标识调用应用而非用户的 token。用户信息可能仍然需要，但它不严格需要通过颁发的 token 传递。

与每个 confidential client 流程一样，这里的关键点是特别注意客户端凭据的配置和维护，例如确保除应用本身外没有其他实体能够访问其凭据。此场景的另一个关键方面（虽然标准未明确覆盖但至关重要）是仔细选择分配给应用的特权以及行使这些特权的应用逻辑。最小特权原则（least privilege principle）在此场景中仍是关键最佳实践。

#### 1. Access Token 请求

客户端应用通过直接联系 token endpoint 来请求 token，与我们目前学习的所有 grant 的服务端部分所观察到的类似。

在我们一直讨论的示例场景中，调用是在用户会话期间执行的——然而这完全是任意的。请记住，Client Credentials grant 仅依赖客户端自身的身份，而非请求用户的委托授权。因此，从 OAuth 2.0 的角度看，这里描述的流程同样可能发生在命令行工具、长时间运行的进程，或通常任何可以在其中分发和保护客户端凭据的上下文中执行的应用中。

请求是一个标准的 HTTP POST，携带众所周知的 `client_id`、`client_secret` 和 `grant_type`（这次设置为 `client_credentials`）。

观察 POST 消息体，与目前遇到的所有 grant 的一个显著区别是：发往 token endpoint 的消息除了 `client_secret` 之外不包含任何其他产物。相比之下，Authorization Code grant 和 Refresh Token grant 都包含了其他需要兑换的实体。这再次说明了为什么其他流程在公共客户端下也是可行的，而 Client Credentials grant 没有客户端凭据就不可能实现。

这里有必要强调：客户端凭据（client credentials）和 Client Credentials grant 是两个独立且不同的概念。Client ID 和 client secret 是分配给 confidential client 应用的客户端凭据，用于在每次与 token endpoint 通信时标识客户端应用。Client Credentials grant 是一种碰巧只需要客户端凭据而不需要任何其他产物即可执行的 grant。在使用这些术语时容易产生混淆：每当听到有人提到”客户端凭据”（client credentials）时，有必要弄清楚他们说的是 grant 还是仅仅指 client ID 和 client secret。

对请求消息的最后一点观察：`audience` 参数必须向 authorization server 指明客户端请求访问的资源。对于可以保护多个 resource server 的 authorization server 来说，这一信息是必要的；因此不存在 authorization server 可以引用的默认资源。正如我们之前讨论 `audience` 参数时提到的，向 authorization server 传递该信息的标准方式是通过 resource indicators 规范中定义的 `resource` 参数。在撰写本文时，Auth0 不支持 resource indicators。 

#### 2. Token 响应

Token endpoint 的响应完全在意料之中。它返回所请求的 access token，与其他 grant 的描述相同。当然，没有 `id_token`，因为该 grant 不涉及任何用户身份。

值得注意的是，refresh token 也不存在。在这种场景下，它根本没有用。Refresh token 的目的是允许客户端应用获取新的 access token 来替换过期的，而无需用额外提示打扰用户。然而这里不需要向用户索取任何东西，因为客户端凭据随时可供客户端应用请求新 token。

#### 重要提示

不应滥用此机制。一旦客户端请求并获得了 access token，应在其有效期内保留它（以任务所需的所有安全措施存储），并在需要调用 API 时使用它。丢弃仍然有效的 access token 并每次都从 authorization server 请求新的 access token，在所有层面都是一种代价高昂的反模式：

- 安全性（每次在网络上发送凭据都有出错的机会）。

- 性能（网络调用）。

可用性（可能被限流、瞬时网络故障）。

- 费用（各种提供商按颁发的 token 计费）。

注意，在这种特定情况下，Auth0 使用 scope 来表示客户端能做什么。根据我们前面关于 scope 的讨论，这有些争议。可以说 scope 通常约束的是客户端可以从用户特权中使用的特权集合，而这里没有用户。即使看起来不太合适，Auth0 目前就是这么做的。它只是表示已授予客户端应用的特权。这不存在真正的安全风险：如果 resource owner 将传入的 scope 解读为我们前面讨论的委托授权概念，它们赋予调用者的权力会更少而非更多。然而，这是一个值得注意的例外。 

#### 3. 调用 API

正如预期的那样，对 API 的调用照常进行，不依赖于客户端如何获取用于保护该调用的 access token。

这完成了我们理解如何利用 OAuth 2.0 和 OpenID Connect 从传统 Web 应用（以及通常任何 confidential client）调用 API 的旅程。

在下一章中，我们将看看原生客户端：移动客户端以及几乎任何最终用户可以直接操作的应用……但不是浏览器。 

## 第 5 章 桌面和移动应用

终于到了我最喜欢的话题之一：如何保护运行在桌面或移动设备上的应用程序。

### Public Client（公开客户端）

不过在此之前，我需要介绍我们场景中的又一个角色：我想花些时间来描述一下什么是 public client。

我们已经知道，confidential client（机密客户端）是指拥有凭证并能使用这些凭证向授权服务器证明自身身份（与用户身份无关）的客户端。你猜对了：public client 做不到这一点。通常是因为将凭证分发给 public client 本身就很困难，而且要保密更是难上加难。

举个例子，想象一下你从应用商店在移动设备上安装应用程序的情形。你下载的是这个应用程序的二进制文件，它们将驻留在你的设备上。在应用程序分发过程中，并没有一种协议能同时给你一个代表该应用特定实例的密钥。

即便我们能获得这样的密钥，它也会成为特定应用实例的专属秘密。如果像我们在服务器上使用的 client ID 那样，用它来标识客户端，那么攻击面将直接延伸到潜在攻击者的口袋里。

正如我们之前所说，如果你把凭证分配给一个网站，攻击者需要攻破服务器才能窃取该凭证。但在这里，设备就在我口袋里。它由我支配，有时我还会借给他人使用。我可以安装多个应用而无需进行严格的技术审查。换言之，我的设备可能无意中暴露在恶意攻击之下，风险比服务器更高。因此，一个代表我特定应用实例的密钥，其价值远不如与 client ID 关联的 client secret，在这种场景下使用密钥毫无意义。

有趣的是，我们可能并不太在意这个限制，主要是因为当你使用这类应用时，最重要的因素是用户。所以，如果我在手机上使用 Slack，另一个同事也在手机上使用 Slack，最终授权决策基于的是 Slack 这个身份。当然，Slack 可能需要一组已被授权的 scope。但最关键的因素始终是用户及其权限。

最理想的情况是有某种机制能防止人们拿走 token 并在其他设备上使用。在没有这种机制的情况下，我们可以考虑到没有秘密凭证这一事实，相应地调整授权决策。

这里有一个极其重要的点：如果 client ID 偶尔看起来很晦涩，即难以人类阅读，那并不意味着它是一个秘密。它根本不是秘密。client ID 是公开的。你必须假设它是公开的。事实上，分发到此类客户端的每个标识符或凭证都是公开的。

因此，当你的原生应用是一个 public client 时，你必须假设任何人都能获取那个 client ID 并伪装成你的应用。这是设计使然，也是意料之中的。所以，你永远不应该在服务器端基于原生客户端的 ID 做出授权决策，因为那只是一个提示，根本不能证明任何事情。这一点至关重要！ 

### 原生应用与浏览器

那么我们该怎么做呢？我们已经看到，当我们使用授权服务器、OpenID Connect 提供者及类似服务时，典型的交互方式是通过某种网页，这非常方便，因为我们可以随时更改 UI，也可以注入多种身份验证因素。我们不需要客户端缓存任何东西，也不需要客户端有专门的代码来执行提示等操作。

但在这里，我们有的是代码运行在设备上的原生客户端。那么，我们如何与授权服务器交互呢？诀窍是在需要身份验证时打开一个浏览器小窗口。所以，即使我是一个原生客户端，我也总能提供某种能渲染 HTML 的界面，用这个界面来驱动与授权 endpoint 的所有交互。一旦完成并收到我需要的产物——token、claim 等——然后我就可以从我的代码接管，关闭用作浏览器的窗口，继续我的流程。

有不同的实现方式。最初应用的传统做法——现在已不再推荐，一些授权服务器也不再支持——是使用嵌入式浏览器或嵌入式 WebView。嵌入式 WebView 是一个原生组件，比如操作系统或窗口管理系统的组件，你可以把它放在应用的界面上，就像按钮、标签一样，由它来渲染 HTML。

这样做有风险。一个特别的风险是，应用程序可以控制其界面上发生的一切。所以如果它弹出一个嵌入在应用内的浏览器窗口，用户输入凭证时，应用可以记录每一次按键，这显然很危险。假设你正在使用一个需要通过 Facebook 进行用户认证的应用。以我为例，我是《纽约时报》的订阅者，并将订阅与我的 Facebook 账号关联。如果 Facebook 的登录页面嵌入在 WebView 中，该应用就能截获我的凭证，并在其他场景中冒充我的身份。

另一个问题是，这个嵌入式 WebView 在设计上与设备上的浏览器是隔离的。因此，用户体验会受到一些不便。

考虑前面提到的《纽约时报》阅读应用：每当由于某种原因我未登录时，就会出现一个小窗口，提示”通过 Facebook 登录”。

当这种情况发生在嵌入式 WebView 中时，不管我是否已在 Facebook 应用中登录，我都必须重新登录。我会被要求输入用户名和密码，因为 WebView 与设备的 cookie 存储是隔离的。这极其烦人，以至于很多时候我直接关掉它然后放弃操作，因为输入这些东西太麻烦了。

现在，当今的移动操作系统供应商提供了解决此问题的方案：一种从应用中通过编程方式调用系统浏览器的方法。因此，当移动应用（如 iOS 或 Android 应用）需要从授权服务器获取 token 时，你可以使用系统调用来打开系统浏览器。应用将焦点切换到系统浏览器的一个切片：iOS 上的 Safari view controller 或 Android 上的 Custom Chrome Tab。

这是系统浏览器的一个单标签页视图，可以访问所有数据（包括 cookie），最重要的是，它不在应用的内存空间中。它是浏览器。此时，用户可以输入凭证、进行 MFA，利用已有的 cookie 存储，而不会向调用应用泄露任何凭证。这非常强大且便捷。

问题在于，这增加了一些额外的攻击面，因为当你使用系统浏览器（而非嵌入式浏览器）时，浏览器和你的应用之间的通信不再发生在应用的内存空间中。

所以，假设你正在进行 Authorization Code flow。我使用浏览器获取 authorization code，然后将 code 传递给应用。我需要将 code 从浏览器传递到应用。这意味着如果有人在中间——比如另一个应用——他们可能会截获这个 code。鉴于应用没有凭证——记住这是一个 public client——截获 code 的人可能会用它来获取 token，取而代之。 

### 认识 PKCE

为了防止有人在 authorization code 从浏览器传递到应用的过程中截获它，你可以使用一种机制，实质上将 code 的请求与应用动态创建的一个秘密绑定在一起。

应用必须在兑换 code 时证明它知道这个秘密。因此，如果有人在传输过程中窃取了 code，在不知道这个秘密的情况下将无法使用它。下面我会详细说明这意味着什么。

所以，当你使用系统浏览器时，不应该只使用 Authorization Code flow，还应该添加这个机制来保护 code 的通信安全。这个机制叫做 Proof Key of Code Exchange（PKCE，发音为”pixie”），被定义为 Authorization Code flow 的扩展。 

### 桌面应用与浏览器

现在，这里是另一个有争议的点。关于在移动应用中使用 OAuth 的最佳实践文档基本上就是我刚才说的：你应该使用系统浏览器，并用 PKCE 保护应用与系统浏览器之间的通信。

该文档还告诉你，桌面应用——即在 Windows、Mac 或 Linux 机器上运行的应用——也应该这样做。坦白说，这并不实际。也就是说，如果你尝试对桌面应用做同样的事情，在调用系统浏览器时可能会遇到一些问题。例如，你不知道机器上安装了什么浏览器。而且，这个浏览器可能不会出现在最前面，因为你无法控制浏览器窗口的 Z 轴顺序，或者用户可能打开了多个浏览器窗口，或者只有一个窗口但应用运行在模态窗口中。

最重要的是，如果你真的想要跨多个操作系统兼容，为了将通信返回给你的应用，你需要一个在本地机器上运行的小型 Web 服务器。这个 Web 服务器监听你的 redirect URI，接收 authorization code，然后将其传回应用。

换言之，如果你的操作系统没有像 iOS 和 Android 那样的机制来直接将系统浏览器纳入事务中，这会让终端用户的体验非常糟糕。这也使流程变得复杂，并让安全人员紧张，因为在你的机器上打开 socket 并不是什么好事。

此外，当你使用的设备根本没有浏览器时，你无法使用这个流程，因为它的全部前提就是浏览器的可用性——无论是嵌入式还是系统浏览器。

所以，如果你在开发命令行应用，只有在目标机器有浏览器的情况下才能使用这个流程。另一种 grant——Device Authorization grant——允许你在不同设备上使用浏览器并完成整个循环。但我在这里不详述。 

### Authorization Code Grant 配合 PKCE

让我们按照图 5.2 所示的流程图来看看这些事情是如何发生的。在这个场景中，我有一个想要调用的 API。还有我常用的授权服务器，包含它那老牌的 authorization endpoint、token endpoint 和 discovery endpoint。

在客户端这边，有很多新东西。我们有原生应用，配备了实现 OAuth 的常用 SDK 和保存 token 的缓存。系统浏览器是在同一设备上运行的另一个应用。现在，让我们按照编号步骤，了解我们的应用获取用于调用 API 的 token 的完整流程。 

#### 1. Authorization Request

在原生客户端的情况下，我不能先访问资源然后被重定向到授权 endpoint，因为我的应用自己渲染 UI。我不依赖服务器将 authorization request 重定向到应用。因此，我需要先完成获取 token 的所有必要步骤，然后再调用服务，这就是为什么这个流程的图不是从指向资源的线开始的，而是从指向授权服务器的线开始的。

应用使用操作系统 API 来调用系统浏览器，让它与授权 endpoint 通信。以下是发送到授权服务器的请求： 

```txt
(1) GET https://flosser.auth0.com/authorize??
?audience=https://flosser.com/api/
&response_type=code
&scope=openid profile offline_access read:appointments
&redirect_uri= id102://flosser.auth0.com/cb
&code_challenge=KuijWZHBwJ0_ieZqeYmyyHrkcn-d
&code_challenge_method=S256 
```


图 5.1


实际上，我们正在做 Authorization Code grant，所以你不应该对请求的内容感到惊讶，至少大部分应该很熟悉。

第一个参数是 audience。我们已经了解了 audience 的含义，即我们想要访问的特定资源。我们看到这是 Auth0 特有的，虽然 OAuth 2.0 存在一个等效扩展，但目前尚不支持。

我们有 response_type 参数，值为 code。我们没有指定 response mode，所以我们知道会在查询字符串上收到它。

然后是我们之前调用同一个 API 时使用的 scope 列表：openid、profile、read:appointments。我们还请求了 offline_access。

对于原生客户端来说，refresh token 在某种程度上代表了你的会话，因为它是你控制的主要产物，赋予你获取新 token 的能力。所以，当你从原生客户端应用注销时，你也要处理掉 refresh token。

redirect_uri 带来了第一个新东西：不再使用 HTTPS，而是使用 id102。这个 id102 字符串只是我们在操作系统上配置此应用时创建的协议句柄。这个协议句柄代表我们的应用。它告诉操作系统，每当看到有人尝试访问以 id102 开头的链接时，应该激活我们的应用。

这是一种确保我们收到 authorization code 后，它传递给我们的应用而不是留在浏览器中的方式。

最后，我们有 code_challenge 参数。我之前在介绍 PKCE 机制时提到了 code challenge。应用将此 code 提供给授权服务器，授权服务器会将 authorization code 与这个 challenge 绑定。我们很快就会看到这是如何配合的。

code_challenge_method 只是用于生成 challenge 的算法的实现细节。

#### 2. Authorization Response

现在你将经历所有常见的来回步骤来完成认证过程，包括同意步骤和可能涉及的 MFA。从协议角度来看，重要的是我们收到了常规响应，其中你可以看到在我们指定的 location 上的 302 HTTP 状态码。

这是我们的 redirect URI，我们收到了一个 authorization code，正是我们要求的。我们还有成功认证后的常规 set-cookie。

#### 3. 重定向到应用

接下来是最关键的部分，当浏览器执行 302 重定向时。由于协议句柄是 id102，这实际上是设备内部的通信。浏览器将控制权交还给应用并传递 code。在这一步中，即使有人在中间截获了 code，也无关紧要，因为他们无法使用它。我稍后会告诉你原因。

#### 4. 兑换 Authorization Code

现在我们的应用中已经有了 authorization code，我们可以转向 token endpoint 了。这是一个经典的兑换流程，唯一的注意事项是我们不提供 secret。记住 public client 没有 secret。 

```txt

&client_id=IrbblpeZJRMewq8suTx0PcmHlporj6yZ
&redirect_uri=id102%3A%2F%2Fflosser.auth0.com%2Fcb
&grant_type=authorization_code
&code_verifier=NmFCZzFkSHI1a0R0M3NnSkVuY1FZNzRBOEE1REJ2VE0
&code=35b0NnPFeolksUQp 
```

这是向 token endpoint 发送的常规 POST 请求。我们有 audience 参数和 client_id。我们没有 secret，但有 redirect_uri 参数，再次出于安全目的指定。如你所见，它仍然是带有 id102 协议句柄的那个。

grant_type 是 authorization_code，我们将 authorization code 作为 URL 中的最后一个参数提供。

这里的一个新元素是 code_verifier，它向授权服务器证明我们的应用仍然是 authorization code 的同一个请求者。任何在 token 从系统浏览器传递到应用时窃取 token 的人都无法生成这个 code。这非常方便。

#### 5. 获取 Token

作为响应，我们收到了常规的 access token、refresh token、ID token、已授权的 scope 列表、expires_in 值（因为我们无法查看 access token 内部）以及 token_type：所有常规内容。非常简单直接。

#### 6. 调用 API

现在，我们的应用拥有了使其正常工作所需的 token。它拥有包含用户 claim 的 ID token，以及用于调用 API 的 access token，调用方式与我们在第 4 章中学到的相同。 

### Refresh Token 的问题

如果我们的原生应用收到了 refresh token，它的使用方式与 confidential client 相同。然而，与 confidential client 不同的是，我们的应用不提供 secret，因为它根本没有。当然，这是一个问题，因为来自 public client 的 refresh token 可以直接使用。它们就像有魔力一样，会不断生成 token，不需要做任何额外的事情。

在我深入讨论之前，让我先展示一下 Refresh Token grant 在这种情况下是如何工作的。假设我想获取一个新的 access token。我向 token endpoint 发送以下消息： 

```txt
(1) POST https://flosser.auth0.com/oauth/token HTTP/1.1
client_id=xHMI55zgwY0PnaztfSQflbFAwxxHUM8_
&refresh_token=zisHo7hCAqOQzGDST1GmF_E10EV40VxS5GYOzIBh_uZrA
&grant_type=refresh_token
&redirect_uri=id102://flosser.auth0.com/cb 
```


图 5.4


这里没有什么值得注意的。与我们之前看到的完全一样，区别只在于我们没有 secret。所以这里没有 code，没有 PKCE。只是兑换 refresh token 但没有 secret。

这显然是一个问题，事实上，很多人对此非常担忧，尽管这是主流做法。所有人都在这样做。因此，作为一个行业，我们正在寻找其他解决方案，这些方案不一定需要在原生设备上创建 confidential client。更多的是关于找到将 token 绑定到接收它们时使用的通道的方法。 

#### Token Binding

业界已经有一些努力。一个叫做 Token Binding，它是一组规范，用于扩展 HTTPS 协议栈和浏览器暴露 HTTP 协议栈属性的能力，这些属性可以嵌入到 token 中。

当你使用 token 时，授权服务器和资源服务器可以验证 token 是否在请求它们的同一通道中被使用。如果不是，那基本上意味着有人窃取了该 token 并试图从其他地方使用它。所以你可以通过拒绝服务请求来防止这种情况发生。

这是一个好主意，但它需要许多条件同时满足。而这些条件并没有对齐：Apple 没有宣布支持它；Chrome 曾经支持它，但后来 Google 宣布将停止支持 Token Binding。最终，该规范被退役了。 

#### Mutual TLS 认证

另一个规范是 Token Binding flow 的替代方案：Mutual TLS Client Authentication（双向 TLS 客户端认证）。这个规范的巨大优势在于使用了浏览器和操作系统已有的能力，如客户端证书。

授权服务器可以要求应用使用客户端证书进行认证并获取 token。这种认证发生在网络（TLS）层。然后，在使用获取的 token 时也可以要求同样的证书。因此，如果你拿走其中一个 token 并试图在没有该证书的设备上使用它，你将无法成功。 

#### 应用层所有权证明（DPoP）

鉴于 Token Binding 没有成为一种普遍可用的机制，OAuth 2.0 Demonstrating Proof of Possession（DPoP），即现在的 RFC 9449，被引入。考虑到应用层的约束，该规范允许能够生成不可提取的非对称密钥的客户端来证明其所有权，这反过来允许授权服务器将 token 绑定到它们。与 Mutual TLS 类似，用 DPoP 绑定的 token 除非同时拥有它们所绑定的密钥的访问权限，否则无法使用。这种机制不如 Mutual TLS 强大，但不会带来部署困难和浏览器 UX 障碍——后者源于 TLS 证书系统弹窗，可能会困扰 Mutual TLS 的部署。 

#### 最后一点说明

关于原生客户端环境下的 refresh token，我想提到的最后一件事是再次强调我之前说过的话。refresh token 是追踪你获取 token 能力的产物。它们帮助你为用户提供流畅的、无中断的会话体验，你通常需要遵循所有会话管理步骤来确保安全。例如，当你想要终止会话时，你通常也需要从缓存中删除 refresh token。 

### Resource Owner Grant（资源所有者密码授权）

让我们来谈谈在创建原生应用时你可能遇到的另一个有争议的 grant：Resource Owner Password grant（资源所有者密码授权）。这基本上就是你能想到的：一种允许你获取用户的用户名和密码，然后通过编程方式将它们发送到授权服务器以获取 token 的 grant。简单粗暴但有效。 

#### 糟糕之处

在委托授权的上下文中，直接使用凭证是危险的。它不会给你提供我们迄今为止所见的所有机制的表达能力。总的来说，它只是鼓励用户做错误的事情。它训练用户在非凭证所属的界面中输入凭证。通常，人们想这样做是因为他们想拥有自己的 UI 而不是网页。

但总的来说，每当你处理原始凭证时，特别是在原生应用领域，我们使用外部浏览器而不是嵌入式浏览器。每次使用原始凭证，你都会将自己置于潜在的危险中。事实上，Resource Owner Password grant 在即将发布的 OAuth 2.1 规范中已被弃用。

除了安全方面，以下是直接使用用户名和密码的部分缺点列表： 

**无法提示用户同意**

因此，任何需要用户同意才能访问的资源都无法使用，除非你事先采取步骤注册同意。这在机制和观感上都很糟糕：用户不知道发生了什么，也不知道那些凭证将如何被使用。 

**无法进行多因素认证**

除非你在客户端应用中嵌入相应的功能。这就是多年前在引入浏览器进行认证之前的情况。我向你保证，那一点都不好玩。每当你想要更改哪怕是最微小的东西，你都必须将代码重新分发给所有客户端，其中一些可能只是偶尔连接。所以会有用户在几年后首次打开应用时发现它已经不能用了。真的很糟糕。 

**无法进行逐步认证（step-up authentication）**

如果你有不同的资源需要不同级别的认证，你无法真正做到。你只能在登录时发送用户名和密码。 

**无法使用多个身份提供者**

想想在认证过程中，你被提示一个身份提供者列表，可以从中选择。也许有一个"使用 Facebook 登录"按钮和一个"使用 Google 登录"按钮，或者一个用于输入企业邮箱的字段，它将重定向到你的企业身份提供者。如果你使用 Resource Owner Password grant，你的应用无法做到这一点。即使有办法扩展或收缩这个列表，你也无法连接到那些不允许你以编程方式发送凭证的提供者——大多数正规的身份提供者都是如此。 

· 最后，没有单点登录（single sign-on）。如果我在某处有一个表示我已经通过 Facebook 认证的 cookie，然后我点击一个"使用 Facebook 登录"的小按钮，我就只会被来回跳转。我将利用我已经有会话跟踪 cookie 这一事实，无需输入任何凭证。

如你所知，在 OAuth 和 OIDC 上下文中，使用原生输入框来获取用户名和密码是非常糟糕的。然而，作为身份领域的专业人士，你将面临许多支持 Resource Owner Password flow 而反对其他基于浏览器的 flow 的论点。

你必须预料到外行会经常且强烈地要求使用 Password flow，主要是因为它简单。他们可能会被这上百万个部件、弹出的浏览器、重定向等搞得不知所措。简单性是这个 flow 诱人的方面。

此外，你经常会听到关于用户体验控制权的担忧。幸运的是，在 Auth0 中，我们没有这个问题，因为开发者可以控制登录页面的用户体验。

人们可能会因为重定向而对性能有顾虑。他们可能会想到重定向然后说，"哇，这对性能是个打击"。通常，一个好主意是实际测试并向人们展示情况并非如此。这通常很简单直接。

但至少有一个场景，在我的职业生涯中，我从未找到避免这个 flow 的方法：纯遗留系统场景。

想象一下，你有一个已经在收集用户名和密码的应用，而你无法修改其代码。假设修改认证部分的代码非常困难。那个代码库可能很旧了；也许拥有代码的人已经不在公司了。

或者想想你有一个带有连接字符串的脚本的场景。如果连接字符串只有用户名和密码，而你应该原样使用这个脚本，那么你需要弥补一些差距。

所以，对于纯遗留系统场景以及有计划逐步脱离纯用户名密码认证的情况，如果人们请我审查那种场景，我通常不会过多抱怨。但这是唯一的场景。对于所有其他场景，我始终会坚持使用更精细的方案，因为这个 flow 就是一个等待发生的问题。

### 原生应用的其他 Grant

还有一些与原生应用相关的其他 grant，本书不会详细涵盖。不过，它们至少值得提一下。

#### Device Authorization Grant

其中一个 grant 是 Device Authorization grant（设备授权）。它适用于涉及智能电视、媒体控制台或其他 IoT 设备的场景。这些设备显示能力有限且没有浏览器。服务器机房中的服务器是另一个可以使用此 grant 的例子。通常，你有一个运行在没有图形能力的服务器上的 CLI，但你仍然想调用 API。你已经了解到在委托授权场景中，你需要浏览器。那么，如何对运行在没有浏览器的环境中的应用进行认证和授权呢？

为此，Device Authorization flow 使用了一个技巧。它在基于文本的界面上显示一个 code，并指示你拿出另一个有浏览器的设备，导航到指定地址。到达后，在浏览器加载的页面上输入该 code。完成这些后，你将经历认证所需的经典体验：MFA、同意，以及授权服务器认为合适的任何其他步骤。

运行在纯文本设备上的客户端会不断轮询授权服务器。一旦你给予同意，轮询就会成功，应用将收到它需要的 token。这非常直接且极其方便。

#### Token Exchange Grant

我想提到的第二个 flow 是 Token Exchange grant（令牌交换）。此 grant 允许你用一个 token 交换另一个 token。简而言之，考虑一个场景：客户端调用一个 API，而这个 API 需要回头调用另一个 API，同时携带原始调用者的身份。

第一个 API 可以使用传入的 access token 作为 grant 来交换一个用于调用另一个 API 的 access token。

#### Extension Grant

Token Exchange grant 实际上是 OAuth 2.0 规范提供的 extension grant（扩展授权）机制的具体实现。

此机制允许你在现有标准 grant 不适用于特定场景时定义自定义 grant。

extension grant 类型的另一个例子是 OAuth 2.0 的 SAML Profile。此 grant 类似于 Token Exchange grant：应用已经有一个通过遗留场景获取的 SAML token，想要将其转换为 OAuth 2.0 access token。在 Token Exchange grant 的情况下，你仍在同一个 OAuth 上下文中，而 SAML profile grant 则实现了不同授权上下文之间的互操作性。

这些是你可能在原生客户端上下文中遇到的 grant。不过，你也可以将它们用于 confidential client，尤其是 Token Exchange grant。

关于原生客户端——即桌面和移动客户端——的内容就到这里。接下来，我们将探索 Single Page Application（单页应用）的世界。 

## 第 6 章 单页应用

什么是 Single Page Application（SPA，单页应用）？我相信你们大多数人知道我们在说什么，但让我们试着更好地描述这种类型的 Web 应用。

### 单页应用的本质

你使用像 Gmail 或 Outlook Web Access 等信息密集的 Web UI。这些是同时呈现大量信息的网页。用户与这类应用的自然交互只需要更新界面的部分内容。如果你使用经典的 postback 来实现这种交互模型，你会做大量无用功。

想象一个典型的布局：左侧是消息列表，右侧是显示所选消息内容的面板。每当我点击不同的消息，我只需要选中状态移动到我点击的位置，预览面板的内容自动更新。

就这样。

如果我使用经典的基于 postback 的 Web 应用，我会回到服务器重新请求整个页面：消息列表、周围的所有视觉元素和图标。当然，我可以做缓存，也有各种技巧来改善，但那会产生大量流量，性能也会很差。

在大多数情况下，使用 Single Page Application 时，我在开始时从服务器只获取一个页面。这个页面包含基本的视觉元素，同时还有大量 JavaScript，它可以向服务器请求仅需要的数据。JavaScript 获取这些数据并用来反映 UI 中的变化。它以编程方式注入新数据使其显示出来。所以我不必每次都获取整个页面，只需要需要的部分。这非常方便。没有它，我们就不会有今天所享受的现代 Web 体验。 

### 单页应用的安全挑战

从安全角度来看，这是一个有趣的难题。它是运行在浏览器内的网页，受到浏览器内所有经典攻击的威胁。由于它在浏览器内运行，它与设备本身是隔离的。同时，我刚才描述的交互很大程度上依赖于客户端调用 API。

听起来很熟悉？是的。这与我们在原生客户端中看到的几乎一样。现在我们有了一个有趣的挑战：如何保护这个东西？在这种情况下如何处理身份问题？

我们通常将这类应用某种程度上当作原生客户端来对待。它获取 token 并使用它们来调用受保护的 API。然而，我们将看到这种方法带来了在原生客户端场景中不存在的挑战。

API 的部署位置影响很大。如果 Web API 与获取应用单页面的来源在同一个域下，那么你可以考虑使用我们过去保护网站流量的任何方式。所以 cookie 是可行的，但当你需要向域外发起 API 调用时，cookie 就不再可行了，因为它们与域绑定。你的浏览器不能简单地将 cookie 附加到对其他域的 JavaScript 调用上。因此，基于 token 的方式是最通用的，是你在任何情况下都能使用的方式。但同时也是组件最多、因此也最脆弱的方式。我们将看看这在权衡方面意味着什么。 

### 单页应用与隐式授权

在 OAuth 的早期，保护 Single Page Application 的传统方式是通过 Implicit Grant（隐式授权）。现在，你之前在 Web 应用登录的上下文中已经听到过"implicit"这个词了，特别是 Implicit flow with Form Post 这种特定风格。从 OAuth 的角度来看，当你直接从 authorization endpoint 获取 token 而不需要与 token endpoint 交换任何东西时，这个 grant 就是隐式的。这是 implicit 的正式定义。

有趣的是，当人们谈到 implicit 时，最容易想到的、最经典的、文献中的默认含义，正是我现在要讲的与 Single Page Application 相关的场景，即使用 Implicit grant 并在 URI fragment（井号 # 后面的部分）中传递 token。这个场景充满了问题，但其他场景是完全没问题的，比如我们使用 Form Post 并遵循所有必要的 ID token 验证步骤的情况。

用于 Single Page Application 的 Implicit Grant 给整个 Implicit Grant 家族带来了坏名声。事实上，这个坏名声只在 Single Page Application 的上下文中名副其实，而我们之前在前端通道上为 Web 登录所做的一切都是完全没问题的，前提是应用只获取 ID token 并正确验证它。

所以不用担心，每次你使用 Implicit Grant with Form Post 时，准备好解释它并不是那种"Implicit"。

让我更具体地说明我所说的坏名声是什么意思。与任何 Implicit Grant 一样，我们使用网页重定向直接从 authorization endpoint 请求和获取 token。在 Implicit Grant with Form Post 的情况下，token 在页面正文中传递。在 Implicit Grant with Fragment 的情况下，token 作为重定向 URL 中的 fragment 元素返回。

所以，将 token 放在页面正文中的安全风险略低于放在 URL 中。问题在于，虽然身份领域的专家从来都不喜欢这个 flow，但当时它是唯一的选择，因为浏览器那时无法支持跨域 POST 请求。所以几乎所有的 Single Page Application 都使用了 Implicit flow 和 fragment 方式。我们将看到事情很快发生了变化。

这里的复杂性在某种程度上是由我们身处浏览器（即一个开放平台）这一事实所固有的，平台越开放，攻击面越大。出问题的方式多种多样。例如，如果你将 token 保存在本地存储中，在跨站脚本攻击导致你的整个本地存储内容被导出时，你的 token 就泄露了。如果你在 URL 中接收 token，这个 URL 最终会出现在浏览器历史记录中，还会出现在 referral header 中。所以泄露 token 的途径更多。

为了减轻 token 泄露的影响，我们可以缩短其有效期。这意味着需要续期，但在浏览器中使用像 refresh token 这样强大的产物（就像我们对原生客户端所做的那样）也可能是个问题。驻留在本地资源上的原生客户端可以有某种程度的保护，但我们在浏览器中无法承受这一点。因此，除非你进行 token rotation（令牌轮换），否则不建议在浏览器中使用 refresh token。

总而言之，Implicit flow 有许多复杂性，最终导致 OAuth 工作组建议我们放弃它，转而使用其他方案。 

### SPA 与 Authorization Code 配合 PKCE Flow

如我所述，OAuth 工作组发现使用 Implicit Grant 获取 access token 的风险超过了其便利性。此 flow 的替代方案依赖于在 Implicit flow 最初设计时并不普遍可用的功能，例如 Cross-Origin Resource Sharing（CORS，跨源资源共享）。

核心思想是，我们实际上可以将在原生客户端中描述的完全相同的 flow 用于 Single Page Application。在这种情况下，你使用 JavaScript 来实现 Authorization Code flow with PKCE。请看图 6.1，它展示了与原生应用完全相同的流程。

像往常一样，你使用浏览器从 authorization endpoint 获取 authorization code。一旦你有了这个 code，你可以使用 JavaScript 访问 token endpoint，然后你将从该通道获取 access token。该通道不会将 access token 暴露给浏览器历史记录，也不会在 header 中暴露它。因此，比你使用重定向直接在 URI 中获取 token 时必须做的所有那些混乱操作要容易保护得多。

注意，这里我们可以使用经典的 HTTPS 协议，而不是我们为原生应用使用的自定义协议句柄。

这个 flow 也可以使用 refresh token，只要你使用我之前建议的某种保护 refresh token 的机制。

一种是 refresh token rotation（刷新令牌轮换）：当你使用一个 refresh token 时，它就不再有效了。你会获得一个新的 refresh token，从此刻起使用它。这种机制被认为足以在浏览器中保护 refresh token。替代机制是使用标准约束机制如 DPoP，我在原生客户端章节的末尾描述过。

### 带后端的 SPA

我们使用 token 来保护 Single Page Application。这允许我们调用 API，无论 API 部署在哪里——无论是我们的后端还是其他地方。我们可能根本没有后端，直接调用他人暴露的 API 或我们自己部署在不同域上（如 serverless 环境）的 API。然而，拥有自己的后端是相当常见的。例如，如果你通过某种动态页面技术（如 Node.js 或 ASP.NET）来提供 SPA，那么你就有一个后端。

让我们探讨一下如何利用 SPA 拥有后端这一优势。 

#### SPA 与 API 在同一域下

如果你暴露的 API 仅供你的 JavaScript 展示层使用，那么从技术上讲，没有什么能阻止你使用我们用于 Web 登录的相同技术。从浏览器的角度来看，无论你试图在 Web 服务器上访问的是供人查看的页面，还是用于获取数据的 API endpoint，都无所谓。它只是一个发往某个域的 HTTP 请求。如果浏览器中存在该服务器域的 cookie，它就会附加并发送。在后端，如果中间件位于这些路由之前，它也不关心请求是访问页面还是 API endpoint。因此，只要我们处于这种特定场景中，后端就会执行我最初为 Web 登录描述的相同操作。在首次请求 SPA 时也是如此。从那时起，你可以直接使用 cookie 来保护你的 API 调用。在首页触发登录的同一个中间件将强制要求 cookie 的存在，你将使用它进行认证。

API 与 SPA 在同一域下让我可以简化事情。我可能会使用 Implicit flow with Form Post 来获取 ID token，然后将 token 换成 cookie。或者，如果我想做服务端 flow，我可以做 Authorization Code flow 并将 code 兑换为 ID token。我们为 Web 登录描述的所有技术都可以在此上下文中应用。

在这个场景中，我只有两个挑战需要考虑。

考虑到认证 flow 依赖于重定向。假设你的 cookie 在某个时刻过期了，而你正在从 SPA 发起 HTTP 请求。你会收到一个 302 HTTP 状态码，因为从中间件的角度来看，你试图访问此页面但未认证。因此，中间件会把你送到授权 endpoint。但来自 SPA 的 HTTP 请求并不真正知道如何处理 302 状态码。你需要一个错误管理逻辑来拦截这个 302 状态码，并向用户提供某种操作提示，比如"点击此处重新认证"。

我不建议自动重定向用户，因为如果他们正在填写表单的过程中你把他们带走了，你提供的用户体验就很差。你可以显示一个弹窗，但弹窗有争议，因为有时它们会被拦截。所以，一般来说，我的建议是显示一个小的 toast 提示，告诉用户需要重新认证。

另一个挑战是你的基于 JavaScript 的应用迟早会想要访问用户信息。为此，你期望你的 JavaScript 能够去某个地方查找用户的名字、邮箱等。这些都是你通常在获取 ID token 时能拿到的信息。但在这种场景中，ID token 是由你的后端接收的，而你的 SPA 使用 cookie 工作。Cookie 对客户端是有意不透明的，所以你无法从中提取信息。

你需要在你的 API 中添加一个路由，允许 JavaScript 应用查询 token 的内容，并通常获取 JavaScript 层所需的用户信息。 

#### Token-Mediating Backend 模式（令牌中介后端模式）

为我们的 SPA 配备后端使我们能够执行从网页调用 API 的所有 flow。例如，作为一个 confidential client，你的后端可以使用 Authorization Code flow 获取 token。我们能利用这种场景来调用第三方 API（即需要 access token 的 API）吗？答案是肯定的。实现这一目标的一种方式是使用 Token-Mediating Backend 模式。

你有一个想要调用 API 的 JavaScript 应用和你的后端。你的用户经历一个交互式 Web flow，这会引导执行经典的 Authorization Code flow，兑换 code，获取 access token 和 refresh token。然后你可以将 access token 发送回你的 JavaScript 应用。你的 SPA 可以使用该 token 从 JavaScript 调用 API。

与之前的情况一样，我们不依赖于在客户端上发生的任何 OAuth flow 部分。客户端实际上没有任何获取 token 的代码。

这种方法利用了 confidential client（即后端）的安全配置来从授权服务器获取 token。这使你的 SPA 免于实现授权 flow 及其所有潜在问题。然而，这是以服务器端显著增加复杂度为代价的，服务器端除了获取 token 外，还必须负责将 access token 提供给客户端。

请注意，只有 access token 被发送回 SPA。refresh token 保留在后端并与用户的会话关联。当 API 拒绝 access token 时，SPA 联系后端请求新的 access token。然后后端使用与当前用户关联的 refresh token 来请求新的 access token 并将其发送回 SPA。换言之，SPA 永远不必直接处理 refresh token。

通过将获取 token 的工作委托给后端，你减少了 SPA 的攻击面。然而，JS 应用仍然暴露在允许攻击者窃取 access token 并调用远程 API 的攻击之下。在这方面，遵循最佳实践来缓解这些风险是明智的，例如避免在浏览器中本地存储 token。有关此模式及其安全考虑的更多细节，请参阅 OAuth 2.0 规范。 

#### Backend for Frontend（BFF）模式

还有另一种方式可以利用后端的潜力并减轻 SPA 的安全负担：将交互授权服务器和管理 token 的责任完全委托给后端。我们将要探讨的模式叫做 Backend for Frontend（BFF），它基本上是让后端同时充当面向授权服务器和第三方 API 的中介角色。

后端负责作为 confidential client 与授权服务器交互：它将用户重定向到授权 endpoint 进行认证，获取 authorization code，将其兑换为 ID token、access token 和 refresh token，表现为一个正常的 confidential client。

与之前的情况一样，后端使用 cookie 跟踪用户的已认证会话，但与 Token-Mediating Backend 不同的是，它不会将 token 转发给 JavaScript 应用。它将 token 存储在服务器上，并暴露一些端点供 SPA 满足其所有需求。例如，它暴露一个提供用户资料的端点，后端从 ID token 中提取这些信息。此外，后端充当 SPA 和 API 之间的代理。所有从 SPA 发往 API 的调用都经过后端，后端为此暴露一个或多个端点。当后端收到对这些端点的请求时，它检查是否包含会话 cookie，然后在附加 access token 后将请求转发给 API。收到 API 的响应后，后端将其转发给 SPA，就这样。

这种模式为像 SPA 这样的 public client 提供了 confidential client 的安全性。JavaScript 应用永远不接触 token，因此不存在 token 在 SPA 层面被窃取的风险。正如规范所说：”由于 BFF 架构模式的本质，它提供了强大的安全保证。使用 BFF 还确保应用的攻击面不会因使用 OAuth 而增加。唯一可行的攻击模式是劫持用户浏览器中的客户端应用，这是 Web 应用固有的问题。”

使用这种架构提高 SPA 安全性的代价是实现 JavaScript 应用调用和远程 API 之间代理机制的复杂度增加。这会影响调用的性能，因此在考虑采用此模式时必须考虑到这一点。

要了解更多关于 SPA 的威胁、攻击后果、安全考虑和最佳实践，你可以查阅 OAuth 2.0 for Browser-Based Applications 文档。 

## 结论

我们已经到达了本书的结尾，但对于有志于成为身份领域专业人士的人来说，这并不是旅程的终点。本书涵盖的主题只是基于 OAuth 2.0 和 OpenID Connect 的身份基础。

我们没有涵盖 OAuth 框架中的几个主题。有些只是简单提及，如 Device Authorization grant、Token Exchange grant 以及 JWT Profile for Access Tokens；有些概念我们在较高层次上做了描述但没有深入细节，如 DPoP 和 MTLS；还有一些主题我们甚至没有提及，如 Dynamic Client Registration（动态客户端注册）、Token Revocation（令牌撤销）以及许多其他扩展。

此外，最近我们看到对更高安全性的需求不断增长，特别是在金融、保险、医疗和公用事业等领域。在这些领域，安全和隐私至关重要。为此，OAuth 和 OIDC 社区已经制定了一系列扩展，以加强这些协议在关键场景中的使用。这些努力的一部分推动了 FAPI 规范的发展。

然而，除了这里解释的概念之外，我希望你领会了 OAuth 和 OpenID Connect 的核心精神、它们诞生背后的动机、事物之所以如此的原因，以及它们为满足行业身份安全需求而演进的动机。有了这个基础，我希望你能够设计和实现更健壮的应用，并为你的架构场景需求选择最合适的解决方案。
