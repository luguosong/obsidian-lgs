## 1. 引言

本规范是 Internet X.509 公钥基础设施（PKI）系列标准的一部分。

本规范定义了 Internet PKI 中证书和证书撤销列表（CRL）的格式和语义。描述了在 Internet 环境中处理证书路径的过程。最后，附录中提供了本规范定义或引用的所有数据结构的 [[ASN.1]] 模块。

- 第 2 节描述了 Internet PKI 的需求及影响本文档范围的假设。
- 第 3 节提出了架构模型，并描述了其与先前 IETF 和 ISO/IEC/ITU-T 标准的关系，特别是与 IETF PEM 规范和 ISO/IEC/ITU-T X.509 文档的关系。
- 第 4 节定义了 X.509 版本 3 证书的 [[Profile]]，第 5 节定义了 X.509 版本 2 CRL 的 [[Profile]]。这些 [[Profile]] 包括识别可能在 Internet PKI 中有用的 ISO/IEC/ITU-T 和 ANSI 扩展。[[Profile]] 采用 1988 年 [[ASN.1]]（Abstract Syntax Notation One）而非最新 ISO/IEC/ITU-T 标准使用的 1997 年 [[ASN.1]] 语法。
- 第 6 节包含证书路径验证过程。这些过程基于 ISO/IEC/ITU-T 的定义。实现必须产生相同的结果，但不要求使用指定的过程。

[RFC3279]、[RFC4055] 和 [RFC4491] 定义了公钥材料和[[数字签名]]的识别与编码过程。本规范的实现不要求使用任何特定的密码算法。但是，使用上述 RFC 中标识的算法的符合性实现，必须按照这些规范中的描述识别和编码公钥材料和[[数字签名]]。

最后，提供了三个附录以帮助实现者。附录 A 包含本规范中定义或引用的所有 [[ASN.1]] 结构。附录 B 包含本规范中使用的 [[ASN.1]] 表示法中较不熟悉功能的说明。附录 C 包含符合性证书和符合性 CRL 的示例。

> [!note] 与 RFC 3280 的差异
> 本规范废止了 [RFC3280]。主要差异包括：
> - **第 7 节**增强了对国际化名称的支持，规定了编码和比较国际化域名（IDN）、国际化资源标识符（IRI）和 Distinguished Name 的规则，与 [RFC3490]、[RFC3987] 和 [RFC4518] 中确立的比较规则保持一致。
> - **第 4.1.2.4 节和第 4.1.2.6 节**纳入了 [RFC4630] 中规定的继续使用传统文本编码方案的条件。在已建立的 PKI 中使用时，转换为 UTF8String 可能因名称链接失败或名称约束处理不正确而导致拒绝服务。
> - 删除了 RFC 3280 第 4.2.1.4 节中指定 `privateKeyUsagePeriod` 证书扩展但已弃用的内容。此 ISO 标准扩展在 Internet PKI 中既不弃用也不推荐使用。
> - **第 4.2.1.5 节**建议将策略映射扩展标记为关键。RFC 3280 要求策略映射扩展标记为非关键。
> - **第 4.2.1.11 节**要求将策略约束扩展标记为关键。RFC 3280 允许标记为关键或非关键。
> - 添加了 [RFC4325] 中规定的 Authority Information Access (AIA) CRL 扩展作为第 5.2.7 节。
> - 第 5.2 节和第 5.3 节澄清了处理未识别的 CRL 扩展和 CRL 条目扩展的规则。
> - 删除了 RFC 3280 第 5.3.2 节中指定 `holdInstructionCode` CRL 条目扩展的内容。
> - 第 6 节中的路径验证算法不再跟踪证书链中证书策略扩展的关键性。
> - 安全考虑部分涉及了在 CRL 分发点、Authority Information Access 或 Subject Information Access 扩展中使用 https 或类似方案可能产生的循环依赖风险。
> - 附录 A 中的 [[ASN.1]] 模块与 RFC 3280 相比仅将 `ub-emailaddress-length` 从 128 改为 255，以与 PKCS #9 [RFC2985] 对齐。

本文件中的关键词 "MUST"、"MUST NOT"、"REQUIRED"、"SHALL"、"SHALL NOT"、"SHOULD"、"SHOULD NOT"、"RECOMMENDED"、"MAY" 和 "OPTIONAL" 应按照 [RFC2119] 中的描述进行解释。

## 2. 需求与假设

本规范的目标是制定一个 [[Profile]]，以便那些希望使用 X.509 技术的社区能够在 Internet 应用中使用 X.509 证书。这些应用可能包括 WWW、电子邮件、[[用户认证]]和 IPsec。为了消除使用 X.509 证书的一些障碍，本文档定义了一个 [[Profile]]，以促进证书管理系统和应用工具的开发，以及由策略决定的互操作性。

一些社区需要补充或替换此 [[Profile]]，以满足专门应用领域或具有额外授权、保证或操作需求的环境。然而，对于基本应用，定义了常用属性的通用表示，使应用开发者能够获取必要信息，而无需考虑特定证书或证书撤销列表（CRL）的颁发者。

证书用户在依赖特定证书中公钥相关的认证或不可否认服务之前，应审查证书颁发机构（CA）生成的证书策略。为此，本标准不规定具有法律约束力的规则或义务。

随着补充授权和属性管理工具的出现（如属性证书），可能需要限制证书中包含的已认证属性。这些其他管理工具可能提供更合适的方法来传递许多已认证属性。

### 2.1. 通信与拓扑

证书用户将在各种通信拓扑环境中运行，特别是安全电子邮件用户。此 [[Profile]] 支持没有高带宽、实时 IP 连接或高连接可用性的用户。此外，该 [[Profile]] 允许存在防火墙或其他过滤通信。

此 [[Profile]] 不假设部署了 X.500 目录系统 [X.500] 或轻量级目录访问协议（LDAP）目录系统 [RFC4510]。该 [[Profile]] 不禁止使用 X.500 目录或 LDAP 目录；然而，可以使用任何分发证书和证书撤销列表（CRL）的方式。

### 2.2. 可接受性标准

Internet 公钥基础设施（PKI）的目标是满足确定性、自动化识别、认证、访问控制和授权功能的需求。对这些服务的支持决定了证书中包含的属性，以及证书中的辅助控制信息，如策略数据和证书路径约束。

### 2.3. 用户期望

Internet PKI 的用户是使用客户端软件的人员和进程，也是证书中命名的主体。这些使用包括电子邮件的读写者、WWW 浏览器的客户端、WWW 服务器以及路由器中 IPsec 的[[密钥管理]]器。此 [[Profile]] 认识到这些用户所用平台的局限性以及用户自身在复杂程度和注意力方面的局限性。这体现在最小化的用户配置职责（例如，可信 CA 密钥、规则）、证书内明确的平台使用约束、保护用户免受许多恶意行为影响的证书路径约束，以及合理自动化验证功能的应用。

### 2.4. 管理员期望

与用户期望一样，Internet PKI [[Profile]] 的结构旨在支持通常操作 CA 的人员。为管理员提供无限制的选择会增加细微的 CA 管理员错误导致广泛妥协的机会。此外，无限制的选择大大增加了处理和验证 CA 创建的证书的软件的复杂性。

## 3. 方案概述

以下是使用 X.509 的公钥基础设施（PKIX）规范所假设的架构模型的简化视图。

此模型中的组件包括：

- **终端实体（end entity）**：PKI 证书的用户和/或作为证书主体的终端用户系统。
- **CA（Certification Authority）**：证书颁发机构。
- **RA（Registration Authority）**：注册机构，即 CA 委托某些管理功能的可选系统。
- **CRL 颁发者（CRL issuer）**：生成和签署 CRL 的系统。
- **仓库（repository）**：存储证书和 CRL 并作为向终端实体分发这些证书和 CRL 手段的系统或分布式系统集合。

CA 负责指示其颁发证书的撤销状态。撤销状态信息可以使用[[在线证书状态协议]]（OCSP）[RFC2560]、证书撤销列表（CRL）或其他机制提供。通常，当使用 CRL 提供撤销状态信息时，CA 也是 CRL 颁发者。然而，CA 可以将颁发 CRL 的责任委托给不同的实体。

### 3.1. X.509 版本 3 证书

公钥用户需要确信相关的私钥由正确的远程主体（人员或系统）拥有，该主体将使用加密或[[数字签名]]机制。这种信心通过使用公钥证书获得，公钥证书是将公钥值绑定到主体的数据结构。该绑定由可信 CA 对每个证书进行[[数字签名]]来声明。CA 可以基于技术手段（即通过挑战-响应协议的占有证明）、私钥的呈现或主体的声明来做出此断言。证书具有有限的有效期，这在其签名内容中指示。因为证书的签名和时效性可以由证书使用客户端独立检查，所以证书可以通过不可信的通信和服务器系统分发，并可以在证书使用系统的非安全存储中缓存。

ITU-T X.509（前身为 CCITT X.509）或 ISO/IEC 9594-8 作为 X.500 目录建议的一部分于 1988 年首次发布，定义了标准证书格式 [X.509]。1988 年标准中的证书格式称为版本 1（v1）格式。当 X.500 于 1993 年修订时，添加了两个额外字段，产生了版本 2（v2）格式。

Internet 隐私增强邮件（PEM）RFC 于 1993 年发布，包含基于 X.509 v1 证书的公钥基础设施规范 [RFC1422]。在尝试部署 RFC 1422 过程中获得的经验清楚地表明，v1 和 v2 证书格式在几个方面存在缺陷。最重要的是，需要更多字段来承载 PEM 设计和实现经验证明必要的信息。为响应这些新需求，ISO/IEC、ITU-T 和 ANSI X9 开发了 X.509 版本 3（v3）证书格式。v3 格式通过添加额外扩展字段的条款来扩展 v2 格式。特定的扩展字段类型可以在标准中指定，或由任何组织或社区定义和注册。1996 年 6 月，基本 v3 格式的标准化完成 [X.509]。

ISO/IEC、ITU-T 和 ANSI X9 也开发了用于 v3 扩展字段 [X.509][X9.55] 的标准扩展。这些扩展可以传达诸如附加主体识别信息、密钥属性信息、策略信息和证书路径约束等数据。

然而，ISO/IEC、ITU-T 和 ANSI X9 的标准扩展在适用性上非常广泛。为了开发可在 Internet 上互操作的 X.509 v3 系统实现，有必要为 Internet 使用的 X.509 v3 扩展指定一个 [[Profile]]。本文档的目标之一是为 Internet WWW、电子邮件和 IPsec 应用指定一个 [[Profile]]。具有额外需求的环境可以在此 [[Profile]] 基础上构建或替换它。

### 3.2. 证书路径与信任

需要知道公钥的安全服务用户通常需要获取并验证包含所需公钥的证书。如果公钥用户尚未持有签署该证书的 CA 的公钥、CA 名称和相关信息的可信副本（例如有效期或名称约束），则可能需要额外的证书来获取该公钥。通常，可能需要多个证书的链，包括由一个 CA 签名的公钥所有者（终端实体）的证书，以及零个或多个由其他 CA 签名的 CA 证书。这种链称为证书路径，因为公钥用户仅初始化了有限数量的可信 CA 公钥。

CA 可以以不同方式配置，以便公钥用户能够找到证书路径。对于 PEM，RFC 1422 定义了严格的 CA 层次结构。PEM 证书颁发机构有三种类型：

1. **Internet 策略注册机构（IPRA）**：在 Internet 学会主持下运作的机构，作为 PEM 证书层次结构的根，位于级别 1。它仅向下一级别的机构（PCA）颁发证书。所有证书路径都从 IPRA 开始。
2. **策略认证机构（PCA）**：PCA 位于层次结构的级别 2，每个 PCA 由 IPRA 认证。PCA 应制定并发布其关于认证用户或下属证书颁发机构的策略声明。不同的 PCA 旨在满足不同的用户需求。例如，一个 PCA（组织 PCA）可能支持商业组织的一般电子邮件需求，另一个 PCA（高保证 PCA）可能有更严格的策略，旨在满足具有法律约束力的[[数字签名]]要求。
3. **证书颁发机构（CA）**：CA 位于层次结构的级别 3，也可以在更低级别。级别 3 的 CA 由 PCA 认证。CA 代表例如特定组织、特定组织单位（例如部门、小组、科室）或特定地理区域。

RFC 1422 还有名称从属规则，要求 CA 只能为名称从属于 CA 自身名称（在 X.500 命名树中）的实体颁发证书。与 PEM 证书路径相关的信任由 PCA 名称暗示。名称从属规则确保 PCA 下的 CA 在可以认证的从属实体的集合上受到合理的约束（例如，组织的 CA 只能认证该组织名称树中的实体）。证书用户系统能够机械地检查是否遵循了名称从属规则。

RFC 1422 使用 X.509 v1 证书格式。X.509 v1 的局限性需要施加几项结构性限制，以明确关联策略信息或限制证书的效用。这些限制包括：

1. 纯自上而下的层次结构，所有证书路径从 IPRA 开始；
2. 限制 CA 主体名称的名称从属规则；
3. 使用 PCA 概念，要求将各个 PCA 的知识内置到证书链验证逻辑中。

使用 X.509 v3，RFC 1422 解决的大部分需求可以通过证书扩展来解决，而无需限制所使用的 CA 结构。特别是，与证书策略相关的证书扩展消除了对 PCA 的需求，约束扩展消除了对名称从属规则的需求。因此，本文档支持更灵活的架构，包括：

1. 证书路径从用户自己域中 CA 的公钥开始，或从层次结构顶部的公钥开始。从用户自己域中 CA 的公钥开始具有某些优势。在某些环境中，本地域是最可信的。
2. 名称约束可以通过在证书中显式包含名称约束扩展来施加，但不是必需的。
3. 策略扩展和策略映射取代了 PCA 概念，允许更大程度的自动化。应用程序可以根据证书内容而非 PCA 的先验知识来确定证书路径是否可接受。

本规范涵盖两类证书：CA 证书和终端实体证书。CA 证书可以进一步分为三类：交叉证书、自签发证书和自签名证书。

- **交叉证书**是颁发者和主体为不同实体的 CA 证书。交叉证书描述两个 CA 之间的信任关系。
- **自签发证书**是颁发者和主体为同一实体的 CA 证书。自签发证书用于支持策略或操作的变更。
- **自签名证书**是自签发证书，其[[数字签名]]可以由绑定到证书中的公钥验证。自签名证书用于传达公钥，以开始证书路径。
- **终端实体证书**颁发给未被授权颁发证书的主体。

### 3.3. 证书撤销

当证书颁发时，预期在其整个有效期内使用。然而，各种情况可能导致证书在有效期到期之前失效。这些情况包括名称变更、主体与 CA 之间关联的变更（例如员工终止与组织的雇佣关系），以及相应私钥的泄露或疑似泄露。在这些情况下，CA 需要撤销证书。

X.509 定义了一种证书撤销方法。此方法涉及每个 CA 定期发布称为证书撤销列表（CRL）的签名数据结构。CRL 是标识已撤销证书的带时间戳列表，由 CA 或 CRL 颁发者签名，并在公共仓库中免费提供。每个被撤销的证书在 CRL 中通过其证书序列号标识。当证书使用系统使用证书时（例如验证远程用户的[[数字签名]]），该系统不仅检查证书签名和有效性，还获取适当最新的 CRL 并检查该证书序列号不在 CRL 上。"适当最新"的含义可能因本地策略而异，但通常意味着最新发布的 CRL。新的 CRL 定期发布（例如每小时、每天或每周）。条目在收到撤销通知后的下一次更新中被添加到 CRL。条目在被撤销证书有效期到期后出现在一个定期发布的 CRL 上之前，不得从 CRL 中删除。

这种撤销方法的优点是 CRL 可以通过与证书本身完全相同的方式分发，即通过不可信的服务器和不可信的通信。

使用不可信通信和服务器的 CRL 撤销方法的一个局限性是撤销的时间粒度限于 CRL 发布周期。

与 X.509 v3 证书格式一样，为了促进多个供应商的可互操作实现，需要为 Internet 使用制定 X.509 v2 CRL 格式的 [[Profile]]。本文档的目标之一是指定该 [[Profile]]。然而，此 [[Profile]] 不要求发布 CRL。支持在线撤销通知的消息格式和协议在其他 PKIX 规范中定义。在线撤销通知方法在某些环境中可以作为 X.509 CRL 的替代方案。在线撤销检查可以显著减少从撤销报告到信息分发给依赖方之间的延迟。一旦 CA 接受撤销报告为真实和有效的，对在线服务的任何查询将正确反映撤销的证书验证影响。然而，这些方法施加了新的安全要求：证书验证器需要信任在线验证服务，而仓库不需要被信任。

### 3.4. 操作协议

操作协议用于将证书和 CRL（或状态信息）传送到证书使用客户端系统。需要提供多种不同的证书和 CRL 传送方式的规定，包括基于 LDAP、HTTP、FTP 和 X.500 的分发过程。支持这些功能的操作协议在其他 PKIX 规范中定义。

### 3.5. 管理协议

管理协议用于支持 PKI 用户和管理实体之间的在线交互。例如，管理协议可以在 CA 和与密钥对关联的客户端系统之间使用，或在相互交叉认证的两个 CA 之间使用。管理协议可能需要支持的功能集包括：

1. **注册**：用户首先向 CA（直接或通过 RA）表明身份的过程，在该 CA 为该用户颁发证书之前。
2. **初始化**：在客户端系统可以安全运行之前，需要安装与基础设施中其他位置存储的密钥具有适当关系的密钥材料。例如，客户端需要使用可信 CA 的公钥和其他可靠信息进行安全初始化，用于验证证书路径。此外，客户端通常需要使用自己的密钥对进行初始化。
3. **认证**：CA 为用户的公钥颁发证书，并将该证书返回给用户的客户端系统和/或将该证书发布到仓库。
4. **密钥对恢复**：作为选项，用户客户端密钥材料（例如用户用于加密目的的私钥）可以由 CA 或密钥备份系统备份。如果用户需要恢复这些备份的密钥材料（例如由于忘记密码或丢失密钥链文件），可能需要在线协议交换来支持此类恢复。
5. **密钥对更新**：所有密钥对需要定期更新，即用新密钥对替换，并颁发新证书。
6. **撤销请求**：授权人员通知 CA 需要证书撤销的异常情况。
7. **交叉认证**：两个 CA 交换用于建立交叉证书的信息。交叉证书是一个 CA 向另一个 CA 颁发的证书，包含用于颁发证书的 CA 签名密钥。

注意，在线协议不是实现上述功能的唯一方式。对于所有功能，都有离线方法可以达到相同的结果，本规范不强制使用在线协议。例如，当使用硬件令牌时，许多功能可以作为物理令牌传递的一部分来实现。

## 4. 证书与证书扩展 Profile

本节提出了一个公钥证书的 [[Profile]]，以促进互操作性和可重用的 PKI。本节基于 X.509 v3 证书格式和 [X.509] 中定义的标准证书扩展。ISO/IEC 和 ITU-T 文档使用 1997 年版本的 [[ASN.1]]；虽然本文档使用 1988 年 [[ASN.1]] 语法，但编码后的证书和标准扩展是等价的。本节还定义了支持 Internet 社区 PKI 所需的私有扩展。

证书可以在广泛的应用和环境中使用，涵盖广泛的互操作性目标和更广泛的操作与保证需求。本文档的目标是为需要广泛互操作性和有限特殊用途需求的通用应用建立通用基线。特别是，重点是支持在非正式 Internet 电子邮件、IPsec 和 WWW 应用中使用 X.509 v3 证书。

### 4.1. 基本证书字段

X.509 v3 证书的基本语法如下。对于签名计算，要签名的数据使用 [[ASN.1]] 可分辨编码规则（DER）[X.690] 编码。[[ASN.1]] DER 编码是每个元素的标签、长度、值编码系统。

```asn1
Certificate  ::=  SEQUENCE  {
     tbsCertificate       TBSCertificate,
     signatureAlgorithm   AlgorithmIdentifier,
     signatureValue       BIT STRING  }

TBSCertificate  ::=  SEQUENCE  {
     version         [0]  EXPLICIT Version DEFAULT v1,
     serialNumber         CertificateSerialNumber,
     signature            AlgorithmIdentifier,
     issuer               Name,
     validity             Validity,
     subject              Name,
     subjectPublicKeyInfo SubjectPublicKeyInfo,
     issuerUniqueID  [1]  IMPLICIT UniqueIdentifier OPTIONAL,
                          -- If present, version MUST be v2 or v3
     subjectUniqueID [2]  IMPLICIT UniqueIdentifier OPTIONAL,
                          -- If present, version MUST be v2 or v3
     extensions      [3]  EXPLICIT Extensions OPTIONAL
                          -- If present, version MUST be v3
     }

Version  ::=  INTEGER  {  v1(0), v2(1), v3(2)  }

CertificateSerialNumber  ::=  INTEGER

Validity ::= SEQUENCE {
     notBefore      Time,
     notAfter       Time }

Time ::= CHOICE {
     utcTime        UTCTime,
     generalTime    GeneralizedTime }

UniqueIdentifier  ::=  BIT STRING

SubjectPublicKeyInfo  ::=  SEQUENCE  {
     algorithm            AlgorithmIdentifier,
     subjectPublicKey     BIT STRING  }

Extensions  ::=  SEQUENCE SIZE (1..MAX) OF Extension

Extension  ::=  SEQUENCE  {
     extnID      OBJECT IDENTIFIER,
     critical    BOOLEAN DEFAULT FALSE,
     extnValue   OCTET STRING
                 -- contains the DER encoding of an ASN.1 value
                 -- corresponding to the extension type identified
                 -- by extnID
     }

#### 4.1.1. 证书字段

Certificate 是三个必需字段的 SEQUENCE。这些字段在以下小节中详细描述。

##### 4.1.1.1. tbsCertificate

此字段包含主体和颁发者的名称、与主体关联的公钥、有效期以及其他相关信息。这些字段在第 4.1.2 节中详细描述；tbsCertificate 通常包括扩展，在第 4.2 节中描述。

##### 4.1.1.2. signatureAlgorithm

`signatureAlgorithm` 字段包含 CA 用于签署此证书的密码算法的标识符。[RFC3279]、[RFC4055] 和 [RFC4491] 列出了支持的签名算法，但也可以支持其他签名算法。

算法标识符由以下 ASN.1 结构定义：

```asn1
AlgorithmIdentifier  ::=  SEQUENCE  {
     algorithm               OBJECT IDENTIFIER,
     parameters              ANY DEFINED BY algorithm OPTIONAL  }

算法标识符用于标识密码算法。OBJECT IDENTIFIER 组件标识算法（如带 SHA-1 的 DSA）。可选 `parameters` 字段的内容将根据所标识的算法而变化。

此字段必须包含与 `tbsCertificate` 序列中 `signature` 字段（第 4.1.2.3 节）相同的算法标识符。

##### 4.1.1.3. signatureValue

`signatureValue` 字段包含对 [[ASN.1]] DER 编码的 `tbsCertificate` 计算的[[数字签名]]。[[ASN.1]] DER 编码的 `tbsCertificate` 用作签名函数的输入。此签名值编码为 BIT STRING 并包含在签名字段中。[RFC3279]、[RFC4055] 和 [RFC4491] 中指定了每种算法的详细过程。

通过生成此签名，CA 认证 `tbsCertificate` 字段中信息的有效性。特别是，CA 认证公钥材料与证书主体之间的绑定。

#### 4.1.2. TBSCertificate

序列 TBSCertificate 包含与证书主体和颁发它的 CA 相关联的信息。每个 TBSCertificate 包含主体和颁发者的名称、与主体关联的公钥、有效期、版本号和序列号；有些可能包含可选的唯一标识符字段。

##### 4.1.2.1. 版本（Version）

此字段描述编码证书的版本。当使用扩展时（如本 [[Profile]] 所预期），版本必须为 3（值为 2）。如果没有扩展但存在 UniqueIdentifier，版本应为 2（值为 1）；但版本也可以为 3。如果只有基本字段，版本应为 1（默认值从证书中省略）；但版本也可以为 2 或 3。

实现应准备好接受任何版本的证书。符合性实现至少必须能识别版本 3 的证书。基于此 [[Profile]] 的实现不预期生成版本 2 的证书。

##### 4.1.2.2. 序列号（Serial Number）

序列号必须是 CA 分配给每个证书的正整数。对于给定 CA 颁发的每个证书必须唯一（即颁发者名称和序列号标识唯一证书）。CA 必须强制 `serialNumber` 为非负整数。

鉴于上述唯一性要求，序列号可以预期包含长整数。证书用户必须能够处理最多 20 个八位字节的 `serialNumber` 值。符合性 CA 不得使用超过 20 个八位字节的 `serialNumber` 值。

> [!note] 注意
> 不符合性的 CA 可能颁发序列号为负或零的证书。证书用户应准备好优雅地处理此类证书。

##### 4.1.2.3. 签名（Signature）

此字段包含 CA 用于签署证书的算法标识符。

此字段必须包含与 Certificate 序列中 `signatureAlgorithm` 字段（第 4.1.1.2 节）相同的算法标识符。

##### 4.1.2.4. 颁发者（Issuer）

颁发者字段标识签署并颁发证书的实体。颁发者字段必须包含非空的 Distinguished Name（DN）。颁发者字段定义为 X.501 类型 Name [X.501]。Name 由以下 [[ASN.1]] 结构定义：

```asn1
Name ::= CHOICE { -- only one possibility for now --
  rdnSequence  RDNSequence }

RDNSequence ::= SEQUENCE OF RelativeDistinguishedName

RelativeDistinguishedName ::=
  SET SIZE (1..MAX) OF AttributeTypeAndValue

AttributeTypeAndValue ::= SEQUENCE {
  type     AttributeType,
  value    AttributeValue }

AttributeType ::= OBJECT IDENTIFIER

AttributeValue ::= ANY -- DEFINED BY AttributeType

DirectoryString ::= CHOICE {
      teletexString           TeletexString (SIZE (1..MAX)),
      printableString         PrintableString (SIZE (1..MAX)),
      universalString         UniversalString (SIZE (1..MAX)),
      utf8String              UTF8String (SIZE (1..MAX)),
      bmpString               BMPString (SIZE (1..MAX)) }

Name 描述了由属性（如国家名称）和相应值（如 US）组成的层次名称。组件 AttributeValue 的类型由 AttributeType 决定；通常为 DirectoryString。

DirectoryString 类型定义为 PrintableString、TeletexString、BMPString、UTF8String 和 UniversalString 的选择。符合性 CA 必须使用 PrintableString 或 UTF8String 编码，有两个例外：当 CA 之前已颁发包含使用 TeletexString、BMPString 或 UniversalString 编码的颁发者字段的证书时，CA 可以继续使用这些编码以保持向后兼容性；此外，添加到现有 CA 已使用 TeletexString、BMPString 或 UniversalString 编码颁发者字段属性域中的新 CA，可以使用与现有 CA 相同的编码。

符合性实现必须准备好接收包含以下标准属性类型的颁发者名称：country（国家）、organization（组织）、organizational unit（组织单位）、distinguished name qualifier（DN 限定符）、state or province name（州/省名称）、common name（公用名）和 serial number（序列号）。

此外，符合性实现应准备好接收以下标准属性类型：locality（地区）、title（头衔）、surname（姓氏）、given name（名字）、initials（首字母缩写）、pseudonym（假名）和 generation qualifier（世代限定符，如 "Jr."、"3rd" 或 "IV"）。

此外，符合性实现必须准备好接收 [RFC4519] 中定义的 domainComponent 属性。域名系统（DNS）提供分层资源标记系统。此属性为希望使用与其 DNS 名称并行的 DN 的组织提供了便利的机制。这不是替代名称扩展中 dNSName 组件的替代品。实现不要求将此类名称转换为 DNS 名称。

证书用户必须准备好处理颁发者 Distinguished Name 和主体 Distinguished Name（第 4.1.2.6 节）字段，以执行证书路径验证（第 6 节）的名称链接。名称链接通过将一个证书中的颁发者 Distinguished Name 与 CA 证书中的主体名称匹配来执行。Distinguished Name 的比较规则在第 7.1 节中指定。如果证书中的颁发者和主体字段按照第 7.1 节中指定的规则匹配，则证书是自签发的。

##### 4.1.2.5. 有效期（Validity）

证书有效期是 CA 保证将维护证书状态信息的时间间隔。该字段表示为两个日期的 SEQUENCE：证书有效期开始的日期（`notBefore`）和证书有效期结束的日期（`notAfter`）。`notBefore` 和 `notAfter` 都可以编码为 UTCTime 或 GeneralizedTime。

符合性 CA 必须始终将到 2049 年的证书有效期日期编码为 UTCTime；2050 年及以后的证书有效期日期必须编码为 GeneralizedTime。符合性应用必须能够处理以 UTCTime 或 GeneralizedTime 编码的有效期日期。

证书的有效期是从 `notBefore` 到 `notAfter`（含两端）的时间段。

> [!tip] 无到期日期
> 在某些情况下，设备获得的证书无法分配良好的到期日期。例如，设备可以获得一个将其型号和序列号绑定到其公钥的证书；此类证书旨在用于设备的整个生命周期。要指示证书没有明确的到期日期，`notAfter` 应分配 GeneralizedTime 值 `99991231235959Z`。

当颁发者无法维护状态信息直到 `notAfter` 日期时（包括 `notAfter` 日期为 `99991231235959Z` 时），颁发者必须确保证书在状态信息维护终止后不存在有效的证书路径。

###### 4.1.2.5.1. UTCTime

通用时间类型 UTCTime 是标准 ASN.1 类型，用于表示日期和时间。UTCTime 通过两个低位数字指定年份，时间精确到一分钟或一秒。UTCTime 包含 Z（表示 Zulu 或格林威治标准时间）或时间差。

对于此 Profile，UTCTime 值必须以格林威治标准时间（Zulu）表示，并且必须包含秒（即时间为 `YYMMDDHHMMSSZ`），即使秒数为零。

- 当 YY 大于或等于 50 时，年份解释为 19YY；
- 当 YY 小于 50 时，年份解释为 20YY。

###### 4.1.2.5.2. GeneralizedTime

广义时间类型 GeneralizedTime 是标准 ASN.1 类型，用于可变精度的时间表示。可选地，GeneralizedTime 字段可以包含本地时间与格林威治标准时间之间的时间差表示。

对于此 Profile，GeneralizedTime 值必须以格林威治标准时间（Zulu）表示，并且必须包含秒（即时间为 `YYYYMMDDHHMMSSZ`），即使秒数为零。GeneralizedTime 值不得包含小数秒。

##### 4.1.2.6. 主体（Subject）

主体字段标识与主体公钥字段中存储的公钥相关联的实体。主体名称可以在主体字段和/或 `subjectAltName` 扩展中承载。如果主体是 CA（即基本约束扩展存在且 `cA` 为 TRUE），则主体字段必须填充非空的 Distinguished Name，与该主体 CA 颁发的所有证书中颁发者字段的内容匹配。如果主体是 CRL 颁发者（即密钥用法扩展存在且 `cRLSign` 为 TRUE），则主体字段必须填充非空的 Distinguished Name，与该主体 CRL 颁发者颁发的所有 CRL 中颁发者字段的内容匹配。如果证书中唯一的主体身份信息仅以替代名称形式存在（例如仅绑定到电子邮件地址或 URI 的密钥），则主体名称必须为空序列，并且 `subjectAltName` 扩展必须标记为关键。

当非空时，主体字段必须包含 X.500 Distinguished Name（DN）。对于由颁发者字段定义的同一 CA 认证的每个主体实体，DN 必须唯一。CA 可以向同一主体实体颁发具有相同 DN 的多个证书。

当编码 DirectoryString 类型的属性值时，符合性 CA 必须使用 PrintableString 或 UTF8String 编码。

> [!warning] 传统编码
> TeletexString、BMPString 和 UniversalString 是为了向后兼容而保留的，不应用于新主体的证书。但是，当名称之前已建立时（包括向现有主体颁发新证书或向名称属性已建立的新主体颁发证书时），可以使用这些类型。

遗留实现存在将电子邮件地址嵌入在主体 Distinguished Name 中作为 emailAddress 属性 [RFC2985] 的情况。生成包含电子邮件地址的新证书的符合性实现必须使用 `subjectAltName` 扩展中的 `rfc822Name` 来描述此类身份。

##### 4.1.2.7. 主体公钥信息（Subject Public Key Info）

此字段用于承载公钥并标识使用该密钥的算法（如 RSA、DSA 或 Diffie-Hellman）。算法使用第 4.1.1.2 节中指定的 AlgorithmIdentifier 结构标识。支持算法的对象标识符和编码公钥材料（公钥和参数）的方法在 [RFC3279]、[RFC4055] 和 [RFC4491] 中指定。

##### 4.1.2.8. 唯一标识符（Unique Identifiers）

这些字段只有在版本为 2 或 3（第 4.1.2.1 节）时才能出现。如果版本为 1 则不得出现。主体和颁发者唯一标识符存在于证书中，以处理主体和/或颁发者名称随时间重用的可能性。此 Profile 建议名称不要为不同实体重用，且 Internet 证书不使用唯一标识符。符合性 CA 不得生成包含唯一标识符的证书。

##### 4.1.2.9. 扩展（Extensions）

此字段只有在版本为 3（第 4.1.2.1 节）时才能出现。如果存在，此字段是一个或多个证书扩展的 SEQUENCE。Internet PKI 中证书扩展的格式和内容在第 4.2 节中定义。

### 4.2. 证书扩展

为 X.509 v3 证书定义的扩展提供了将附加属性与用户或公钥关联以及管理 CA 之间关系的方法。X.509 v3 证书格式还允许社区定义私有扩展以承载该社区特有的信息。证书中的每个扩展被指定为关键或非关键。证书使用系统如果遇到它不识别的关键扩展或包含它无法处理的信息的关键扩展，必须拒绝该证书。非关键扩展如果不被识别则可以忽略，但如果被识别则必须处理。

每个扩展包括一个 OID 和一个 ASN.1 结构。当扩展出现在证书中时，OID 显示为 `extnID` 字段，相应的 ASN.1 DER 编码结构是八位字节字符串 `extnValue` 的值。证书不得包含特定扩展的多个实例。

符合性 CA 必须支持密钥标识符（第 4.2.1.1 和 4.2.1.2 节）、基本约束（第 4.2.1.9 节）、密钥用法（第 4.2.1.3 节）和证书策略（第 4.2.1.4 节）扩展。如果 CA 颁发主体字段为空序列的证书，CA 必须支持 `subjectAltName` 扩展（第 4.2.1.6 节）。对其余扩展的支持是可选的。

至少，符合性应用必须识别以下扩展：密钥用法（第 4.2.1.3 节）、证书策略（第 4.2.1.4 节）、主体替代名称（第 4.2.1.6 节）、基本约束（第 4.2.1.9 节）、名称约束（第 4.2.1.10 节）、策略约束（第 4.2.1.11 节）、扩展密钥用法（第 4.2.1.12 节）和 inhibit anyPolicy（第 4.2.1.14 节）。

#### 4.2.1. 标准扩展

本节标识了 [X.509] 中定义的用于 Internet PKI 的标准证书扩展。每个扩展与 [X.509] 中定义的 OID 关联。这些 OID 是 `id-ce` 弧的成员：

```asn1
id-ce   OBJECT IDENTIFIER ::=  { joint-iso-ccitt(2) ds(5) 29 }

##### 4.2.1.1. Authority Key Identifier

Authority Key Identifier 扩展提供了标识用于签署证书的私钥对应的公钥的方法。当颁发者有多个签名密钥时使用此扩展（由于多个并发密钥对或由于密钥更换）。标识可以基于密钥标识符（颁发者证书中的主体密钥标识符）或颁发者名称和序列号。

符合性 CA 生成的所有证书中必须包含 `authorityKeyIdentifier` 扩展的 `keyIdentifier` 字段，以便于证书路径构建。有一个例外：当 CA 以"自签名"证书的形式分发其公钥时，Authority Key Identifier 可以省略。

```asn1
id-ce-authorityKeyIdentifier OBJECT IDENTIFIER ::=  { id-ce 35 }

AuthorityKeyIdentifier ::= SEQUENCE {
   keyIdentifier             [0] KeyIdentifier           OPTIONAL,
   authorityCertIssuer       [1] GeneralNames            OPTIONAL,
   authorityCertSerialNumber [2] CertificateSerialNumber OPTIONAL  }

KeyIdentifier ::= OCTET STRING

##### 4.2.1.2. Subject Key Identifier

Subject Key Identifier 扩展提供了标识包含特定公钥的证书的方法。为了便于证书路径构建，此扩展必须出现在所有符合性 CA 证书中。

从公钥生成密钥标识符的两种常用方法：

1. `keyIdentifier` 由 BIT STRING `subjectPublicKey` 值（不包括标签、长度和未使用位数）的 160 位 SHA-1 哈希组成。
2. `keyIdentifier` 由四位类型字段（值为 `0100`）加上 BIT STRING `subjectPublicKey` 值（不包括标签、长度和未使用位数）的 SHA-1 哈希的最低有效 60 位组成。

```asn1
id-ce-subjectKeyIdentifier OBJECT IDENTIFIER ::=  { id-ce 14 }

SubjectKeyIdentifier ::= KeyIdentifier

##### 4.2.1.3. 密钥用法（Key Usage）

密钥用法扩展定义了证书中包含的密钥的用途（例如加密、签名、证书签名）。当可用于多个操作的密钥需要被限制时，可以使用用法限制。

符合性 CA 必须在包含用于验证其他公钥证书或 CRL 上[[数字签名]]的公钥的证书中包含此扩展。当存在时，符合性 CA 应将此扩展标记为关键。

```asn1
id-ce-keyUsage OBJECT IDENTIFIER ::=  { id-ce 15 }

KeyUsage ::= BIT STRING {
     digitalSignature        (0),
     nonRepudiation          (1), -- recent editions of X.509 have
                          -- renamed this bit to contentCommitment
     keyEncipherment         (2),
     dataEncipherment        (3),
     keyAgreement            (4),
     keyCertSign             (5),
     cRLSign                 (6),
     encipherOnly            (7),
     decipherOnly            (8) }

KeyUsage 类型中的位用法如下：

| 位 | 名称 | 用途 |
|---|---|---|
| `digitalSignature` | 数字签名 | 验证数字签名，证书签名（位 5）和 CRL 签名（位 6）除外 |
| `nonRepudiation` | 不可否认 | 验证用于提供不可否认服务的数字签名 |
| `keyEncipherment` | 密钥加密 | 加密私钥或秘密密钥，用于密钥传输 |
| `dataEncipherment` | 数据加密 | 直接加密原始用户数据 |
| `keyAgreement` | 密钥协商 | 用于密钥协商，如 Diffie-Hellman |
| `keyCertSign` | 证书签名 | 验证公钥证书上的签名 |
| `cRLSign` | CRL 签名 | 验证 CRL 上的签名 |
| `encipherOnly` | 仅加密 | 仅在 `keyAgreement` 位设置时有意义 |
| `decipherOnly` | 仅解密 | 仅在 `keyAgreement` 位设置时有意义 |

##### 4.2.1.4. 证书策略（Certificate Policies）

证书策略扩展包含一个或多个策略信息项的序列，每个项由一个对象标识符（OID）和可选限定符组成。可选限定符如果存在，不预期改变策略的定义。证书策略 OID 不得在证书策略扩展中出现多次。

在终端实体证书中，这些策略信息项指示颁发证书所依据的策略以及证书可以使用的目的。在 CA 证书中，这些策略信息项限制包含此证书的证书路径的策略集合。

本规范定义了两种策略限定符类型：**CPS 指针**和**用户通知**限定符。

##### 4.2.1.5. 策略映射（Policy Mappings）

此扩展用于 CA 证书。它列出一个或多个 OID 对；每对包括一个 `issuerDomainPolicy` 和一个 `subjectDomainPolicy`。配对表示颁发 CA 认为其 `issuerDomainPolicy` 等同于主体 CA 的 `subjectDomainPolicy`。符合性 CA 应将此扩展标记为关键。

```asn1
id-ce-policyMappings OBJECT IDENTIFIER ::=  { id-ce 33 }

PolicyMappings ::= SEQUENCE SIZE (1..MAX) OF SEQUENCE {
     issuerDomainPolicy      CertPolicyId,
     subjectDomainPolicy     CertPolicyId }

##### 4.2.1.6. 主体替代名称（Subject Alternative Name）

Subject Alternative Name 扩展允许将身份绑定到证书主体。这些身份可以包含在证书主体字段中的身份之外或替代该身份。定义的选项包括 Internet 电子邮件地址、DNS 名称、IP 地址和统一资源标识符（URI）。

当 `subjectAltName` 扩展包含 Internet 邮件地址时，地址必须存储在 `rfc822Name` 中。当包含域名系统标签时，域名必须存储在 `dNSName`（IA5String）中。当包含 IP 地址时，地址必须以"网络字节顺序"存储在八位字节字符串中。当包含 URI 时，名称必须存储在 `uniformResourceIdentifier`（IA5String）中。

```asn1
id-ce-subjectAltName OBJECT IDENTIFIER ::=  { id-ce 17 }

SubjectAltName ::= GeneralNames

GeneralNames ::= SEQUENCE SIZE (1..MAX) OF GeneralName

GeneralName ::= CHOICE {
     otherName                       [0]     OtherName,
     rfc822Name                      [1]     IA5String,
     dNSName                         [2]     IA5String,
     x400Address                     [3]     ORAddress,
     directoryName                   [4]     Name,
     ediPartyName                    [5]     EDIPartyName,
     uniformResourceIdentifier       [6]     IA5String,
     iPAddress                       [7]     OCTET STRING,
     registeredID                    [8]     OBJECT IDENTIFIER }

##### 4.2.1.7. 颁发者替代名称（Issuer Alternative Name）

与第 4.2.1.6 节类似，此扩展用于将 Internet 风格的身份与证书颁发者关联。颁发者替代名称不作为第 6 节中证书路径验证算法的一部分进行处理。

```asn1

IssuerAltName ::= GeneralNames

##### 4.2.1.8. 主体目录属性（Subject Directory Attributes）

Subject Directory Attributes 扩展用于传递主体的识别属性（例如国籍）。符合性 CA 必须将此扩展标记为非关键。

##### 4.2.1.9. 基本约束（Basic Constraints）

Basic Constraints 扩展标识证书主体是否为 CA 以及包含此证书的有效证书路径的最大深度。

`cA` 布尔值指示认证的公钥是否可以用于验证证书签名。如果 `cA` 布尔值未被设置，则密钥用法扩展中的 `keyCertSign` 位不得设置。

`pathLenConstraint` 字段仅在 `cA` 布尔值被设置且密钥用法扩展（如果存在）设置了 `keyCertSign` 位时才有意义。在这种情况下，它给出了在有效证书路径中可以跟随此证书的非自签发中间证书的最大数量。

符合性 CA 必须在所有包含用于验证证书数字签名的公钥的 CA 证书中包含此扩展，并且必须将此类证书中的扩展标记为关键。

```asn1
id-ce-basicConstraints OBJECT IDENTIFIER ::=  { id-ce 19 }

BasicConstraints ::= SEQUENCE {
     cA                      BOOLEAN DEFAULT FALSE,
     pathLenConstraint       INTEGER (0..MAX) OPTIONAL }

##### 4.2.1.10. 名称约束（Name Constraints）

Name Constraints 扩展只能在 CA 证书中使用，指示一个命名空间，在该命名空间内，证书路径中后续证书的所有主体名称必须位于其中。限制适用于主体 Distinguished Name 和主体替代名称。

限制以许可或排除名称子树的形式定义。符合性 CA 必须将此扩展标记为关键。

```asn1
id-ce-nameConstraints OBJECT IDENTIFIER ::=  { id-ce 30 }

NameConstraints ::= SEQUENCE {
     permittedSubtrees       [0]     GeneralSubtrees OPTIONAL,
     excludedSubtrees        [1]     GeneralSubtrees OPTIONAL }

GeneralSubtrees ::= SEQUENCE SIZE (1..MAX) OF GeneralSubtree

GeneralSubtree ::= SEQUENCE {
     base                    GeneralName,
     minimum         [0]     BaseDistance DEFAULT 0,
     maximum         [1]     BaseDistance OPTIONAL }

BaseDistance ::= INTEGER (0..MAX)

##### 4.2.1.11. 策略约束（Policy Constraints）

Policy Constraints 扩展可用于颁发给 CA 的证书中。它以两种方式约束路径验证：可以用于禁止策略映射或要求路径中的每个证书包含可接受的策略标识符。符合性 CA 必须将此扩展标记为关键。

```asn1
id-ce-policyConstraints OBJECT IDENTIFIER ::=  { id-ce 36 }

PolicyConstraints ::= SEQUENCE {

     inhibitPolicyMapping            [1] SkipCerts OPTIONAL }

SkipCerts ::= INTEGER (0..MAX)

此扩展指示认证公钥可用于的一个或多个目的，作为密钥用法扩展中指示的基本用途的补充或替代。通常，此扩展只出现在终端实体证书中。

| OID | 名称 | 用途 |

| `id-kp-emailProtection` | 邮件保护 | 电子邮件保护 |
| `id-kp-timeStamping` | 时间戳 | 将对象哈希绑定到时间 |

Authority Information Access 扩展指示如何访问包含该扩展的证书颁发者的信息和服务。信息和服务的示例包括在线验证服务和 CA 策略数据。符合性 CA 必须将此扩展标记为非关键。

## 5. CRL 与 CRL 扩展 Profile

如上所述，此 X.509 v2 CRL [[Profile]] 的目标之一是促进可互操作和可重用的 Internet PKI 的创建。为此，指定了扩展[[使用指南]]，并对 CRL 中包含的信息性质做了一些假设。

CRL 颁发者颁发 CRL。CRL 颁发者是 CA 或已被 CA 授权颁发 CRL 的实体。CA 发布 CRL 以提供其颁发证书的状态信息。然而，CA 可以将此责任委托给另一个可信机构。

每个 CRL 都有特定的范围。CRL 范围是可以出现在给定 CRL 上的证书集合。完整 CRL 列出其范围内所有已撤销的未过期证书。Delta CRL 仅列出其范围内自参考完整 CRL 颁发以来撤销状态发生变化的证书。

符合性 CA 如果提供了其他撤销或证书状态机制，则不要求颁发 CRL。当颁发 CRL 时，CRL 必须是版本 2 CRL，包括 `nextUpdate` 字段、CRL Number 扩展和 Authority Key Identifier 扩展。

### 5.1. CRL 字段

X.509 v2 CRL 语法如下：

```asn1
CertificateList  ::=  SEQUENCE  {
     tbsCertList          TBSCertList,
     signatureAlgorithm   AlgorithmIdentifier,
     signatureValue       BIT STRING  }

TBSCertList  ::=  SEQUENCE  {
     version                 Version OPTIONAL,
                                          -- if present, MUST be v2
     signature               AlgorithmIdentifier,
     issuer                  Name,
     thisUpdate              Time,
     nextUpdate              Time OPTIONAL,
     revokedCertificates     SEQUENCE OF SEQUENCE  {
          userCertificate         CertificateSerialNumber,
          revocationDate          Time,
          crlEntryExtensions      Extensions OPTIONAL
                                    -- if present, version MUST be v2
                                }  OPTIONAL,
     crlExtensions           [0]  EXPLICIT Extensions OPTIONAL
                                    -- if present, version MUST be v2
                                }

### 5.2. CRL 扩展

为 X.509 v2 CRL 定义的扩展提供了将附加属性与 CRL 关联的方法。每个 CRL 中的扩展可以被指定为关键或非关键。符合性 CRL 颁发者必须在所有颁发的 CRL 中包含 Authority Key Identifier（第 5.2.1 节）和 CRL Number（第 5.2.3 节）扩展。

#### 5.2.1. Authority Key Identifier

Authority Key Identifier 扩展提供了标识用于签署 CRL 的私钥对应的公钥的方法。符合性 CRL 颁发者必须使用密钥标识符方法，并且必须在颁发的所有 CRL 中包含此扩展。

#### 5.2.2. 颁发者替代名称（Issuer Alternative Name）

颁发者替代名称扩展允许将附加身份与 CRL 颁发者关联。符合性 CRL 颁发者应将 `issuerAltName` 扩展标记为非关键。

#### 5.2.3. CRL 编号（CRL Number）

CRL Number 是一个非关键 CRL 扩展，为给定的 CRL 范围和 CRL 颁发者传达一个单调递增的序列号。CRL 颁发者必须包含此扩展并将其标记为非关键。

```asn1
id-ce-cRLNumber OBJECT IDENTIFIER ::= { id-ce 20 }

CRLNumber ::= INTEGER (0..MAX)

#### 5.2.4. Delta CRL 指示器（Delta CRL Indicator）

Delta CRL Indicator 是一个关键 CRL 扩展，将 CRL 标识为 delta CRL。Delta CRL 包含对先前分发的撤销信息的更新，而不是完整 CRL 中将出现的所有信息。使用 delta CRL 可以在某些环境中显著减少网络负载和处理时间。

```asn1
id-ce-deltaCRLIndicator OBJECT IDENTIFIER ::= { id-ce 27 }

BaseCRLNumber ::= CRLNumber

#### 5.2.5. 颁发分发点（Issuing Distribution Point）

颁发分发点是一个关键 CRL 扩展，标识特定 CRL 的 CRL 分发点和范围，并指示该 CRL 是否仅覆盖终端实体证书、仅 CA 证书、仅属性证书或有限的原因代码集。

```asn1
id-ce-issuingDistributionPoint OBJECT IDENTIFIER ::= { id-ce 28 }

IssuingDistributionPoint ::= SEQUENCE {
     distributionPoint          [0] DistributionPointName OPTIONAL,
     onlyContainsUserCerts      [1] BOOLEAN DEFAULT FALSE,
     onlyContainsCACerts        [2] BOOLEAN DEFAULT FALSE,
     onlySomeReasons            [3] ReasonFlags OPTIONAL,
     indirectCRL                [4] BOOLEAN DEFAULT FALSE,
     onlyContainsAttributeCerts [5] BOOLEAN DEFAULT FALSE }

#### 5.2.6. 最新 CRL（Freshest CRL）

最新 CRL 扩展标识如何获取此完整 CRL 的 delta CRL 信息。符合性 CRL 颁发者必须将此扩展标记为非关键。此扩展不得出现在 delta CRL 中。

#### 5.2.7. Authority Information Access

本节定义了 Authority Information Access 扩展在 CRL 中的使用。此 CRL 扩展必须标记为非关键。当出现在 CRL 中时，此扩展必须包含至少一个指定 `id-ad-caIssuers` 作为 `accessMethod` 的 AccessDescription。

### 5.3. CRL 条目扩展

为 X.509 v2 CRL 定义的 CRL 条目扩展提供了将附加属性与 CRL 条目关联的方法。如果 CRL 包含应用无法处理的关键 CRL 条目扩展，则应用不得使用该 CRL 来确定任何证书的状态。

#### 5.3.1. 撤销原因（Reason Code）

`reasonCode` 是一个非关键 CRL 条目扩展，标识证书撤销的原因。强烈鼓励 CRL 颁发者在 CRL 条目中包含有意义的原因代码。

```asn1
id-ce-cRLReasons OBJECT IDENTIFIER ::= { id-ce 21 }

CRLReason ::= ENUMERATED {
     unspecified             (0),
     keyCompromise           (1),
     cACompromise            (2),
     affiliationChanged      (3),
     superseded              (4),
     cessationOfOperation    (5),
     certificateHold         (6),
          -- value 7 is not used
     removeFromCRL           (8),
     privilegeWithdrawn      (9),
     aACompromise           (10) }

#### 5.3.2. 失效日期（Invalidity Date）

失效日期是一个非关键 CRL 条目扩展，提供已知或怀疑私钥泄露或证书以其他方式失效的日期。此日期可能早于 CRL 条目中的撤销日期。

#### 5.3.3. 证书颁发者（Certificate Issuer）

此 CRL 条目扩展标识与间接 CRL 中条目关联的证书颁发者。CRL 颁发者必须将此扩展标记为关键。

## 6. 证书路径验证

Internet PKI 的证书路径验证过程基于 [X.509] 中提供的算法。证书路径处理验证主体 Distinguished Name 和/或主体替代名称与主体公钥之间的绑定。该绑定受路径中证书指定的约束和依赖方指定的输入的限制。

本节描述了验证证书路径的算法。符合性实现不要求实现此算法，但必须提供与此过程的外部行为等价的功能。

### 6.1. 基本路径验证

本节描述了 X.509 路径处理算法。该算法在四个基本步骤中呈现：(1) 初始化、(2) 基本证书处理、(3) 为下一个证书做准备、(4) 收尾。

#### 6.1.1. 输入

算法假定向路径处理逻辑提供以下九个输入：

1. 长度为 n 的预期证书路径
2. 当前日期/时间
3. **user-initial-policy-set**：证书用户可接受的证书策略标识符集合
4. **信任锚信息**：包括可信颁发者名称、可信公钥算法、可信公钥和可选的可信公钥参数
5. **initial-policy-mapping-inhibit**：指示是否允许策略映射
6. **initial-explicit-policy**：指示路径是否必须对用户初始策略集中的至少一个策略有效
7. **initial-any-policy-inhibit**：指示 `anyPolicy` OID 是否应被处理
8. **initial-permitted-subtrees**：每种名称类型的许可子树集合
9. **initial-excluded-subtrees**：每种名称类型的排除子树集合

#### 6.1.2. 初始化

初始化阶段基于九个输入建立十一个状态变量，包括 `valid_policy_tree`、`permitted_subtrees`、`excluded_subtrees`、`explicit_policy`、`inhibit_anyPolicy`、`policy_mapping`、`working_public_key_algorithm`、`working_public_key`、`working_public_key_parameters`、`working_issuer_name` 和 `max_path_length`。

#### 6.1.3. 基本证书处理

对证书 i（i 在 [1..n] 中）执行的基本路径处理操作包括：

1. 验证签名、有效期、撤销状态和颁发者名称
2. 验证主体名称在许可子树内
3. 验证主体名称不在排除子树内
4. 处理策略信息
5. 验证显式策略约束

#### 6.1.4. 为证书 i+1 做准备

包括处理策略映射、更新工作变量、处理名称约束、更新约束计数器和验证 CA 状态。

#### 6.1.5. 收尾过程

完成目标证书的处理，包括计算 `valid_policy_tree` 与用户初始策略集的交集。

#### 6.1.6. 输出

如果路径处理成功，过程终止并返回成功指示，连同 `valid_policy_tree` 的最终值、`working_public_key`、`working_public_key_algorithm` 和 `working_public_key_parameters`。

### 6.2. 使用路径验证算法

路径验证算法描述了验证单个证书路径的过程。虽然每个证书路径从特定的信任锚开始，但没有要求特定系统验证的所有证书路径共享单个信任锚。

### 6.3. CRL 验证

本节描述了当 CRL 是证书颁发者使用的撤销机制时，确定证书是否被撤销所需的步骤。该算法定义了一组输入、一组状态变量和对路径中每个证书执行的处理步骤。算法输出是证书的撤销状态。

## 7. 国际化名称的处理规则

国际化名称可能在许多证书和 CRL 字段和扩展中遇到，包括 Distinguished Name、国际化域名、电子邮件地址和国际化资源标识符（IRI）。此类名称的存储、比较和展示需要特别注意。

### 7.1. Distinguished Name 中的国际化名称

符合性实现必须使用 [RFC4518] 中指定的 LDAP StringPrep [[Profile]] 作为比较以 PrintableString 或 UTF8String 编码的 Distinguished Name 属性的基础。符合性实现必须支持使用 `caseIgnoreMatch` 的名称比较。

### 7.2. GeneralName 中的国际化域名

符合性实现必须在存储到 `dNSName` 字段之前将国际化域名转换为 ASCII Compatible Encoding（ACE）格式。比较 DNS 名称时，必须执行不区分大小写的完全匹配。

### 7.3. Distinguished Name 中的国际化域名

在 Distinguished Name 中使用 domainComponent 属性表示域名标签时，必须执行 [RFC3490] 第 4.1 节中指定的 "ToASCII" 标签转换。

### 7.4. 国际化资源标识符

符合性实现必须按照 [RFC3987] 第 3.1 节的指定将 IRI 映射为 URI。比较 URI 时，必须执行区分大小写的完全匹配。

### 7.5. 国际化电子邮件地址

电子邮件地址中包含国际化域名时，域名必须从 IDN 转换为 ACE 格式。两个电子邮件地址在本地部分完全匹配且主机部分使用不区分大小写的 ASCII 比较匹配时被认为是匹配的。

## 8. 安全考虑

本规范的大部分内容涉及证书和 CRL 的格式和内容。由于证书和 CRL 经过[[数字签名]]，不需要额外的完整性服务。证书和 CRL 都不需要保密，不受限制和匿名访问证书和 CRL 没有安全影响。

然而，本规范范围之外的安全因素会影响向证书用户提供的保证。

> [!warning] 关键安全要点
> - **私钥保护**：私钥的保护是关键安全因素。小规模上，用户未能保护其私钥将允许攻击者冒充他们或解密其个人信息。大规模上，CA 私有签名密钥的泄露可能产生灾难性影响。
> - **密钥分离**：强烈不建议将单个密钥对同时用于签名和其他目的。使用单独的签名和[[密钥管理]]密钥对为用户提供了多项好处。
> - **撤销信息的可用性和新鲜度**：影响证书中应具有的保证程度。如果撤销信息不及时或不可用，与绑定相关的保证显然会降低。
> - **信任锚**：路径验证算法依赖于对一个或多个可信 CA 的公钥（和其他信息）的确定了解。信任 CA 的决定是一个重要决定。
> - **循环依赖**：当证书在 CRL 分发点、Authority Information Access 或 Subject Information Access 扩展中包含 https URI 或类似方案时，可能引入循环依赖。

## 附录 B. ASN.1 注释

CA 必须强制 `serialNumber` 为非负整数，即 INTEGER 值的 DER 编码中符号位必须为零。这可以通过在必要时添加前导（最左侧）`\`00'H 八位字节来实现。这消除了八位字节字符串与整数值之间映射的潜在歧义。

如第 4.1.2.2 节所述，序列号可以预期包含长整数。证书用户必须能够处理最多 20 个八位字节长度的 `serialNumber` 值。符合性 CA 不得使用超过 20 个八位字节的 `serialNumber` 值。

如第 5.2.3 节所述，CRL 编号可以预期包含长整数。CRL 验证器必须能够处理最多 20 个八位字节长度的 `cRLNumber` 值。符合性 CRL 颁发者不得使用超过 20 个八位字节的 `cRLNumber` 值。

构造 `SEQUENCE SIZE (1..MAX) OF` 出现在多个 [[ASN.1]] 构造中。有效的 [[ASN.1]] 序列将有零个或多个条目。`SIZE (1..MAX)` 构造约束序列至少有一个条目。MAX 指示上限未指定。实现可以自由选择适合其环境的上限。

字符串类型 PrintableString 支持非常基本的拉丁字符集：小写字母 'a' 到 'z'、大写字母 'A' 到 'Z'、数字 '0' 到 '9'、十一个特殊字符 `= ( ) + , - . / : ?` 和空格。

> [!note] 实现者注意
> 符号 `@` 和下划线 `_` 不受 [[ASN.1]] 类型 PrintableString 支持。这些字符通常出现在 Internet 地址中。此类地址必须使用支持它们的 [[ASN.1]] 类型编码。通常编码为 Distinguished Name 中 emailAddress 属性的 IA5String 或 GeneralName 的 rfc822Name 字段。

字符串类型 TeletexString 是 PrintableString 的超集。TeletexString 支持相当标准的（类似 ASCII）拉丁字符集：带非间距重音符号的拉丁字符和日文字符。

命名位列表是已为值分配名称的 BIT STRING。在 DER 编码命名位列表时，必须省略尾随零。即编码值以设置为 1 的最后一个命名位结束。

字符串类型 UniversalString 支持 [ISO10646] 允许的任何字符。ISO 10646 是通用多八位字节编码字符集（UCS）。

字符串类型 UTF8String 在 1997 年版本的 [[ASN.1]] 中引入，并在 2001 年版本的 [X.520] 中添加到 DirectoryString 的选择列表中。UTF8String 是通用类型，已分配标签号 12。UTF8String 的内容由 RFC 2044 定义，在 RFC 2279 中更新，又在 [RFC3629] 中更新。

实现者应注意，SET OF 值的 DER 编码需要对值的编码进行排序。特别是，此问题出现在 Distinguished Name 方面。

实现者应注意，值为 DEFAULT 的 SET 或 SEQUENCE 组件的 DER 编码会从编码的证书或 CRL 中省略该组件。例如，`cA` 值为 FALSE 的 BasicConstraints 扩展会从编码的证书中省略 `cA` 布尔值。

对象标识符（OID）在本规范中广泛使用，用于标识证书策略、公钥和签名算法、证书扩展等。OID 没有最大大小。本规范要求支持弧元素值小于 2^28 的 OID，即必须在 0 到 268,435,455 之间（含）。这允许每个弧元素在单个 32 位字中表示。实现还必须支持点分十进制字符串表示长度最多 100 字节（含）的 OID。实现必须能够处理最多 20 个元素（含）的 OID。CA 不应颁发包含超过这些要求的 OID 的证书。同样，CRL 颁发者不应颁发包含超过这些要求的 OID 的 CRL。

nameConstraints 扩展中 GeneralName 字段值的编码特定规则与适用于其他扩展的规则不同。如果包含意外值的扩展标记为关键，实现必须拒绝包含未识别扩展的证书或 CRL。对于未标记为关键的扩展中的未知元素应忽略：

1. 忽略位字符串中所有未知的位名称分配；
2. 忽略 ENUMERATED 类型或以枚举方式使用的 INTEGER 类型中所有未知的命名数字，前提是数字作为 SET 或 SEQUENCE 的可选元素出现；
3. 忽略 SET 中、SEQUENCE 末尾或 CHOICE 中的所有未知元素，其中 CHOICE 本身是 SET 或 SEQUENCE 的可选元素。
